## 目的

社内ITは業務範囲が広い。ヘルプデスク対応、品質維持、ルール作成、事業部からの要望対応——
幅広い領域に高度な知識と判断が同時に求められる。緊急度の高い目の前の仕事に意識を奪われ、
走り続けるうちに、次のような問いに答えられなくなりがちである。

- **進むべき方向性としてただいいのか**
- **目の前の対応に追われて、本当に大事なことが後回しになっていないか**
- **組織全体として何が抜けているのか**

このリポジトリは、社内IT業務を一枚のツリーに整理し、二つの役割を持たせることを目的とする。

- **現在位置の地図** —— 自分の業務を全体のどこに位置づけられるかを示す
- **あるべき姿の指針** —— 各領域のベストプラクティスを集積し、改善の方向を示す

会社は違えど、社内ITに求められる方向性や考え方は本質的に似ている。
本リポジトリは、その共通項を集約し、社内ITに携わる人たちの共通の参照点となることを目指す。
実装方法や仕組みは時代とともに変化していくが、目指すべき方向性は大きく変わらない。

## コントリビュート歓迎

このドキュメントは特定の個人のものではなく、有志のコミットで継続的に改善していくことを前提としている。追記・修正の提案は歓迎する。

## 全体像

分類の軸は **「既存のものへどう働きかけるか」** ＝ 運用・改善・変革。そこへ全体を方向づける **統制** を横串のレイヤーとして重ねる。

```
社内IT業務
├── 1. 運用（Run）── 今あるものを止めない（対応サイクル型）
│   ├── ユーザーサポート（ヘルプデスク）
│   ├── アプリケーション・SaaS運用保守
│   ├── インフラ運用
│   ├── 可用性・事業継続管理
│   ├── セキュリティ運用
│   ├── 運用品質・管理
│   └── オフィスIT・ファシリティ
├── 2. 改善（Grow）── 今あるものを良くする・安くする（プロジェクト型）
│   ├── 手作業の自動化（RPA・スクリプト）
│   ├── ツール／SaaS導入（効率化目的）
│   ├── 業務プロセス改善・標準化
│   └── 効率化のための内製・ローコード開発
├── 3. 変革（Transform）── 新しい価値を生む（プロジェクト型）
│   ├── 新規事業・新サービスのIT支援
│   ├── デジタル接点・顧客体験の創出
│   ├── データ活用による新価値創出
│   ├── 先進技術による競争力強化
│   ├── 既存システムの刷新・モダナイゼーション
│   └── 全社DX・変革の推進
└── ※ 統制（Govern）── 上の3つすべてに横串で方向づけ・規律を与える（横断レイヤー）
    ├── IT戦略・企画
    ├── プロジェクト・ポートフォリオ管理（PMO）
    ├── ガバナンス・統制
    ├── データガバナンス
    ├── ベンダー・契約管理
    ├── 全社アーキテクチャ
    └── IT組織・人材マネジメント
```

運用・改善・変革は「働きかけの種類」で並ぶ対等な3区分、統制はその全体に重なるレイヤー。各ノードの詳細は後述の章を参照。

## 分類の基準（運用／改善／変革の分け方）

**まず「既存のものへどう働きかけるか」で大きく分け、改善と変革は「目的」で分ける。**

判定は2段階。

1. **既存を維持するだけか、変えるのか** … 止めずに回すこと自体が仕事なら **運用（Run）**。何かを変える取り組みなら 改善 or 変革へ。
2. **変える目的は「効率化」か「新価値創出」か** … コスト削減・既存業務の効率化が目的なら **改善（Grow）**、売上拡大・新規事業・競争優位など新たな価値の創出が目的なら **変革（Transform）**。この2段目は経済産業省の「攻め／守り」＝目的で分ける考え方に対応する。

> 重要：**「内製開発」「データ活用」「自動化」などは“手段”であり、目的で振り分ける。**
> 同じ技術でも、効率化が目的なら改善、新価値創出が目的なら変革に入る。手段では所属は決まらない。

## このツリーの使い方（現在地の出し方）

自分の業務位置は **「枝 × フェーズ」** の2座標で特定する。

- **枝（What）** … このツリーのどのノードにいるか ＝ 届けたい価値
- **フェーズ（Where）** … その仕事がどの進行段階にあるか

> 例：経理部のExcel集計を自動化中（作り始め）
> → コスト削減が目的なので `改善 > 手作業の自動化` ／ フェーズ＝**構築・導入**

---

## 1. 運用（Run：今あるものを止めない）

既存のサービス・基盤を安定して回し続ける、対応サイクル型の仕事。ITIL／ITSM のプラクティスに対応。

- **[ユーザーサポート（ヘルプデスク）](docs/1-運用/ユーザーサポート.md)**
  - 問い合わせ・依頼対応（リクエスト管理）
  - 障害・トラブル対応（インシデント管理）
  - 端末管理（入退社・キッティング・周辺機器）
- **アプリケーション・SaaS運用保守**
  - 基幹・業務システムの保守（バージョン更新・障害対応）
  - グループウェア・SaaSの運用（提供・ベンダー連携）
- **インフラ運用**
  - サーバー・クラウド運用
  - ネットワーク運用
  - 監視・バックアップ
- **可用性・事業継続管理**
  - BCP／DR（災害復旧・復旧訓練）
  - 可用性・キャパシティ管理
- **セキュリティ運用**
  - ID・アクセス権限管理（アカウント発行〜棚卸し・権限）
  - 脆弱性・パッチ対応
  - セキュリティインシデント対応
- **運用品質・管理**
  - 問題管理（再発防止）
  - 変更管理
  - IT資産・ライセンス管理
- **オフィスIT・ファシリティ**
  - 会議室AV・複合機・固定／IP電話・配線

**フェーズ軸**：受付 → 対応 → 解決 → 記録（＋PDCA）

---

## 2. 改善（Grow：今あるものを良くする・安くする）

既存業務をITで楽に・安くする、プロジェクト型の仕事（目的＝効率化）。

- 手作業の自動化（RPA・スクリプト）
- ツール／SaaS導入（効率化目的）
- 業務プロセス改善・標準化
- 効率化のための内製・ローコード開発

**フェーズ軸**：発見 → 分析 → 企画 → 構築 → 定着 → 改善

---

## 3. 変革（Transform：新しい価値を生む）

売上拡大・新規事業・競争優位・新たな収益モデルの創出（経産省の「攻めのIT」が指す領域）。

- **新規事業・新サービスのIT支援**（内製・データ活用を含む、新価値創出が目的のもの）
- **デジタル接点・顧客体験の創出**（EC・Web・新チャネル）
- **データ活用による新価値創出**（売上・意思決定に効く分析・予測、データの新規活用）
- **先進技術による競争力強化**（AI／生成AIで新価値）
- **既存システムの刷新・モダナイゼーション**（レガシー刷新・クラウド移行・システム統合＝「2025年の崖」対応）
- **全社DX・変革の推進**（事業モデル変革のドライブ）

**フェーズ軸（プロジェクト型）**：発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善

---

## ※ 統制（Govern：横断レイヤー）

運用・改善・変革のすべてに横串で方向づけ・規律を与える上位の仕事。働きかけの3区分と並ぶのではなく、全体に重なるレイヤーである。COBIT／TOGAF に対応。

- **IT戦略・企画**
  - 中長期戦略・ロードマップ
  - 予算・コスト管理
  - 投資判断・システム選定
- **プロジェクト・ポートフォリオ管理（PMO）**
  - 案件の優先順位づけ・予算配分
  - 全社横断の進捗・品質管理
- **ガバナンス・統制**
  - ITポリシー・規程
  - 内部統制・監査対応（IT統制・J-SOX）
  - コンプライアンス（個人情報・法令）
  - ITリスク管理（全般）
- **データガバナンス**
  - データ管理方針・品質
  - データ・連携基盤の方針
- **ベンダー・契約管理**
  - ベンダー選定・評価
  - 契約・SLA管理
- **全社アーキテクチャ**
  - システム全体構成・技術標準
- **IT組織・人材マネジメント**
  - 採用・育成・スキル管理
  - 推進体制・組織設計
  - 全社ITリテラシー教育

**フェーズ軸（年次サイクル＋案件サイクル）**：策定 → 運用 → 評価 → 見直し

---

## 補足：分類が迷いやすい箇所（境界メモ）

- **SaaS／ツール1つでも複数分類にまたがる** … 選定・契約＝統制、導入プロジェクト＝改善（効率化目的）or 変革（新価値目的）、稼働後の運用・ライセンス＝運用。ライフサイクルの段階で置き場が変わる。
- **同じ技術（内製・データ・AI）は目的で振り分け** … 効率化＝改善／新価値創出＝変革。
- **アカウント／ID管理は「セキュリティ運用 > ID・アクセス権限管理」に一本化** … 旧版でユーザーサポート側と重複していたのを解消。

## 出典・位置づけ

| 要素 | 根拠 |
|------|------|
| 運用／改善／変革の3区分（働きかけの種類） | Gartner の Run／Grow／Transform（IT投資・活動の分類モデル）に対応 |
| 改善／変革の分かれ目（目的で分割） | 経済産業省（DXレポート／攻めのIT活用指針 等）の攻め／守り＝目的で分ける考え方。効率化＝改善、新価値創出＝変革 |
| 運用配下の中身 | ITIL／ITSM のプラクティス |
| 統制配下 | COBIT（統制）／TOGAF（アーキテクチャ） |
| ツリーへの統合・各枝の粒度・フェーズ軸 | 上記の概念を自社向けに再構成した合成（特定の単一出典なし） |

個々の用語・大分類は既存フレームワークに準拠。ツリーの構造自体は自社向けのカスタム整理。
