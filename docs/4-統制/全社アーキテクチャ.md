# 全社アーキテクチャ

## 概要

全社のシステム全体構成（どのシステムがどう繋がるか）と技術標準（採用してよい技術・方式・基盤の指針）を定め、個別最適の乱立・サイロ化・無秩序な技術選定を防ぐ業務。今の整合だけでなく、数年先の変化への対応力を担保する土台でもある。EA（エンタープライズアーキテクチャ：全社の構造を体系的に整理する考え方）を社内ITに適用したものである。

> 枝＝`統制 > 全社アーキテクチャ` ／ フェーズ軸＝**策定 → 運用 → 評価 → 見直し**

---

## あるべき姿

- **全体像が見える** … どのシステムがどのデータでどう繋がっているかの地図があり、ブラックボックス化していない
- **標準が迷いを減らす** … 採用してよい技術・方式・基盤が示され、案件ごとにゼロから悩まなくて済む
- **例外を扱える** … 標準外を全面禁止にせず、理由を添えて選べる例外プロセスがあり、現実に即している
- **個別最適が暴走しない** … 各部署・各案件の都合による技術の乱立が、全体の整合の中に収まっている
- **技術的負債が見える** … 技術的負債（その場しのぎの設計が将来の足かせになること）が可視化され、計画的に解消されている
- **変化を支える** … 改善・変革・刷新が来たとき、土台が整っていて素早く動ける
- **現場と握れている** … 標準が現実離れせず、実際に守られている（形骸化していない）

## 基本方針（汎用）

会社や使う技術が違っても、全社アーキテクチャの土台となる考え方は共通する。

- **全体最適と個別最適を調整する** … 各部署・各案件は自分の都合で技術を選びたがる。それ自体は自然だが、放置すると全社では重複・分断・非効率になる。全体の整合の観点から調整する
- **標準は「縛り」でなく「迷わない指針」** … 標準の目的は禁止ではなく、案件ごとの判断コストを下げること。「これを選んでおけば外さない」という道しるべとして示す
- **例外を許容するプロセスを持つ** … 標準外が必要な場面は必ずある。全面禁止にせず、理由・期限・撤退条件を添えて選べる経路を用意する。例外を裁ける仕組みが、かえって標準を生かす
- **連携の地図を持つ** … システム間連携・データ連携の全体像を描き、最新に保つ。地図が無いと、一つの変更がどこに波及するか誰も分からなくなる
- **技術的負債を可視化し、計画的に返す** … 場当たりの設計は積み重なる。負債を一覧化し、放置・先送りではなく、いつ・どう返すかを計画に乗せる
- **今だけでなく数年先を見る** … 目先の最適だけで選ぶと、将来の刷新で大きな代償を払う。拡張性・移行のしやすさ・撤退のしやすさを判断軸に入れる
- **現場と握って実効性を持たせる** … 現場が守れない標準は守られない。理想論で縛る「象牙の塔」にせず、現場と合意し、実情に合わせて更新し続ける

## 主な業務

### システム全体構成・技術標準

全社のシステムがどう繋がるかの全体像を描き、採用してよい技術・方式・基盤の標準を整える業務。EA を社内ITに適用する中核であり、TOGAF（アーキテクチャ整備の体系的な枠組み）の考え方に対応する。内容に即して、下位に二つの柱を置く。

#### 全体構成（システム間連携の地図）

「どのシステムが、どのデータを、どう繋いでいるか」の全体像を描き、最新に保つ。

- 主要システムと、その間の連携（API：システム同士をつなぐ接続口、ファイル連携、データ連携など）を一枚の地図にする
- 業務の流れとシステムの流れを対応づけ、「この業務はどのシステム群で支えられているか」を見えるようにする
- 一つのシステムを変えたとき、どこに波及するかを地図から追えるようにし、ブラックボックス化を防ぐ
- 地図は作って終わりにせず、システムの増減・連携の変化に合わせて更新する。古い地図は誤った地図より危ういこともある

#### 技術標準・方式の整備

採用してよい技術・方式・基盤の指針を定め、案件ごとの技術選定の迷いと乱立を減らす。

- 言語・基盤・認証・データ連携方式・SaaS選定などについて、標準（推奨）の選択肢を示す
- 標準は「これを選べば外さない」という道しるべであり、思考停止の禁止リストではない
- 標準外が必要な場合の**例外プロセス**を併せて定める。理由・期限・撤退条件を添えて申請・記録し、全面禁止にしない
- 採否の判断軸（拡張性・移行のしやすさ・運用負荷・コスト・スキル調達のしやすさ）を明文化し、判断のばらつきを減らす
- 標準は現場と握って決め、現実に合わなくなったら見直す。守られない標準は無いのと同じである

## フェーズ別の進め方

統制のフェーズ軸（策定 → 運用 → 評価 → 見直し）に沿って、各段でやること・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 策定 | 現状のシステム構成・連携を棚卸しし、全体像（As-Is）とあるべき姿（To-Be）、技術標準・例外プロセスを定める | システム全体構成図・技術標準・例外プロセス | 現場を見ずに理想だけで標準を作り、現実離れする／棚卸しが終わらず策定が止まる |
| 運用 | 案件・調達・設計のレビューで標準を適用し、地図を最新に保つ。例外申請を裁く | レビュー記録・更新された地図・例外台帳 | 標準が形だけで案件に効いていない／地図の更新が止まり実態とずれる |
| 評価 | 標準の遵守状況・例外の傾向・技術的負債を点検し、効いているかを測る | 遵守状況・負債一覧・評価レポート | 例外が常態化しているのに放置／負債を可視化しても返済計画に繋がらない |
| 見直し | 評価を踏まえ標準・全体像を更新し、技術的負債の解消を計画に乗せる | 改訂版の標準・全体像・負債解消ロードマップ | 一度作った標準を更新せず塩漬けにする／技術の変化に追従できない |

> フェーズは一度回して終わりではなく、年次サイクルと案件サイクルの両方で回す。案件のたびに標準を適用し（運用）、定期的に全体を見直す（評価・見直し）。

## やらないこと

全社アーキテクチャは「標準と全体像を示し導く」側であり、個別の実装・運用そのものを奪わない。**あえてやらない・抱え込まないこと**を明確にして、本来の役割を守る。

- **個別システムの構築・運用を抱えること** … 個々のシステムの作り込み・日々の運用は、目的に応じて `運用 > インフラ運用`・`改善 > 効率化のための内製・ローコード開発` などの担当へ渡す。アーキテクチャは全体像と標準を示す側であり、個別実装を奪わない
- **刷新プロジェクトの実行を抱えること** … レガシー刷新・モダナイゼーションの実行そのものは `変革 > 既存システムの刷新・モダナイゼーション` が担う。アーキテクチャは刷新の土台となる方針・標準・移行先の絵を示し、実行を導く側に立つ
- **基盤の日常運用を抱えること** … サーバー・クラウド・ネットワークの日々の運用は `運用 > インフラ運用` の領域。アーキテクチャは「どの基盤を標準とするか」を示すが、運用当番にはならない
- **理想論で現場を縛る「象牙の塔」になること** … 現場の実情を見ずに机上の理想だけで標準を押し付けない。守られず形骸化するだけでなく、現場の不信を招く。標準は現場と握って実効性を持たせる
- **標準外を一律で禁止すること** … 「標準にないから不可」で思考を止めない。例外プロセスに乗せ、理由・期限・撤退条件とともに記録する。禁止ではなく統制された選択にする

> ここで線を引くことは、現場の自由を奪わないためでもある。**標準と全体像で方向づけ、実装・運用は各担当に委ねる**。導くことと、奪うことは違う。

## よくある課題

- **個別最適の乱立（サイロ化）** … 各部署・各案件が独自に技術を選び、似たシステムが重複し、繋がらないデータが散在する
- **連携のブラックボックス化** … 「どこと繋がっているか誰も知らない」連携が増え、変更が怖くて手を出せなくなる
- **技術的負債の放置** … その場しのぎの設計が積み重なり、可視化も返済もされず、刷新時に一気に跳ね返る
- **標準の形骸化** … 標準はあるが現実離れしていて誰も守らない、あるいは更新されず古いまま塩漬けになっている
- **「象牙の塔」化** … アーキテクチャが現場から浮き、机上の理想だけを語って案件に効かない
- **例外の常態化** … 例外プロセスが甘く、なし崩しに標準外が増え、標準が標準でなくなる

## ベストプラクティス

- **現状（As-Is）と目標（To-Be）の二枚の地図** … 今の全体構成と、目指す姿を分けて描く。ギャップが見え、刷新・統合の優先順位が立つ
- **技術標準と例外台帳をセットで運用** … 標準だけでなく、例外の申請・承認・期限・撤退条件を台帳で管理する。例外の傾向から標準の見直し点が見える
- **アーキテクチャレビューを案件の節目に組み込む** … 大きな調達・設計の節目でレビューを通す。後追いでなく、決まる前に全体整合の観点を入れる
- **技術的負債の一覧化と返済計画** … 負債を一覧化し、影響度・返済コストで優先順位をつけ、計画的に解消する。`統制 > IT戦略・企画` の投資判断・ロードマップと連動させる
- **連携・データ基盤の方針は連携先と握る** … システム間連携・データ連携の方針は `統制 > データガバナンス` と整合させ、二重管理を避ける
- **標準を現場参加で決める** … 標準策定に現場の設計者を巻き込む。一方的に配るより、合意して作った標準のほうが守られる
- **判断軸を明文化する** … 「拡張性・移行のしやすさ・運用負荷・コスト・スキル調達」など採否の軸を言語化し、属人的な好みでの選定を防ぐ

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 個別最適 | 場当たり | 全体像も標準も無い。各部署・各案件が独自に技術を選び、システムが乱立・サイロ化している |
| Lv2 可視化 | 地図を持つ | システム全体構成・連携の地図があり、現状が見える。ブラックボックスが減り始める |
| Lv3 標準化 | 指針で導く | 技術標準と例外プロセスが整い、案件のレビューで適用される。判断のばらつきが減る |
| Lv4 進化対応 | 変化を支える | 技術的負債が可視化・計画返済され、As-Is/To-Be が更新され続ける。刷新・変革を素早く支える土台になっている |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（全体像の可視化）を固める**こと。地図が無ければ、標準を作っても効かせる先が見えず、技術的負債も測れない。

## 参考資料

- TOGAF（The Open Group Architecture Framework：アーキテクチャ整備の体系的な枠組み）… EA を策定・運用する手順・成果物の考え方の出典
- COBIT（ITガバナンスの枠組み）… 統制レイヤーとしての位置づけ・評価の考え方
- EA（エンタープライズアーキテクチャ）の一般的知見（ビジネス／データ／アプリケーション／技術の各層を整理する考え方）
- 関連ノード：`変革 > 既存システムの刷新・モダナイゼーション`（刷新の実行先）、`運用 > インフラ運用`（基盤の日常運用先）、`統制 > IT戦略・企画`（投資判断・ロードマップとの連動）、`統制 > データガバナンス`（データ・連携基盤の方針との整合）、`改善 > 効率化のための内製・ローコード開発`（個別実装の担当先の一例）
