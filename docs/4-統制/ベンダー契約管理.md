# ベンダー・契約管理

## 概要

外部ベンダー・委託先・SaaS（クラウド型のソフトウェア提供形態）提供元との関係と契約を、全社的に管理する横断機能。誰から・どんな条件で・いくらで・どこまでの品質保証で買うかを統制し、ベンダー選定の公平性、契約とSLA（Service Level Agreement、サービス品質の合意）の実効性、コスト最適化、ベンダー依存のリスクを束ねて管理する。

> 枝＝`統制 > ベンダー・契約管理` ／ フェーズ軸＝**策定 → 運用 → 評価 → 見直し**

複数のSaaS導入・システム刷新・日々の運用に共通して効く横串の規律であり、個別案件ごとにバラバラな判断で発注・契約することを防ぐ。READMEの境界メモ「SaaS／ツールの選定・契約＝統制」の受け皿にあたる。

---

## あるべき姿

- **選定に根拠がある** … 発注先が情実や惰性ではなく、評価基準と複数社の比較に基づいて選ばれている
- **契約が一元的に見える** … 何の契約が・いつ切れ・いくらかかっているかが台帳で一覧でき、気づいたら自動更新・値上げが起きない
- **SLAが実効性を持つ** … 品質の約束が文書にあるだけでなく、達成状況がモニタリングされ、未達時の扱いが定められている
- **乗り換え余地が残っている** … 特定ベンダーに過度に縛られず、データの持ち出しと移行の道筋（出口戦略）が確保されている
- **コストに無駄がない** … 重複契約・使われていないライセンス・放置された自動更新が定期的に棚卸しされている
- **ベンダーのリスクが見えている** … 委託先の経営状態・再委託・サプライチェーン上の弱点が把握され、依存度に応じて管理されている

## 基本方針（汎用）

会社や扱う製品が違っても、ベンダー・契約管理の土台となる考え方は共通する。

- **選定は公平・透明に** … 評価基準を先に決め、複数社を同じ物差しで比べる。「いつもの会社だから」「紹介だから」で発注しない。判断の根拠を残す
- **契約は締結して終わりではない** … 契約とSLAは結んだ後の履行確認まで含めて初めて意味を持つ。約束された品質が実際に守られているかを見続ける
- **期限を握る** … 契約期限・更新日・解約通知期限を台帳で管理する。自動更新条項と通知期限を見落とすと、不要な契約が惰性で延びコストが膨らむ
- **出口を最初に考える** … 入る前に出方を決める。データの形式・移行可否・解約条件を契約時点で確認し、ベンダーロックイン（特定業者に縛られて乗り換え困難になる状態）を避ける
- **依存を分散する** … 一社・一製品への過度な集中は、値上げ・品質低下・倒産時に交渉余地を失わせる。代替可能性を常に意識する
- **窓口と統制を分ける** … 日々のベンダーとのやり取りは運用が担い、契約・コスト・リスクの統括は本機能が担う。実務の連絡経路と契約管理の責任を混同しない

## 主な業務

### ベンダー選定・評価

新たに製品・サービス・委託先を選ぶ際の、**公平で根拠ある選定**と、選定後のベンダーの継続的な評価を担う。

- 要件を整理し、必要に応じてRFP（Request for Proposal、提案依頼書）を作成して複数社へ提示する。規模の小さい調達でも、最低限の比較観点（機能・コスト・サポート・実績・継続性）を揃える
- 評価基準を選定前に定め、各社を同じ物差しで採点する。価格だけで決めず、総保有コスト（導入後の運用・保守・移行まで含めた費用）で比較する
- 製品の技術的妥当性や投資の是非そのものは `統制 > IT戦略・企画`（投資判断・システム選定）と連携して判断する。本機能は「どのベンダーと・どんな条件で契約するか」を担い、投資の最終判断は抱え込まない
- 発注後も取引を一度きりで終えず、納期・品質・対応の実績をベンダー評価として蓄積し、次回以降の選定と更新判断の材料にする

### 契約・SLA管理

締結した契約とSLAを台帳で管理し、**期限・コスト・品質の約束が守られているか**を統括する。

- 契約情報（契約先・対象・金額・契約期間・更新日・解約通知期限・自動更新の有無）を台帳で一元管理する。属人的な記憶や個別フォルダに散らさない
- SLAでは品質指標（稼働率・応答時間・サポート対応時間など）と、未達時のペナルティ（料金減額・是正措置）を明確にする。達成状況を定期的にモニタリングし、未達があれば記録して交渉する
- NDA（Non-Disclosure Agreement、秘密保持契約）や個人情報の取り扱い条項など、リスクに関わる契約条件を確認する。契約上の法務・コンプライアンス観点は `統制 > ガバナンス・統制` と連携する
- 更新期限の前に棚卸しを行い、継続・条件変更・解約を能動的に判断する。重複契約・使われていないライセンス・放置された自動更新を洗い出し、コストを最適化する

## フェーズ別の進め方

統制のフェーズ軸（策定 → 運用 → 評価 → 見直し）に沿って、各段でやること・つまずきを示す。年次サイクルと個別の契約サイクルの両方に当てはまる。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 策定 | 選定・評価の基準づくり、RFP作成、複数社比較、契約・SLA条件の交渉と締結 | 評価基準・RFP・契約書・SLA・契約台帳への登録 | 評価基準を決めずに発注先ありきで進める／出口条件を詰めずに契約する |
| 運用 | 契約台帳の維持、SLA達成状況のモニタリング、日々のベンダー連携（窓口は運用） | 更新された台帳・SLAモニタリング記録 | 締結後に放置しSLAを誰も見ない／窓口対応と契約統括が混在する |
| 評価 | ベンダーの品質・コスト・リスクの定期評価、契約・ライセンスの棚卸し | ベンダー評価結果・コスト棚卸しレポート | 使っていないライセンス・重複契約を見逃す／値上げに気づかない |
| 見直し | 更新・条件変更・解約・乗り換えの判断、出口戦略の発動 | 更新／解約の意思決定・移行計画 | 自動更新で惰性継続する／乗り換え手段がなくロックインに陥る |

## やらないこと

ベンダー・契約管理は「ベンダーに関することは何でも引き受ける部門」になりやすい。**あえてやらない・抱え込まないこと**を明確にして、本来の統括機能に集中する。

- **個別SaaSの日々の運用・障害一次対応を担うこと** … 稼働中サービスの運用・問い合わせ・障害一次対応は `運用 > アプリケーション・SaaS運用保守` が担う。本機能はベンダーとの契約・SLA・評価を統括するのであり、日々のオペレーションは抱え込まない
- **技術標準そのものを決めること** … どの技術・製品群を全社標準とするかは `統制 > 全社アーキテクチャ` の領域。本機能はその標準に沿ってベンダーと契約条件を整える側に立つ
- **投資の最終判断を代行すること** … 投資の是非・予算配分の最終判断は `統制 > IT戦略・企画` が行う。本機能は選定・契約面の情報と評価を提供する。発注の可否を独断で決めない
- **法務・コンプライアンスの最終解釈を担うこと** … 契約条項の法的妥当性や法令適合の最終判断は `統制 > ガバナンス・統制` および法務と連携する。契約のリスク観点を拾い上げるが、法的判断を抱え込まない
- **特例発注を黙認すること** … 現場主導でルートを飛ばした発注・契約を「すでに使っているから」で追認しない。記録し、正規の選定・契約プロセスに乗せ直す

> ここで線を引くことは「縦割り」ではない。**契約・コスト・リスクの統括に集中させ、運用・戦略・標準の担い手を明示する**ためのもの。

## よくある課題

- **情実・惰性での発注** … 「いつもの会社」「紹介された会社」に比較なしで発注し、条件が割高・不利になっていることに気づかない
- **気づいたら自動更新・値上げ** … 解約通知期限を逃して不要な契約が延長され、改定価格で更新されてしまう
- **SLAが飾り** … 契約書にSLAはあるが、達成状況を誰も見ておらず、未達でもペナルティが運用されない
- **ベンダーロックイン** … データの持ち出しや移行が難しい設計で、値上げや品質低下に交渉余地なく従わざるを得ない
- **コストの無駄** … 重複した契約、使われていないライセンス、退職者分の課金が棚卸しされず払い続けられている
- **契約情報の散在** … 契約書が担当者個人のフォルダやメールに散り、全体像が把握できない。担当者の異動で所在不明になる
- **ベンダー依存リスクの放置** … 委託先の経営難・再委託・サプライチェーン上の弱点を把握しておらず、いざという時に止まる

## ベストプラクティス

- **契約台帳の整備** … 契約先・対象・金額・契約期間・更新日・解約通知期限・自動更新の有無を一覧化し、更新前にアラートが出る形にする。「期限を握る」を仕組みで担保する
- **評価基準とRFPの定型化** … 選定の観点（機能・コスト・サポート・実績・継続性・出口条件）をテンプレート化し、調達のたびに複数社を同じ物差しで比べる
- **SLAのモニタリング** … 稼働率・応答時間などの指標と未達時のペナルティを契約に明記し、達成状況を定期的に確認して記録する。守れない品質を約束させない・約束しない
- **出口戦略の事前確認** … 契約時にデータ形式・移行可否・解約条件を確認する。代替手段を1つ以上想定し、ロックインの度合いを把握しておく
- **コストの定期棚卸し** … 重複契約・休眠ライセンス・自動更新を年次（または更新前）で洗い出し、解約・統合・条件交渉でコストを最適化する
- **ベンダーリスクの定期評価** … 主要ベンダーの経営状態・再委託状況・情報セキュリティ体制を定期的に確認し、依存度の高い相手ほど代替可能性を厚く持つ
- **内製とのバランス検討** … すべてを外部に委ねず、中核となる業務知識や運用ノウハウは社内に残す。依存しすぎると交渉力と継続性を失う

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 個別発注 | 場当たり | 案件ごとに担当者の裁量で発注。契約情報が散在し、比較も棚卸しもない |
| Lv2 台帳管理 | 一覧化 | 契約が台帳で一元管理され、期限・金額・更新日が見える。自動更新の見落としが減る |
| Lv3 標準化 | 仕組みで選ぶ | 評価基準・RFP・SLA・棚卸しのプロセスが整い、選定と契約が同じ物差しで回る |
| Lv4 最適化循環 | 戦略的に管理 | ベンダー評価・コスト・リスク・出口戦略が継続的に回り、依存度に応じてポートフォリオを最適化する |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（契約の台帳化と期限管理）を固める**こと。契約の全体像が見えなければ、コスト棚卸しもSLA管理も成り立たない。
