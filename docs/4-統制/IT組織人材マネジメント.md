# IT組織・人材マネジメント

## 概要

社内ITの業務を担う人と組織をつくり、育て、維持する領域。採用・育成・スキル管理、推進体制の設計、そして全社員のITリテラシー（ITを理解し使いこなす力）の底上げまでを含む。どれほど優れた戦略や仕組みも、それを動かす人と組織がなければ機能しない。運用・改善・変革・他の統制すべてを支える土台である。

> 枝＝`統制 > IT組織・人材マネジメント` ／ フェーズ軸＝**策定 → 運用 → 評価 → 見直し**（年次サイクル＋採用・育成サイクル）

---

## あるべき姿

- **一人に依存していない** … 特定個人がいなくなると止まる業務がない。重要な業務は複数人が担え、知識が個人の頭ではなく組織に残っている
- **必要なスキルが見えている** … 今後やるべきことに対し、どんなスキルが・どれだけ・誰に必要かが整理され、現状との差（スキルギャップ）が把握できている
- **計画的に育っている** … ギャップを埋める育成・採用が場当たりでなく計画として動いている。育成は研修をやって終わりにせず、現場での行動が変わったかまで見ている
- **内製と外注の線引きがある** … どの業務を自社で担い（内製：外注せず自社で担う体制）、どこを外部に委託するかが意図して決まっている。なんとなく全部外注、なんとなく全部抱え込みになっていない
- **全社のITリテラシーが底上げされている** … IT部門だけでなく全社員が、業務に必要なITスキルとセキュリティ意識を一定水準で持っている。現場が弱点になっていない
- **担い手が確保されている** … 既存の運用を支える人材と、変革・新領域を担う人材の双方が確保・育成され、採用難・人材流出にも備えがある

## 基本方針（汎用）

会社や事業が違っても、IT組織・人材マネジメントの土台となる考え方は共通する。

- **属人化を放置しない** … 一人に依存する体制は、その人の休暇・退職・体調で業務が止まる重大なリスクである。重要業務は手順を残し、複数人が担える状態（多能工化）を意図してつくる。「あの人しか分からない」を成果ではなく負債として扱う
- **必要なスキルを起点に逆算する** … いま誰がいるかから考えるのではなく、戦略・ロードマップが求めるスキルから逆算する。必要スキルと現状の差（ギャップ）を可視化し、育成と採用で埋める計画につなげる
- **内製と外注は目的で決める** … コア（自社の競争力・継続的な知見蓄積が要る領域）は内製に寄せ、ノンコア・一時的な需要は外注に寄せる。コスト・スピード・知見の残し方を天秤にかけ、意図して配分する。丸投げは知見が社内に残らず、抱え込みは人が枯れる
- **教育は一過性にしない** … 研修・eラーニングは「やって満足」で終わらせない。知識の定着と現場での行動変容まで追い、繰り返し・更新していく。一度きりの講習で意識は定着しない
- **全社のITリテラシーに責任を持つ** … 情報漏洩・標的型攻撃などの事故は、技術の穴より人の行動に起因することが多い。IT部門だけが強くても現場が弱ければ事故は起きる。全社員のリテラシーとセキュリティ意識の底上げを自部門の責務と捉える
- **既存運用と新領域の両輪で人を育てる** … 今あるものを止めない人材と、変革・DX（デジタル技術による変革）を担う新領域の人材は、求められる力が異なる。どちらかに偏らず、両輪で確保・育成する
- **人事制度の設計は引き取らない** … 採用実務・評価制度・報酬制度の設計は人事部門の主管である。IT部門は「どんな人材・スキルが要るか」の要件提示と育成で連携する。制度づくりそのものを抱え込まない

## 主な業務

### 採用・育成・スキル管理

IT業務を担う人材を確保し、必要なスキルへ育て、その状態を管理する。組織の足腰をつくる中核。

- 戦略・ロードマップが求めるスキルを定義し、現有スキルとの差（スキルギャップ）を可視化する。スキルマップ（誰が何をどの水準でできるかの一覧）で見える化すると、育成・採用・配置の判断材料になる
- ギャップを埋める手段を、育成（既存メンバーのスキルアップ・リスキリング：新しいスキルの学び直し）と採用（外部からの確保）に振り分け、計画として動かす
- 属人化を解消する。重要業務は手順化・ドキュメント化し、副担当を置く（多能工化）。一人に依存する業務を棚卸しし、優先して分散させる
- 採用は人事部門と連携し、IT部門は求める人物像・スキル要件・技術面の見極めを担う。採用実務（募集・選考プロセス・条件提示）の主管は人事に置く
- 人材流出に備える。知識を個人ではなく組織に残し、待遇・成長機会・働き方で定着を図る。流出を前提に、抜けても回る冗長性を持たせる
- 内製と外注の配分を見直す。委託先に依存しすぎて社内に知見が残らない状態を避け、コアは社内に蓄積する

### 推進体制・組織設計

社内ITの仕事を「誰が・どんなチームで・どう連携して」進めるかの体制を設計し、維持する。

- 運用・改善・変革・統制それぞれを担う役割と責任の所在を明確にする。誰が何に責任を持つか（役割分担）が曖昧だと、抜け漏れと押し付け合いが起きる
- 変革・DXを担う体制と、既存運用を支える体制の双方を設計する。新領域に人を割いた結果、運用が手薄になる事態を避ける
- 社外の専門人材・委託先を含めた体制（内製と外注の組み合わせ）を設計する。どこを誰が担うかの全体像を描く
- 経営層・事業部門・人事・セキュリティなど他部門との連携窓口と意思決定の流れを定める。IT部門だけで完結しない業務の橋渡しを設計する
- 組織は固定ではない。事業や戦略の変化に合わせて体制を見直す。一度組んだ体制を見直さず形骸化させない
- 体制づくりの方向性は `統制 > IT戦略・企画` と、案件横断の進め方は `統制 > プロジェクト・ポートフォリオ管理（PMO）` と整合させる

### 全社ITリテラシー教育

IT部門内にとどまらず、全社員のITリテラシーとセキュリティ意識を一定水準まで引き上げる。情シスだけ強くても現場が弱ければ事故は起きる、を前提にした全社施策。

- 全社員向けに、業務に必要な基礎ITスキルとセキュリティの基本動作（パスワード管理・不審メールの扱い・情報の取り扱い）を教育する。標的型攻撃・情報漏洩は人の行動に起因することが多く、現場の底上げが最大の防御になる
- 教育は一過性にしない。入社時研修・定期研修・標的型攻撃の模擬訓練（実際に近い形で疑似攻撃メールを送り反応を見る訓練）などを組み合わせ、継続的に行う
- 「受講させて終わり」にしない。理解度の確認、現場での行動変容（実際に行動が変わったか）まで追い、効果の低い内容は見直す
- 部門・職種ごとに必要なリテラシーは異なる。一律ではなく、対象に合わせた内容にする
- 新しいツール・脅威・働き方（リモートワーク・生成AI利用など）の変化に合わせ、教育内容を更新し続ける
- セキュリティ規程そのものの策定は `統制 > ガバナンス・統制` が主管する。本領域はその規程を全社員の理解・行動へ落とし込む教育を担う

## フェーズ別の進め方

統制のフェーズ軸（策定 → 運用 → 評価 → 見直し）に沿って、各段でやること・成果物・つまずきを示す。年次サイクル（体制・育成方針）と採用・育成サイクル（個別の確保・育成）が並走する。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 策定 | 必要スキルとギャップを洗い出し、育成・採用・体制・全社教育の方針と計画を定める | スキルマップ・育成計画・体制図・教育計画 | 現状の人員ありきで考え戦略から逆算しない／属人化リスクを棚卸ししない |
| 運用 | 計画に沿って育成・採用・体制運営を進め、全社教育を実施する | 研修記録・採用実績・体制の稼働・教育受講状況 | 育成が日々の業務に押されて進まない／教育が受講させて終わりになる |
| 評価 | スキルの伸び・属人化の解消度・教育の行動変容・離職や採用の状況を測る | スキル充足状況・多能工化の進捗・教育効果・離職率 | 受講率だけ見て行動変容を見ない／属人化が解消したか検証しない |
| 見直し | 評価結果と事業・技術・脅威の変化を踏まえ、スキル要件・体制・教育内容を更新する | 改訂版スキル要件・体制・教育計画 | 一度組んだ体制を見直さず形骸化する／変化に教育内容が追従しない |

> 統制のフェーズは一過性ではなく回り続ける輪である。人材・組織は短期では変わらない。評価から見直しへ確実につなぎ、毎年更新し続けることが、属人化の再発と教育の形骸化を防ぐ要となる。

## やらないこと

IT組織・人材マネジメントは「人に関わることは何でもIT部門が」と抱え込みやすい。**あえてやらない・抱え込まないこと**を明確にし、本来の役割に集中する。

- **全社の人事制度の設計** … 評価制度・報酬制度・等級・採用実務の主管は人事部門である。IT部門は求める人物像・スキル要件の提示と育成で連携する。制度づくりそのものを引き取らない
- **採用実務そのものの代行** … 募集・選考フロー・条件提示・労務手続きは人事が担う。IT部門は技術面の見極めと要件定義に集中する。採用の事務まで抱え込まない
- **セキュリティ規程そのものの策定** … 規程・ポリシーの策定は `統制 > ガバナンス・統制` の領域。本領域はその規程を全社員の理解と行動へ落とし込む教育を担う。規程本体は引き取らない
- **特定個人への依存を放置すること** … 「できる人がいるから回っている」状態を成果と見なさない。属人化は負債として扱い、手順化・多能工化で計画的に解消する
- **教育をやって終わりにすること** … 研修・eラーニングの実施自体を目的化しない。理解度・行動変容まで追わない教育は、コストをかけて何も変えない結果になる
- **個別案件の構築・運用そのもの** … 人材・組織を整えるのが役割であり、運用・改善・変革の実装そのものは各枝が担う。体制を整えることと、現場の作業を引き受けることを混同しない

> ここで線を引くのは、本領域を「ITの担い手と全社のリテラシーを育てること」に集中させ、制度設計・実務・実装を本来の担い手へ正しく渡すためのもの。抱え込むほど、肝心の人と組織を育てる余力が削れる。

## よくある課題

- **属人化・特定個人依存** … 重要業務が一人に集中し、休暇・退職で止まる。本人も抜けられず、組織も育たない悪循環になる
- **スキルギャップが見えない** … 必要なスキルと現状の差が把握されておらず、育成も採用も場当たりになる
- **目の前の運用に追われ育成が進まない** … 火消しと日常業務で手一杯になり、育成・スキル管理が後回しになり続ける
- **採用難・人材流出** … 求めるIT人材が採れない・採れても定着しない。知見が個人とともに流出する
- **内製と外注のバランスが崩れる** … 丸投げで社内に知見が残らない、あるいは抱え込みすぎて人が枯れる。意図せず偏る
- **現場のリテラシーが弱い** … 情シスが強くても現場が弱く、不審メール・誤送信・私物利用など人起因の事故が起きる
- **教育の形骸化** … 研修を毎年やっているが受講させるだけで、行動が変わらず事故が減らない

## ベストプラクティス

- **スキルマップによる可視化** … 誰が何をどの水準でできるかを一覧化し、必要スキルとの差（ギャップ）を見える化する。育成・採用・配置の判断を勘から数字へ移す
- **計画的なリスキリング・育成** … ギャップを埋める育成を年次計画に組み込み、日常業務に押し流されないよう時間を確保する。外部研修・資格・OJT（実務を通じた育成）を組み合わせる
- **多能工化と手順化で属人化を解消** … 重要業務に副担当を置き、手順をドキュメント化する。知識を個人の頭から組織のナレッジへ移す。属人化業務の棚卸しを定期的に行う
- **内製と外注の意図的な配分** … コア業務（競争力・継続的知見が要る領域）は内製に寄せ、ノンコア・一時需要は外注に寄せる。委託先依存度を把握し、知見が社内に残る形にする
- **全社教育の継続実施と効果測定** … 入社時・定期研修・標的型攻撃の模擬訓練を組み合わせ、継続的に行う。受講率だけでなく、理解度テスト・訓練の開封率や報告率など行動変容で効果を測り、内容を改善する
- **変革・DX人材の確保・育成** … 既存運用人材と新領域人材を両輪で育てる。社外の専門人材の活用と社内育成を組み合わせ、新領域の担い手を計画的に確保する
- **人事部門との連携の型化** … スキル要件の提示・採用での技術見極め・育成計画について、人事との連携プロセスを定例化する。役割分担（制度は人事、要件と育成はIT）を明確にする

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 個人頼み | 属人運用 | 必要スキルも体制も明文化されず、業務は特定個人の力量で回っている。一人欠けると止まる。全社教育も場当たり |
| Lv2 体制化 | 役割が見える | 役割分担・体制が文書化され、重要業務に副担当が置かれ始める。全社向けの基本的なセキュリティ教育が定期実施される |
| Lv3 計画化 | スキルで動かす | スキルマップでギャップが可視化され、育成・採用が計画として動く。内製と外注の配分が意図して決まる。全社教育の効果を測り内容を見直している |
| Lv4 循環・適応 | 育つ組織 | スキル要件・体制・教育を年次で見直す循環が回り、属人化が継続的に解消される。変革・DX人材を含め、事業の変化に人と組織が追従できる |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（体制と役割の文書化、副担当の設置）を固める**こと。誰が何を担うかが曖昧なままでは、スキル管理も育成計画も土台を持たない。
