# プロジェクト・ポートフォリオ管理（PMO）

## 概要

全社のIT案件群（ポートフォリオ）を俯瞰し、限られたヒト・カネ・時間をどの案件に配分するかを決め、横断で進捗・品質・リスクを見続ける統括業務。個別案件を進めるプロジェクトマネジメントの実務とは別に、案件群全体を一段上から見て、全社最適の取捨選択と可視化を担う。PMO（Project Management Office、プロジェクト管理を支援・統括する機能）が中核を担う。

> 枝＝`統制 > プロジェクト・ポートフォリオ管理（PMO）` ／ フェーズ軸＝**策定 → 運用 → 評価 → 見直し**

---

## あるべき姿

- **全体が一枚で見える** … 走っている案件・止まっている案件・これからの案件が、一覧で俯瞰できる。「今いくつ動いているか」を誰も答えられない状態を脱している
- **優先順位の根拠がある** … どの案件を進め、どれを後回し・撤退とするかが、戦略との整合・投資対効果・リソース制約で説明できる。「声の大きい人順」で決まらない
- **リソースの限界を直視できる** … やりたい案件は無限、ヒト・カネ・時間は有限という前提に立ち、配分のトレードオフを表に出して合意している
- **遅延・炎上が早く見つかる** … 横断の可視化により、案件の遅れ・予算超過・品質劣化の兆候を、手遅れになる前に拾える
- **進め方に型がある** … 案件の立ち上げ・報告・終結の最低限の型（テンプレート）が共有され、案件ごとに一から手探りしない
- **管理が軽い** … 標準化が現場の足かせになっていない。報告のための報告・形だけのゲート審査で各プロジェクトマネージャー（PM：プロジェクト管理者）の時間を奪っていない

## 基本方針（汎用）

会社や使うツールが違っても、ポートフォリオ管理の土台となる考え方は共通する。

- **個別最適でなく全社最適** … 一案件が「自分は重要だ」と主張するのは当然である。PMOは各案件の外に立ち、全社の戦略・リソースから見た優先順位を判断する。部分の合計が全体の最適にならないことを前提に置く
- **有限のリソースを配分する** … 案件を足すなら、何かを止めるか後回しにする。すべてを同時に最優先にはできない。配分は「やる／やらない」「今やる／後でやる」の選択であると割り切る
- **進捗は止めるためでなく助けるために見る** … 横断の可視化は、遅延や炎上を早く見つけて手を打つためにある。粗探しや責任追及のための監視にすると、現場が悪い情報を隠し始め、可視化そのものが壊れる
- **標準化は軽く、例外を許す** … 型は最低限にとどめ、小さな案件まで重い手続きで縛らない。案件の規模・リスクに応じて管理の軽重を変える（テーラリング：案件に合わせて手続きを調整すること）
- **横断の調整役に徹する** … PMOは個別案件のPMを奪わない。各案件の実務は各PMに任せ、PMOは案件をまたぐ可視化・調整・標準化に集中する
- **権限と実態を伴わせる** … 優先順位や撤退を決められる裏づけ（経営の後ろ盾・予算への関与）がなければ、PMOは集計するだけの存在になる。可視化した情報が意思決定に使われる経路を確保する

## 主な業務

### 案件の優先順位づけ・予算配分

「どの案件を、どの順で、いくらの予算で進めるか」を全社視点で決め、ヒト・カネ・時間を配分する。**個別最適の積み上げでなく、全社最適の取捨選択**が役割である。

- 全社のIT案件を一覧化し、要望・構想段階のものも含めて「案件の母集団（ポートフォリオ）」として見える化する
- 各案件を戦略との整合・投資対効果・リスク・必要リソースなどの共通のものさしで評価し、優先順位をつける
- 進める案件を選ぶことは、進めない案件を選ぶことでもある。後回し・見送り・**撤退（中止）の判断**も正面から扱う。沈んだコスト（既に使った費用）に引きずられて、続けるべきでない案件を続けない
- 限られたリソースをどの案件に充てるかを配分し、過剰に詰め込まない。人を多くの案件へ薄く割り当てる掛け持ちの常態化は、全体の遅延を生む
- 投資方針・戦略そのものは決めない。それは `統制 > IT戦略・企画` の領域であり、PMOはその方針を案件群への配分へ翻訳する立場である

### 全社横断の進捗・品質管理

個別案件のPMが各案件を回す一方で、PMOは**案件をまたいで進捗・品質・リスクを俯瞰**し、全体の健全性を保つ。

- 各案件の進捗・予算消化・課題・リスクを共通の様式で集約し、ポートフォリオ全体の状況を一枚で把握する
- 遅延・予算超過・品質劣化の兆候を早期に捉え、手遅れになる前に支援・エスカレーション・優先順位の組み替えにつなげる
- 案件間の依存関係・リソースの取り合い・成果物の重複を横断で調整する。個別案件のなかからは見えない衝突を見つけるのがPMOの価値である
- 案件の立ち上げ・報告・終結の型（テンプレート・節目の確認観点）を整え、各PMが一から手探りしない土台を提供する
- QCD（Quality・Cost・Delivery、品質・コスト・納期）のバランスを全社の観点で見る。一案件の納期死守が他案件を圧迫していないかを俯瞰する
- 個別案件の中身（実装・要件・PM実務）には踏み込みすぎない。現場のPMを信頼し、横断で必要な情報の吸い上げと調整に徹する

## フェーズ別の進め方

統制のフェーズ軸（策定 → 運用 → 評価 → 見直し）に沿って、各段でやること・つまずきを示す。年次の予算・計画サイクルと、随時持ち込まれる案件のサイクルが重なって回る。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 策定 | 案件を棚卸しし、評価基準と優先順位・予算配分・管理の型を決める | 案件一覧・優先順位・配分計画・進行テンプレート | 評価基準が曖昧で「声の大きい人順」になる／全案件を盛り込み配分が破綻する |
| 運用 | 各案件の進捗・予算・リスクを横断で集約し、調整・支援する | ポートフォリオ状況レポート・課題／リスク一覧 | 報告のための報告で現場を疲弊させる／集めるだけで調整・支援に動かない |
| 評価 | 案件・ポートフォリオ全体の成果と健全性を点検する | 進捗・効果のレビュー結果・撤退／継続の判定 | 始めた案件を惰性で続ける／沈んだコストに縛られ撤退を決められない |
| 見直し | 優先順位・配分・管理の型を組み替え、次サイクルへ反映する | 配分の見直し案・改善された標準・テンプレート | 見直しが形だけで配分が変わらない／管理を足すばかりで軽くしない |

## やらないこと

PMOは「全プロジェクトを仕切る司令塔」と誤解され、何でも抱え込みやすい。**あえてやらない・抱え込まないこと**を明確にして、横断・統括の役割に集中する。

- **個別案件のPM実務を奪うこと** … 各案件の実装・要件定義・進行管理は、その案件の担当（`運用`／`改善`／`変革` の各ノード）とPMが担う。PMOが現場のPMを兼ねて案件を握り込むと、横断の視点を失い、現場の自律も損なう
- **戦略・投資方針そのものを決めること** … 中長期戦略・投資判断は `統制 > IT戦略・企画` の領域。PMOはその方針を案件群への配分・実行へ翻訳する立場であり、方針自体を作る役ではない
- **管理のための管理で現場を縛ること** … 全案件に一律の重い手続き・大量の報告様式を課さない。案件の規模・リスクに合わせて管理を軽くする。型は現場を助けるためにあり、型を守らせること自体が目的化してはならない
- **形だけのゲート審査で足を止めること** … 通過が前提の儀式的な承認会議で各PMの時間を奪わない。節目の確認は、判断材料がそろい、止める・組み替える選択が実際に起こりうる場に限る
- **悪い情報の通報係になること** … 遅延・炎上を吊し上げて責任を追及する立場に立たない。それを始めると現場が情報を隠し、可視化が機能しなくなる。見つけた問題は支援とエスカレーションへつなぐ
- **集計だけの「ただの集計係」に留まること** … 状況を集めて表にするだけで、優先順位の組み替え・撤退・調整に踏み込まないなら、PMOを置く意味は薄い。可視化を意思決定につなげる経路を確保する

> ここで線を引くことは「役割の放棄」ではない。**横断の可視化・調整・標準化という本来の価値に集中し、現場のPMと統制の各機能に正しく仕事を委ねる**ためのもの。抱え込むのではなく、全体を見て交通整理する。

## よくある課題

- **全体像が誰にも見えない** … 何件が同時に走り、どれが遅れているかを答えられる人がいない。案件が各部署・各PMに散在している
- **優先順位が政治で決まる** … 評価基準がなく、役職や声の大きさで案件の順番が決まる。重要でない案件にリソースが流れる
- **撤退できない** … 始めた案件が惰性で続く。既に使った費用が惜しくて、見込みのない案件を止められない
- **掛け持ちの常態化** … 一人が多くの案件を薄く掛け持ちし、全体が少しずつ遅れる。リソースの限界を直視できていない
- **報告のための報告** … 横断管理が重い報告様式と会議に化け、各PMが資料作りに追われる。可視化のはずが現場の負担源になる
- **権限なきPMO** … 優先順位も撤退も決められず、状況を集計するだけ。意思決定に絡めず形骸化する

## ベストプラクティス

- **案件の一覧化（ポートフォリオの可視化）** … 全案件を一覧で管理し、状態（構想・進行中・保留・完了・中止）・予算・担当・優先度を一目で見えるようにする。まず「全部が一枚で見える」を作る
- **共通の評価基準** … 戦略整合・投資対効果・リスク・必要リソースなど、案件を同じものさしで採点する基準を決める。判断の属人性と政治性を下げる
- **優先順位とリソースの突き合わせ** … 優先順位だけでなく、使えるリソースの総量と並べて「ここまでしか同時にできない」を可視化する。線を引いた下の案件は後回し・見送りと明示する
- **節目での継続・撤退判断** … 案件の区切りで「このまま続けるか・組み替えるか・止めるか」を見直す（ステージゲート：段階ごとに継続可否を判断する仕組み）。沈んだコストでなく、これから先の見込みで判断する
- **横断ダッシュボード** … 進捗・予算消化・リスクを共通様式で集約し、遅延・超過の兆候を早期に検知する。赤信号が手遅れになる前に灯る仕組みにする
- **管理のテーラリング** … 案件の規模・リスクに応じて手続きの重さを変える。小さく低リスクの案件は軽い型で回し、大きく高リスクの案件に管理を厚くする
- **PMO機能の支援的運営** … テンプレート提供・データ集約・調整の支援に徹し、現場のPMの自律を尊重する。集めた情報が優先順位・配分の意思決定に使われる経路を経営とつなぐ

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 案件分散 | 各自バラバラ | 案件が各部署・各PMに散在し、全社で何が動いているか誰も把握していない |
| Lv2 一覧化 | 見える化 | 全案件が一覧で集約され、状態・予算・担当が俯瞰できる。横断の進捗が見える |
| Lv3 優先順位づけ | ものさしで選ぶ | 共通の評価基準で優先順位・予算配分が決まり、リソース制約と突き合わせて取捨選択している |
| Lv4 ポートフォリオ最適 | 全社最適で回す | 節目で継続・撤退を判断し、配分を組み替え続ける。可視化が意思決定に直結し、管理の軽重も最適化されている |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（案件の一覧化）を固める**こと。全体が見えなければ、優先順位づけも撤退判断も成り立たない。一方で、可視化を意思決定につなげる権限の裏づけがないと、Lv3 以降は形骸化する。
