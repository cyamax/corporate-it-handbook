# データガバナンス

## 概要

全社のデータを資産として正しく管理するための規律。データの定義・品質・所在・取り扱いのルールを定め、安心してデータを使える土台を整える業務。活用して価値を生む側ではなく、その活用を支える管理方針・品質基準・連携基盤の方針を整える側に立つ。

> 枝＝`統制 > データガバナンス` ／ フェーズ軸＝**策定 → 運用 → 評価 → 見直し**

> データガバナンス（データを全社で適切に管理する仕組み）は、データを「活用する」業務とは役割が異なる。データから新たな価値を生み出すのは `変革 > データ活用による新価値創出` が担い、本ノードはその活用が成り立つ前提となるルール・品質・基盤方針を整える。両者は対立せず、土台と活用の関係にある。

---

## あるべき姿

- **言葉の定義が揃っている** … 「売上」「顧客」「稼働率」が部署ごとに違う意味で使われていない。全社共通の定義（メタデータ：データの定義・属性情報）が決まっている
- **基準となるデータが一つ** … 顧客・商品などの基準データ（マスタデータ：全社共通の基準データ）が重複・不整合なく、どこを見ても同じ値になっている
- **データの所在と意味が分かる** … どこに何のデータがあり、何を意味するかが一覧（データカタログ：社内データの所在・定義の一覧）で辿れる。担当者の頭の中にしかない状態になっていない
- **誰が何を見てよいか決まっている** … 機密区分とアクセス権限の方針が定まり、個人情報や機微なデータが無秩序に閲覧・持ち出しされない
- **活用を促す規律である** … ルールが活用の邪魔ではなく後押しになっている。安全に使える範囲が明示され、現場が萎縮せずデータを使える
- **品質が伴っている** … 方針が文書にあるだけでなく、実際のデータの正確性・最新性・網羅性が担保されている

## 基本方針（汎用）

会社や扱うデータが違っても、データガバナンスの土台となる考え方は共通する。

- **活用のためのガバナンス** … 規律は活用を窒息させるためではなく、安心して活用できるようにするためにある。守ること自体が目的化し、使えないほど縛るのは本末転倒である。規制と活用のバランスを常に意識する
- **定義の統一を最優先に** … 品質も連携も、まず「言葉の意味が揃っていること」から始まる。同じ語が部署で別物を指す状態では、どんな集計も信頼できない
- **基準データは一元管理** … 顧客・商品コードのような全社共通データは、正の源（マスタ）を一つに定め、各システムはそこを参照する。各所で勝手に持つと必ずずれる
- **所在を見える化する** … 何のデータがどこにあるかが分かる状態を保つ。属人化・サイロ化（部署ごとにデータが孤立し、つながらない状態）を放置しない
- **機密区分で扱いを分ける** … すべてのデータを一律に守るのではなく、機密度で区分し、区分ごとに扱い・権限・保管のルールを変える。守るべきものに資源を集中する
- **ルールと品質を両輪で** … 方針を作るだけで終わらせない。実データの品質が伴わなければ「絵に描いたデータ戦略」になる。定めたルールが守られ、品質が測られている状態まで持っていく
- **オーナーを決める** … データの種類ごとに責任者（データオーナー：そのデータの定義・品質に責任を持つ業務部門側の担当）を置く。情シスだけで品質は守れない。定義や正しさは業務部門が持ち、情シスは仕組みと横断調整を担う

## 主な業務

### データ管理方針・品質

データを資産として扱うための**ルールと品質基準を定め、維持する**業務。データガバナンスの中核にあたる。

- 全社共通の用語・定義を統一する。「売上」「顧客」「アクティブユーザー」などが部署で別の意味になっている状態を正し、メタデータ（データの定義・属性情報）として明文化する
- マスタデータ（顧客・商品など全社共通の基準データ）を管理する。コードの重複・表記ゆれ・不整合を解消し、正の源を一つに定める
- データの所在と意味を見える化する。データカタログ（社内データの所在・定義の一覧）を整備し、どこに何があるかを辿れるようにする。属人化・サイロ化を解く
- データ品質の基準を定める。正確性・最新性・網羅性・一貫性といった観点で「良いデータとは何か」を決め、満たせているかを定期的に点検する
- アクセス権限・機密区分・個人情報の取り扱い方針を定める。誰がどのデータを見てよいか、どこまで持ち出せるかを区分ごとに決める。ただし権限・アカウントの実体管理は `運用 > セキュリティ運用` が担い、本ノードは「データ観点の区分・方針」を定める側に立つ

### データ・連携基盤の方針

データを全社で活かすための**基盤と連携のあり方を方針として示す**業務。基盤を日々動かすことではなく、どうつなぐか・どう貯めるかの設計思想を定める。

- データをどこに集約し、どう連携させるかの方針を定める。システムごとに孤立したデータをつなぐ考え方（データを集約する基盤、システム間の連携方式など）の標準を示す
- 連携のルールを統一する。データの受け渡し方・更新タイミング・正の源の決め方を揃え、二重入力や食い違いを防ぐ
- 全社のシステム構成・技術標準と整合させる。基盤の方針は `統制 > 全社アーキテクチャ` と足並みを揃え、データだけが独自方向に進まないようにする
- 基盤の方針を示すところまでが本ノードの役割。決まった基盤を実際に構築・運用・監視するのは `運用 > インフラ運用` が担う。設計思想と日常運用の線引きを保つ

## フェーズ別の進め方

統制のフェーズ軸（策定 → 運用 → 評価 → 見直し）に沿って、各段でやること・つまずきを示す。統制は単発ではなく、この4段を年次で回し続けるサイクルである。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 策定 | 用語定義・品質基準・機密区分・連携方針を定め、データオーナーを決める | データ管理方針・用語定義集・連携基盤の方針 | 現場を巻き込まず情シスだけで作り、実態に合わない／活用を縛りすぎる |
| 運用 | 定めた方針に沿ってマスタ・カタログを整備し、品質を保ちながら使ってもらう | データカタログ・マスタ・運用記録 | 方針が文書のまま運用されない／品質が伴わず形骸化する |
| 評価 | 品質指標・定義の浸透度・違反やずれを測り、活用が進んでいるかを点検する | 品質レポート・棚卸し結果 | 指標を測らず「うまくいっている気がする」で済ます |
| 見直し | 評価結果と事業・活用側の要望を踏まえ、方針・基準・区分を改訂する | 改訂版方針・次期計画 | 一度作って放置／厳しすぎる規律を見直さず活用を阻害し続ける |

## やらないこと

データガバナンスは「データに関わるものすべて」を抱え込みやすい。**あえてやらない・抱え込まないこと**を明確にし、活用を支える本来の役割に集中する。

- **データを活用して価値を生むこと** … 分析・予測でデータから新たな価値を引き出すのは `変革 > データ活用による新価値創出` が担う。本ノードはその土台となるルール・品質・基盤方針を整える側であり、活用そのものは行わない。両者は土台と活用の関係にある
- **基盤の日常運用** … データ基盤を実際に構築し、動かし、監視・バックアップするのは `運用 > インフラ運用` が担う。本ノードは「どうつなぐか・どう貯めるか」の方針を示すところまでで、稼働後の運用は持たない
- **全社の規程・コンプライアンス全般** … ITポリシー・内部統制・個人情報コンプライアンスの全般は `統制 > ガバナンス・統制` が担う。本ノードはその中のデータに特化した部分（定義・品質・データの機密区分方針）を受け持つ。全社規程を一から作る役ではない
- **権限・アカウントの実体管理** … 誰のアカウントにどの権限を付けるかの実運用は `運用 > セキュリティ運用` が担う。本ノードは「どのデータを誰に見せてよいか」のデータ観点の区分・方針を定める側に立つ
- **活用を窒息させるほどの統制** … 安全のためと称してすべてを許可制にし、現場がデータに触れられない状態を作らない。規律は活用を後押しするためにある。守ることの自己目的化を避ける
- **品質を伴わないルール作り** … 方針文書だけ立派で実データの品質が放置された「絵に描いたデータ戦略」にしない。ルールは品質が伴って初めて意味を持つ

> ここで線を引くことは役割放棄ではない。**活用・運用・全般統制それぞれの担い手へ正しく渡し、データガバナンスを「土台を整える」ことに集中させる**ためのもの。

## よくある課題

- **定義のばらつき** … 「売上」「顧客」が部署ごとに違う意味で使われ、全社で数字を合わせると合わない
- **マスタの不整合** … 顧客・商品コードが重複・表記ゆれし、同じ取引先が別人として登録されている
- **サイロ化・属人化** … データが部署・システムごとに孤立し、どこに何があるかが特定の担当者しか分からない
- **絵に描いたデータ戦略** … 立派な方針はあるが実データの品質が伴わず、文書だけが独り歩きしている
- **厳しすぎて使えない** … 統制を強めた結果、現場がデータに触れられず、活用が止まってしまう
- **オーナー不在** … データの正しさに責任を持つ業務部門側の担当がおらず、情シスだけで品質を抱え込んで回らない

## ベストプラクティス

- **用語定義集（ビジネス用語の辞書）の整備** … 全社で使う主要な言葉の定義を一覧化し、合意を取る。定義の統一はすべての品質の起点になる
- **マスタデータ管理の一元化** … 顧客・商品などの基準データの正の源を一つに定め、各システムはそこを参照する。重複登録・二重入力を仕組みで防ぐ
- **データカタログの導入** … 社内データの所在・定義・オーナーを一覧化し、検索できるようにする。サイロ化・属人化を「見える化」で解く
- **機密区分の階層化** … データを機密度で区分し、区分ごとに権限・保管・持ち出しのルールを変える。すべてを一律に守ろうとせず、守るべきものに集中する
- **データオーナー制の導入** … データの種類ごとに業務部門側の責任者を置く。定義と品質は業務が持ち、情シスは仕組みと横断調整に徹する
- **品質指標の定期測定** … 重複率・欠損率・更新の鮮度などを定期的に測り、方針が実データに反映されているかを点検する。測らないルールは形骸化する
- **活用側との対話を継続する** … `変革 > データ活用による新価値創出` の現場と定期的に対話し、規律が活用の邪魔になっていないかを確認しながら基準を調整する

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 無管理 | 各自バラバラ | 定義が部署ごとに違い、マスタは重複だらけ。どこに何があるか誰も把握していない |
| Lv2 部分整備 | 一部にルール | 重要なマスタや用語に定義ができ始める。ただし全社では揃わず、品質も点検されていない |
| Lv3 方針確立 | 全社の規律 | 用語定義・マスタ・機密区分・連携方針が全社で整い、データカタログとオーナーが機能している |
| Lv4 活用と循環 | 土台が活きる | 品質が継続測定され、活用側の要望を受けて方針が更新される。規律が活用を後押しし、データが資産として回る |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（主要な用語・マスタの定義）を固める**こと。定義が揃わなければ、品質も連携も活用も土台を欠いたまま積み上がらない。
