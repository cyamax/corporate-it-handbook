# ガバナンス・統制

## 概要

全社ITの規律の土台を定める横断レイヤー。ポリシー・規程を策定し、内部統制・監査に応え、法令を守り、ITリスクを管理する。日々の執行（運用）とは役割が違い、ルールを「定める・整備する」側に立つことで、運用・改善・変革のすべてに一貫した規律を与える。

> 枝＝`統制 > ガバナンス・統制` ／ フェーズ軸＝**策定 → 運用 → 評価 → 見直し**

ここでの統制（ガバナンス）は、ITガバナンス（ITを全社で適切に統治する仕組み）の中核をなす。COBIT（ITガバナンスの国際的なフレームワーク）／TOGAF（全社アーキテクチャの枠組み）に対応する横断レイヤーであり、運用・改善・変革という働きかけの3区分と並ぶのではなく、その全体に重なる。

---

## あるべき姿

- **ルールが定まっている** … 何を守り・どこまで許すかが、ポリシー・規程として文書で定まっている。判断が人や場面で揺れない
- **ルールが守られている** … 規程が棚に眠らず、周知・教育・運用設計を通じて現場で実際に守られている。作って終わりにしない
- **証跡が平時から残る** … 統制が効いていることを示す記録（エビデンス）が、日々の運用の中で自然に残る。監査前に慌てて作らない
- **法令に追従できている** … 個人情報保護・各種法令・業界規制の最新動向を継続的に把握し、規程が現行法に追いついている
- **リスクが見えている** … ITに関わるリスクが網羅的に洗い出され、影響度と発生可能性で優先順位づけされ、対策の状況が一覧で見える
- **統制が事業を妨げない** … リスクの軽重に応じて統制の強さを調整し、過剰な規程で現場を縛らない。統制がブレーキでなくガードレールとして働く

## 基本方針（汎用）

会社や規模が違っても、ガバナンス・統制の土台となる考え方は共通する。

- **ルールは守られて初めて意味がある** … 規程を作ることが目的ではない。周知・教育・運用への落とし込みまでをセットで設計する。読まれない・守られない規程は、無いのと同じか、形骸化して逆に危険である
- **統制はブレーキでなくガードレール** … 統制の役割は事業を止めることではなく、安全に速く進めるための境界を引くことである。過剰な統制は現場の生産性を奪い、抜け道（シャドーIT：情シスの管理外で導入・利用されるツール）を生む
- **リスクベースで軽重をつける** … すべてを一律に厳しく縛らない。リスクアセスメント（リスク評価）で影響度と発生可能性を見積もり、高リスクには手厚く、低リスクには軽く、メリハリをつける
- **証跡が残る運用を平時から設計する** … 内部統制・監査は「記録が残っていること」が前提になる。監査の直前に証跡を駆け込みで揃えるのではなく、日々の運用の中で自然に証跡が残る仕組みを設計する
- **法令・規制は変わり続ける** … 個人情報保護法・各種ガイドライン・業界規制は改正される。最新動向を継続的にキャッチアップし、規程を定期的に見直す。一度作った規程を放置しない
- **定める側と執行する側を分ける** … ガバナンスはルールを「定める・整備する」役で、日々の権限運用やパッチ適用といった執行は運用が担う。ルールを作るが、現場の実務までは奪わない。両者の役割を混ぜると、統制も執行も中途半端になる
- **トップの関与を得る** … ポリシー・規程は経営の意思として全社に効力を持つ。情シス単独で決めて終わらせず、経営層の承認と後ろ盾を得て初めて全社の規律になる

## 主な業務

### ITポリシー・規程

全社ITの「何を守り・どこまで許すか」を文書として定める。情報セキュリティポリシー、利用規程、データ取り扱い基準などの体系を整え、全社に効力を持たせる中核業務。

- ポリシー（基本方針）・スタンダード（基準）・プロシージャ（手順）の階層で体系立てる。上位ほど変わりにくい原則、下位ほど具体的な手順とし、改定の影響範囲を分ける
- ポリシー・規程の策定はガバナンスの役割であり、その執行（権限運用・パッチ適用など）は `運用 > セキュリティ運用` が担う。**定める側と回す側を分ける**。運用が現場で独自のルールを発明しないよう、拠り所となる規程をここで用意する
- 策定して終わりにしない。周知（全社への展開）・教育（理解と納得）・運用設計（実務への組み込み）までをセットで進める。読まれない規程は形骸化する
- 規程は生き物として維持する。法令改正・事業変化・新技術の登場に合わせて定期的に見直し、現実と乖離した規程を放置しない
- データそのものの取り扱い方針・品質基準は `統制 > データガバナンス` が担う。ここでは全社共通のIT規律の枠組みを定め、データ固有の方針はそちらへ委ねる

### 内部統制・監査対応（IT統制・J-SOX）

IT統制（ITに関わる業務が正しく統制されている状態）の枠組みを整え、内部監査・外部監査・J-SOX（内部統制報告制度）に応える。財務報告の信頼性を支えるITの仕組みが、設計どおりに機能していることを示す業務。

- IT全般統制（ITGC：アクセス管理・変更管理・運用管理などの土台となる統制）が機能している状態を整え、その有効性を評価できるようにする
- 統制は「証跡が残る運用」を平時から設計するのが要点である。誰がいつ何を承認・実行したかの記録が、日々の運用の中で自然に残る仕組みを作る。監査の直前に証跡を駆け込みで集めるのは、統制が効いていない証拠でもある
- 監査対応そのもの（評価・指摘・是正のサイクル）はここが設計する。一方、運用は監査に出せる記録を残す側に徹する。`運用 > セキュリティ運用` がログ・承認記録を残すのは、この受け皿に乗せるためである
- 指摘事項は記録し、是正の責任者と期限を定めて追跡する。指摘の出しっぱなし・是正の放置をなくす

### コンプライアンス（個人情報・法令）

コンプライアンス（法令・規範の遵守）を全社ITの観点から担保する。個人情報保護、各種法令、業界規制を把握し、規程と運用がそれらに適合している状態を保つ。

- 自社が守るべき法令・規制・ガイドラインを棚卸しし、ITに関わるものを特定する。守る対象が分からなければ適合も確認できない
- 個人情報・機微情報の取り扱いは特に注意を要する。収集・保管・利用・廃棄の各段でルールを定め、漏洩・目的外利用を防ぐ。委託先・再委託先の管理も含む
- 法令・規制は変わり続ける。改正動向を継続的にキャッチアップし、規程と運用を追従させる。専門領域は法務部門・外部専門家と連携し、情シス単独で抱え込まない
- 適合状況を定期的に点検し、ずれを早期に検知する。違反は事業に直接の損害（行政処分・賠償・信用失墜）を招くため、後追いでなく予防的に管理する

### ITリスク管理（全般）

ITに関わるリスクを網羅的に洗い出し、影響度と発生可能性で評価し、優先順位をつけて対策状況を管理する。セキュリティ・コンプライアンス・可用性・委託先など、IT全般のリスクを横断的に束ねる業務。

- リスクを網羅的に洗い出す。一部の分かりやすいリスクだけでなく、抜け漏れなく棚卸しする。見えていないリスクは管理できない
- リスクアセスメント（リスク評価）で**影響度 × 発生可能性**を見積もり、優先順位をつける。すべてに同じ労力をかけず、高リスクから手厚く対処する
- 各リスクに対し、回避・低減・移転・受容のいずれかの対応方針を決め、責任者と対応状況をリスク台帳で一元管理する。台帳は定期的に更新し、新たなリスクを追加する
- 個別領域の執行（脆弱性対応・インシデント対応など）は運用が担う。ここはそれらを横断的に俯瞰し、全社のリスク状況を経営へ可視化する役割を持つ

## フェーズ別の進め方

統制のフェーズ軸（策定 → 運用 → 評価 → 見直し）に沿って、各段でやること・つまずきを示す。年次のサイクルとして回す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 策定 | リスク・法令を踏まえ、ポリシー・規程・統制の枠組みを定め、経営承認を得る | ポリシー・規程・リスク台帳・統制一覧 | 現場の実態を見ずに理想論で作る／経営の承認を得ず情シス止まりになる |
| 運用 | 規程を周知・教育し、現場の実務へ組み込み、証跡が残る形で回す | 周知記録・教育記録・日々の証跡 | 規程が棚に眠り守られない／証跡が残らず後で追えない／過剰統制で抜け道が生まれる |
| 評価 | 統制の有効性を点検し、監査に応じ、リスク・適合状況を測る | 監査結果・点検結果・指摘一覧 | 監査直前に証跡を駆け込みで揃える／指摘を記録するだけで是正しない |
| 見直し | 指摘・法令改正・事業変化を反映し、規程と統制を改定する | 改定版の規程・是正完了記録 | 一度作った規程を放置する／法令改正に追従できず現行法とずれる |

> フェーズは一巡で終わらず、年次サイクルとして回し続ける。とりわけ**策定だけで満足せず、運用・評価・見直しまで一周させる**こと。作って配って終わりの規程は、効いているように見えて効いていない。

## やらないこと

ガバナンス・統制は「ITのルールに関わること全部」を抱え込み、現場の実務まで奪いやすい。**あえてやらない・抱え込まないこと**を明確にして、定める側という本来の役割を守る。

- **日々のセキュリティ執行を抱えること** … パッチ適用・権限付与・棚卸し・インシデント初動といった日々の執行は `運用 > セキュリティ運用` が担う。ガバナンスは何を守るかの方針・規程・基準を定める側であり、現場の手を動かす執行役ではない。ルールを作るが、現場実務までは奪わない
- **権限運用そのものを代行すること** … 「この人にこの権限を与えるか」の日々の付与・失効・棚卸しは運用の領域。ガバナンスは権限管理のルール（最小権限・職務分掌などの方針）を定め、運用がそれを回す。両者を混ぜない
- **データ品質・基盤方針を抱えること** … データの取り扱い方針・品質基準・連携基盤の方針は `統制 > データガバナンス` の領域。ここでは全社共通のIT規律の枠組みを定め、データ固有の方針はそちらへ委ねる
- **過剰な規程で現場を縛ること** … リスクの軽重を無視して何でも一律に厳しく縛らない。過剰統制は現場の生産性を奪い、抜け道（シャドーIT）を生み、結果として規程が守られず形骸化する。リスクベースで軽重をつける
- **棚に眠る規程を作ること** … 周知・教育・運用設計を伴わない規程を量産しない。読まれず守られない規程は、統制が効いている錯覚を生むだけで、かえって危険である
- **ベンダー個別の契約・SLA管理を抱えること** … 委託先の選定・評価・契約条件の管理は `統制 > ベンダー・契約管理` の領域。ガバナンスは委託先管理に関わるリスク・コンプライアンスの方針を定める側に徹し、個別契約の実務はそちらへ渡す

> ここで線を引くことは「現場に丸投げする」ことではない。**ガバナンスを定める・整備するという本来やるべきことに集中させ、執行の引き渡し先と拠り所を明示する**ためのもの。ルールを作るが、現場の実務は奪わない。

## よくある課題

- **規程の形骸化** … 規程は立派だが誰も読まず守らない。周知・教育がなく、棚に眠っている
- **証跡の駆け込み準備** … 監査の直前になって証跡をかき集める。平時に記録が残る運用が設計されていない
- **過剰統制とシャドーIT** … リスクの軽重を無視して何でも厳しく縛り、現場が抜け道を探してシャドーITが横行する
- **法令改正への追従漏れ** … 法令・規制が改正されても規程が更新されず、現行法とずれたまま運用が続く
- **リスクの洗い出し漏れ** … 分かりやすいリスクだけ管理し、見えにくいリスクが抜け落ちる。台帳が作りっぱなしで更新されない
- **執行との役割の混線** … ガバナンスが現場の執行まで抱え込む、または運用が独自にルールを発明する。定める側と回す側の境界が曖昧

## ベストプラクティス

- **ポリシー体系の階層化** … 基本方針・基準・手順の3層で規程を体系立てる。上位は変わりにくい原則、下位は具体的な手順とし、改定の影響範囲を分ける
- **周知・教育とセットの展開** … 規程の策定時に、周知計画・教育（eラーニング・研修）・運用への組み込みを必ずセットで設計する。理解度の確認まで含める
- **証跡が自然に残る運用設計** … 承認フロー・操作ログ・変更記録を、ツールの仕組みで自動的に残す。人手で証跡を作るのではなく、運用すれば証跡が残る形にする
- **リスク台帳の運用** … リスクを影響度 × 発生可能性で評価し、対応方針・責任者・状況を台帳で一元管理する。定期的に棚卸しして更新する
- **統制マトリクスの整備** … 守るべきリスクと、それに対応する統制（コントロール）を対応表で可視化する。統制の抜け漏れと重複を見える化する
- **法令ウォッチの仕組み化** … 法令・ガイドラインの改正動向を継続的に追う担当・経路を決める。法務部門・外部専門家・業界団体と連携し、改正を早期に捕捉する
- **監査前提でなく平時前提の設計** … 「監査で見られるから」ではなく「日々正しく回すために」統制を設計する。結果として監査にも自然に応えられる状態を目指す
- **経営への定期報告** … リスク状況・統制の有効性・コンプライアンス適合状況を経営層へ定期的に報告し、トップの関与と意思決定を引き出す

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 場当たり | ルールなし | 明文化された規程がなく、判断が人や場面で揺れる。監査・法令対応はそのつど場当たりで、リスクは把握されていない |
| Lv2 整備着手 | 規程はある | ポリシー・規程が文書化され、リスク台帳もある。ただし周知・教育が伴わず、証跡は監査前に駆け込みで集める。棚に眠りがち |
| Lv3 標準化 | 守られて回る | 規程が周知・教育を伴って現場で守られ、証跡が平時から残る。リスクが影響度 × 発生可能性で評価され、統制と対応づいている |
| Lv4 継続改善 | 仕組みで前へ | 法令動向を継続的に捕捉し、評価・見直しのサイクルが年次で回る。リスクベースで統制の軽重が最適化され、経営にリスク状況が可視化されている |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv3（守られて回る・証跡が残る）を固める**こと。規程を作っただけ（Lv2）では統制は効いていない。守られて初めて、そして証跡が残って初めて、統制は意味を持つ。
