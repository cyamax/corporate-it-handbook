# IT戦略・企画

## 概要

会社の経営・事業戦略を、ITの中長期方針・予算・投資判断へ翻訳し、社内IT全体の進む方向を方向づける最上位の企画機能。運用・改善・変革のどの取り組みに、どれだけの資源を、どの順番で投じるかの土台を整える。個別の構築や運用そのものは行わず、全体に方向と規律を与える。

> 枝＝`統制 > IT戦略・企画` ／ フェーズ軸＝**策定 → 運用 → 評価 → 見直し**（年次サイクル＋案件サイクル）

---

## あるべき姿

- **経営とつながっている** … IT戦略が事業計画・経営目標と結びつき、IT単独で宙に浮いていない。なぜそのITをやるのかが事業の言葉で説明できる
- **絵に描いた餅にしない** … 中長期ロードマップが単年度の予算・投資判断に落ち、実行されている。方針と現実の支出が連動している
- **優先順位がついている** … やりたいこと全部ではなく、何を先にやり何を後回しにするかが明示され、判断の根拠が残っている
- **判断材料が揃っている** … 投資の可否を費用対効果・TCO（総保有コスト：導入から廃棄までの総額）・リスクで評価する物差しがあり、勘や声の大きさで決まらない
- **生きた文書である** … 戦略・ロードマップが毎年見直され、環境変化に追従する。一度作って棚に眠る形骸化を防いでいる
- **コストが見えている** … IT支出の全体像（運用維持と新規投資の内訳）が把握され、何にいくら使っているかが説明できる

## 基本方針（汎用）

会社や事業が違っても、IT戦略・企画の土台となる考え方は共通する。

- **事業に資するIT戦略であること** … ITの都合からではなく、事業が何を目指すかから逆算する。IT単独で戦略は立たない。経営・事業戦略との整合が出発点である
- **方針と予算をつなぐ** … 中長期ロードマップは、単年度の予算と投資判断に接続して初めて意味を持つ。実行に落ちない方針は絵に描いた餅になる
- **全部はやらないと決める** … 資源（人・金・時間）は有限。投資は費用対効果・TCO・リスクで評価し、優先順位をつけて取捨選択する。何をやらないかを決めることが戦略である
- **運用維持と新規投資を分けて見る** … 既存システムを止めないための支出と、新たに価値を生むための投資を区別して把握する。維持に埋もれて投資余力が見えなくなる事態を避ける
- **選択肢を整え、決めるのは経営** … 戦略・企画は経営の意思決定を代行しない。選択肢と判断材料（費用・効果・リスク・代替案）を整え、経営が決められる状態にする
- **見直しを前提に作る** … 環境は変わる。戦略は確定文書ではなく、毎年見直す前提の生きた文書として運用する。形骸化の最大の原因は「作って終わり」にすること

## 主な業務

### 中長期戦略・ロードマップ

事業計画を見据え、社内ITが数年スパンで何を目指し、どの順で実現するかの道筋を描く。

- 経営・事業戦略を起点に、ITのありたい姿（数年後の目標状態）を定義する。ITの都合からではなく事業の目標から逆算する
- ありたい姿と現状の差分を洗い出し、それを埋める取り組みを時間軸に並べてロードマップにする
- ロードマップは単年度の予算・投資判断に接続する。年度ごとに「今年は何に着手するか」へ落ちて初めて実行される
- 技術標準や全体構成の整合は `統制 > 全社アーキテクチャ` と、案件の横断的な進め方は `統制 > プロジェクト・ポートフォリオ管理（PMO）` と連携する。戦略は方向を示し、実装の標準と進捗運用は各々へ渡す

### 予算・コスト管理

IT支出の全体像を把握し、年度予算を組み、執行を管理する。戦略を金額の裏付けで支える。

- IT支出を運用維持（既存を止めないための費用）と新規投資（新たな価値を生む費用）に区分して把握する。内訳が見えないと投資余力も削減余地も判断できない
- ロードマップと整合する形で年度予算を策定し、実績との差異を追う
- コストは可視化したうえで最適化する。TCOの観点で、目先の導入費だけでなく運用・保守・更新・廃棄までの総額で評価する
- 為替・ライセンス改定・利用量増などによる支出変動を見込み、予算に織り込む

> コスト削減そのものを目的化しない。削るべきは価値を生まない支出であり、必要な投資まで一律に削ると将来の競争力を損なう。効率化（同じ価値を安く）と投資（新たな価値）を分けて議論する。

### 投資判断・システム選定

個別の投資・システム導入について、やるべきか・どれを選ぶかを評価し、優先順位をつける。

- 投資の可否は費用対効果・TCO・リスクで評価する。導入費だけでなく、運用・保守・移行・撤退まで含めた総額（TCO）で比較する
- ROI（投資対効果：投じた費用に対して得られる効果）や定量・定性の効果を見積もり、KPI（重要指標：成果を測る基準）で効果を後から検証できるようにしておく
- 候補が複数ある場合、評価軸（機能・コスト・リスク・拡張性・撤退容易性など）を先に決めてから比較する。先に結論を決めて理由を後付けしない
- 限られた予算の中で優先順位をつける。全部は通らない前提で、事業へのインパクトと緊急度で並べる
- ベンダーの選定・契約の実務は `統制 > ベンダー・契約管理` へ、技術標準との適合判断は `統制 > 全社アーキテクチャ` へ渡す。戦略・企画は「投資する価値があるか」の判断材料を整える

## フェーズ別の進め方

統制のフェーズ軸（策定 → 運用 → 評価 → 見直し）に沿って、各段でやること・成果物・つまずきを示す。年次サイクル（中長期戦略・予算）と案件サイクル（個別投資判断）が並走する。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 策定 | 事業戦略を起点にIT方針・ロードマップを描き、年度予算と投資基準を定める | IT戦略・ロードマップ・年度予算・投資評価基準 | 事業戦略と切り離してIT都合で作る／ロードマップが予算に接続せず絵に描いた餅になる |
| 運用 | 方針に沿って予算を執行し、個別投資を基準で評価・優先順位づけして判断する | 投資判断記録・予算執行状況・優先順位リスト | 基準を使わず勘や声の大きさで決める／緊急案件が割り込み戦略から逸れる |
| 評価 | 支出と投資効果をKPIで測り、ロードマップの進捗と乖離を確認する | 効果測定レポート・予実差異・進捗レビュー | 投資後の効果検証をしない／予算消化の管理だけで効果を見ない |
| 見直し | 評価結果と環境変化を踏まえ、戦略・ロードマップ・予算を翌期へ更新する | 改訂版戦略・次年度方針 | 一度作って見直さず形骸化する／変化を反映せず前年踏襲になる |

> 統制のフェーズは「受付→解決」のような一過性ではなく、回り続ける輪である。評価から見直しへ確実につなぎ、毎年戦略を更新することが形骸化を防ぐ要となる。

## やらないこと

IT戦略・企画は「ITに関わるあらゆる判断の元締め」になりやすい。**あえてやらない・抱え込まないこと**を明確にし、企画機能としての役割に集中する。

- **個別案件の実行そのもの** … 構築・導入・運用は実行しない。効率化目的の実装は `改善` 配下、新価値創出目的の実装は `変革` 配下、既存の運用維持は `運用` 配下が担う。戦略は方向を示し、実行は各枝へ渡す
- **案件横断の進捗・優先順位の実務運用** … 複数案件の日々の進捗管理・リソース調整・品質管理は `統制 > プロジェクト・ポートフォリオ管理（PMO）` が担う。戦略は大枠の方針と投資の枠を示し、運用実務はPMOへ渡す
- **技術標準・全体構成の策定** … どの技術を標準とするか、システム全体をどう構成するかは `統制 > 全社アーキテクチャ` の領域。戦略は「どこへ向かうか」を示し、「何で実現するか」の標準化は全社アーキテクチャへ渡す
- **経営の意思決定そのものの代行** … 投資の最終決定や事業の方向づけを企画の裁量で決めない。選択肢と判断材料（費用・効果・リスク・代替案）を整え、経営が意思決定できる状態をつくることが役割である
- **ベンダー選定・契約の実務** … 個別ベンダーの選定・評価・契約交渉は `統制 > ベンダー・契約管理` が担う。戦略は投資する価値があるかを判断し、選定実務は渡す
- **戦略を立てて終わりにすること** … 策定で完結させない。運用・評価・見直しまで回し、毎年更新する。作りっぱなしは最もありがちな失敗である

> ここで線を引くのは、企画機能を「方向づけと判断材料の整備」に集中させ、実装・実務・最終決定を本来の担い手へ正しく渡すためのもの。抱え込むほど戦略は薄くなる。

## よくある課題

- **事業戦略と切り離されたIT戦略** … ITの内部都合（更新時期・技術的興味）で組まれ、事業に何で資するかが説明できない
- **絵に描いた餅** … 立派なロードマップがあるが、予算・投資判断に接続せず、実行されないまま年を越す
- **優先順位がつかない** … やりたいこと全部を並べるだけで取捨選択できず、結局すべてが中途半端になる
- **効果を検証しない投資** … 導入時のROIは語るが、稼働後に効果を測らず、次の投資判断に活かせない
- **形骸化** … 一度作った戦略が見直されず、環境が変わっても前年踏襲のまま棚に眠る
- **運用維持に予算を食われる** … 既存システムの維持費が膨らみ、新たな価値を生む投資に回す余力が見えない・残らない

## ベストプラクティス

- **事業計画との突き合わせ** … 経営・事業計画を起点にIT方針を定め、各取り組みを「どの事業目標に効くか」で説明できる形にする。IT単独で戦略を語らない
- **ロードマップと年度予算の接続** … 中長期ロードマップを毎年の予算・投資計画へ落とし込む。年度ごとに「今年着手する分」が明示され、方針が支出として実行される状態にする
- **投資評価の物差しを統一** … 費用対効果・TCO・リスクの評価軸をテンプレート化し、案件ごとに同じ基準で比較する。判断のばらつきと後付けの理由付けを防ぐ
- **IT支出の可視化（維持と投資の分離）** … 運用維持の支出と新規投資の支出を区分して把握し、維持に埋もれる投資余力を見えるようにする
- **KPIで効果を後追い** … 投資判断時に効果指標（KPI）を決め、稼働後に実績を測って次の判断へ還元する。やりっぱなしにしない
- **年次の見直しサイクルを定例化** … 戦略・ロードマップ・予算の見直しを年次行事として定例化し、評価結果と環境変化を反映する。生きた文書として更新し続ける

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 場当たり | 戦略不在 | IT戦略が明文化されず、投資はその都度・要望ベース。全体方針も予算の全体像もない |
| Lv2 計画化 | 方針あり | IT方針・年度予算が文書化されている。ただし事業戦略との結びつきは弱く、ロードマップも実行に落ちきらない |
| Lv3 経営連動 | 事業と整合 | IT戦略が事業計画と整合し、ロードマップが予算・投資判断に接続。投資は共通基準（費用対効果・TCO・リスク）で評価・優先順位づけされる |
| Lv4 循環・最適化 | 生きた戦略 | 投資効果をKPIで測り、年次で戦略・ロードマップを見直す循環が回る。IT投資が事業価値に貢献していることを説明でき、ポートフォリオとして最適化されている |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（方針と予算の文書化）を固め、次に Lv3（経営との連動と投資基準の統一）へ進む**こと。文書化されていなければ、整合も見直しも始められない。
