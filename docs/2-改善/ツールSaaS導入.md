# ツール／SaaS導入（効率化目的）

## 概要

既存業務を楽に・速く・安くするために、業務ツールやSaaS（Software as a Service、インターネット経由で使うソフト）を選び、現場に導入して定着させるプロジェクト型の業務。「ツールを入れること」が目的ではなく、「業務課題を解くこと」が目的であり、その手段としてツール導入を選ぶ。

> 枝＝`改善 > ツール／SaaS導入` ／ フェーズ軸＝**発見 → 分析 → 企画 → 構築 → 定着 → 改善**

---

## あるべき姿

- **課題が起点** … 「流行っているから」「便利そうだから」ではなく、解きたい業務課題から出発する。ツールは課題に対する解の一つに過ぎない
- **要件が言語化されている** … 何を実現したいか・何が必須で何が任意かが、導入前に文章で整理されている。要件なき比較は「機能の多さ」「価格の安さ」だけの選定に陥る
- **試してから入れる** … PoC（Proof of Concept、試しに小さく試す検証）やトライアルで現場が実際に触り、効果と使い勝手を確かめてから本導入を決める
- **重複・乱立がない** … 似た機能のツールが部署ごとに別々に契約される状態を避け、既存資産で代替できないかを先に確認する
- **定着まで見届ける** … 導入して終わりにせず、使われているか・効果が出ているかを確認する。使われないSaaSの放置はコストの垂れ流しである
- **正規の導線がある** … 現場が情シスを通さず勝手に契約する状態（シャドーIT：情シスの管理外で導入・利用されるツール）にならないよう、相談・申請の入り口が用意されている

## 基本方針（汎用）

会社や扱う領域が違っても、ツール・SaaS導入の土台となる考え方は共通する。

- **ツール起点でなく課題起点** … 「このツールを入れたい」という相談が来ても、まず「どの業務の何に困っているか」へ立ち返る。課題が曖昧なまま入れたツールは使われない
- **要件を先に固める** … 比較・選定の前に、達成したいこと・必須条件・予算・運用体制を決める。要件が定まればツールは絞り込める
- **小さく試して判断する** … 大きく契約する前に、限定範囲のトライアルや無料枠で現場検証する。カタログやデモではなく、自社のデータと業務で確かめる
- **既存と重複させない** … 新規導入の前に「今あるツールで代替できないか」を必ず確認する。重複契約は費用とデータの分散を招く
- **出口（やめ方・持ち出し）を入口で確認する** … 解約時にデータを取り出せるか、他社へ移れるかを契約前に確認する。入れる時よりやめる時に困るのがSaaSである
- **導入後の定着に責任を持つ** … 「入れた」をゴールにしない。使われて初めて効果が出る。利用状況を見て、使われていなければ手を打つか撤退する
- **勝手契約を生まない導線をつくる** … 現場が困って独自にSaaSを契約してしまう前に、情シスへ相談できる入り口を用意する。禁止だけではシャドーITは止まらない

---

## 主な業務

### 課題起点でのツール要否判断

「ツールを入れたい」という相談や、現場の非効率を、**そもそもツールで解くべきかどうか**から判断する。

- 相談を「どの業務の・何に・どれくらい困っているか」へ翻訳する。ツール名の指名買いをそのまま受けない
- ツール導入以外の解（業務プロセスの見直し、既存ツールの設定変更、手順の標準化）で済まないかを先に検討する
- 既存に同種・近接の機能を持つツールがないかを棚卸しし、重複契約を未然に防ぐ
- 投資に見合う効果が見込めるか（削減できる工数・時間）を粗くでも見積もる

### 要件整理・比較・PoC（試行導入）

導入候補を**自社の業務と要件で評価し、試した上で選ぶ**。

- 必須要件・任意要件・除外条件を整理し、評価軸を先に決める。比較表は「機能の多さ」ではなく要件への適合で作る
- データ移行・既存システムとの連携・退去時のデータ持ち出し可否を、選定段階で確認項目に含める
- セキュリティ評価（データの保管場所、認証、権限管理、第三者認証の有無）を行い、リスクを見える化する
- PoC・トライアルで現場が実際に使い、効果・使い勝手・運用負荷を確かめてから本導入を判断する

### 導入・定着

決めたツールを**現場が使える状態にし、使われ続けるところまで**運ぶ。

- 初期設定・権限設計・既存システムとの連携・データ移行を行う
- 利用者への案内・手順書・初期サポートを用意し、立ち上がりの摩擦を下げる
- 導入後に利用状況と効果を確認する。使われていなければ、要因（使い方・要件のずれ・重複）を調べて手を打つ
- 定着し安定稼働に入ったツールは、定常的な運用・ライセンス管理へ引き渡す

## フェーズ別の進め方

改善のフェーズ軸（発見 → 分析 → 企画 → 構築 → 定着 → 改善）に沿って、各段でやること・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見 | 現場の非効率・相談を拾い、課題として捉える | 課題メモ・相談記録 | ツール名の指名買いをそのまま受ける／課題が曖昧なまま進む |
| 分析 | 業務と要件を整理し、ツールで解くべきか・既存で代替できるかを見極める | 要件一覧・現状業務の整理 | 要件を固めずに比較を始める／既存ツールの棚卸しを飛ばす |
| 企画 | 候補を要件で比較し、セキュリティ・移行・連携・解約条件を評価。PoCの計画を立てる | 比較表・評価結果・導入計画 | 機能の多さ・価格だけで選ぶ／解約時のデータ持ち出しを確認しない |
| 構築 | PoC・トライアルで現場検証し、初期設定・権限設計・データ移行・連携を行う | PoC結果・設定済み環境・移行データ | カタログ評価だけで本導入を決める／移行・連携でつまずく |
| 定着 | 利用案内・手順書・初期サポートで現場に根づかせる | 手順書・利用案内・利用状況 | 「入れて終わり」で使われず放置される |
| 改善 | 利用状況と効果を確認し、定着しなければ要因を潰す。安定後は運用へ引き渡す | 効果測定・引き渡し記録 | 効果を測らない／使われないまま契約が続く |

## やらないこと

ツール・SaaS導入は、ライフサイクルの他の段階や他分類の仕事を抱え込みやすい。**この枝は「効率化目的の導入プロジェクト」だけを扱う**。線引きと引き渡し先を明示する。

- **選定承認・契約そのものを担うこと** … ベンダー選定の最終承認・契約・SLA（Service Level Agreement、サービス品質の合意）の締結は統制の仕事である。導入側は要件と評価結果を渡し、可否判断と契約は `統制 > ベンダー・契約管理` へ。投資判断・選定方針は `統制 > IT戦略・企画` に従う
- **稼働後の運用・ライセンス管理を抱えること** … 定着し安定稼働に入ったツールの日常運用・バージョン追従・ライセンス管理・ベンダー連携は運用の仕事である。`運用 > アプリケーション・SaaS運用保守` へ引き渡す。導入チームが運用まで抱え続けると、次の改善に手が回らなくなる
- **新価値創出を目的とした導入を扱うこと** … 同じツール導入でも、目的が売上拡大・新規事業・新しい顧客接点なら、それは変革であり、この枝ではない。例として顧客向けの新チャネル構築は `変革 > デジタル接点・顧客体験の創出` へ。目的が効率化か新価値かで所属が変わる
- **課題なきツール導入を進めること** … 「便利そう」「他社が使っている」だけを根拠に導入を進めない。解く課題と効果見込みが言えないものは、企画段階で止める
- **現場の勝手契約を黙認すること** … シャドーITを「便利だから」と見て見ぬふりしない。リスクを記録し、正規の検討ルートへ載せる（運用中のシャドーITへの対処自体は `運用 > セキュリティ運用` と連携する）

> 線を引くのは仕事を狭めるためではない。**導入プロジェクトを「課題を解いて定着させる」ことに集中させ、選定承認・契約・稼働後運用・新価値創出を正しい行き先へ渡す**ためのものである。

## よくある課題

- **ツール起点の指名買い** … 「このSaaSを入れたい」が先に来て、解きたい課題が後付けになる。入れても使われない
- **SaaSの乱立** … 部署ごとに似た機能のツールが別々に契約され、費用が重複し、データとノウハウが分散する
- **入れて終わり** … 導入をゴールにしてしまい、定着確認をしない。使われないまま契約だけが続く（コストの垂れ流し）
- **解約時に動けない** … データ持ち出しや他社移行を契約前に確認しておらず、不満があっても乗り換えられない・データを取り出せない
- **シャドーIT化** … 情シスへの相談導線がなく、現場が困って勝手にSaaSを契約する。管理外のツールが増え、セキュリティリスクが見えなくなる
- **カタログ評価だけで決定** … デモや営業資料だけで選び、自社の業務・データで試さないまま本導入し、後から要件とのずれが発覚する

## ベストプラクティス

- **要件定義を先に書く** … 必須・任意・除外を一覧化し、評価軸を固めてから比較に入る。要件があれば候補は自然に絞れる
- **比較表は要件適合で作る** … 機能数や価格の多寡ではなく、自社要件をどれだけ満たすかで並べる。重み付けして点数化すると判断のばらつきが減る
- **PoC・トライアルを必須にする** … 限定範囲・期間を区切って現場が実データで触る。本導入の判断材料を「使った実感」で揃える
- **解約条件とデータ持ち出しを契約前に確認する** … データのエクスポート可否・形式、退去時の扱い、最低利用期間を選定段階のチェック項目に入れる
- **セキュリティ評価を標準化する** … データの保管場所・認証方式・権限管理・第三者認証の有無を定型のチェックリストで評価し、ツールごとのばらつきをなくす
- **既存ツールの棚卸しと統合** … 新規導入の前に保有ツールを一覧化し、重複は統合・整理する。乱立を「増やしてから減らす」のではなく入口で防ぐ
- **相談導線の整備** … 「ツールが欲しい時はまず情シスに相談」という入り口を用意し、勝手契約より相談が楽な状態をつくる。禁止より導線で防ぐ
- **定着の指標を決める** … 利用率・アクティブ利用者数・削減工数など、定着を測る指標を導入時に決め、一定期間後に確認する

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 場当たり導入 | ツール起点 | 相談された名前のツールをそのまま導入。課題・要件が曖昧で、定着確認もない |
| Lv2 要件・比較あり | 選び方が定まる | 要件を整理し、比較・セキュリティ評価をしてから選ぶ。試行で確かめる習慣がつく |
| Lv3 定着まで設計 | 使われる状態をつくる | PoC・解約条件確認・データ移行・定着支援が標準化され、導入が「使われる」ところまで届く |
| Lv4 ポートフォリオ管理 | 全体で最適化 | 保有ツールを一覧で管理し、重複統合・棚卸しが回る。相談導線でシャドーITが起きにくく、効果の出ないツールは撤退判断される |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（要件を固めてから選ぶ）を習慣化する**こと。課題と要件が無ければ、その上の定着もポートフォリオ管理も成り立たない。
