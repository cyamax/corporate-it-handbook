# 業務プロセス改善・標準化

## 概要

現場の業務のやり方（プロセス）そのものを見直し、ムダ・属人・重複を取り除いて、誰がやっても一定の品質・スピードで回る形に整える取り組み。ツール導入や自動化は手段の一部に過ぎず、その手前にある「業務の流れ」を効率化することが目的である。IT導入よりも一段上の、情シスが現場業務に踏み込む領域である。

> 枝＝`改善 > 業務プロセス改善・標準化` ／ フェーズ軸＝**発見 → 分析 → 企画 → 構築 → 定着 → 改善**

---

## あるべき姿

- **流れが見える** … 業務の手順・担当・所要時間が可視化され、どこにムダや滞りがあるかが共有されている
- **属人化していない** … 「あの人しかできない」業務が手順書（SOP：Standard Operating Procedure、標準作業手順）に落ち、引き継ぎ・代替が利く
- **ムダがない** … 不要な承認・重複入力・形だけの会議が排除され、本当に必要な工程だけが残っている
- **標準が守られている** … 決めた手順が現場で実際に使われ、形骸化していない。守れる範囲のルールになっている
- **効果が数字で見える** … 改善前後を時間・件数・コストで比較でき、投資判断と次の改善につながっている
- **現場が主役** … 改善が情シスからの押し付けではなく、業務部門が自分ごととして回している

## 基本方針（汎用）

会社や業種が違っても、プロセス改善の土台となる考え方は共通する。

- **見える化してから改善する** … 現状の流れを書き出さずに改善案を出さない。実態が見えていなければ、何がムダかも、改善が効いたかも判断できない
- **いきなりツールに飛びつかない** … 「このSaaSを入れれば解決する」から入らない。ムダな工程をそのまま自動化すれば、ムダが高速化するだけである。まずプロセスを直し、必要なら手段としてツールを選ぶ
- **定石（ECRS）で削る** … ECRS（Eliminate・Combine・Rearrange・Simplify＝排除・結合・交換・簡素化）の順で工程を見直す。まず「やめられないか」を問い、最後に「簡単にできないか」を考える。足す前に減らす
- **標準化は維持してこそ価値が出る** … 手順書は作って終わりではない。業務が変われば陳腐化する。更新の担当と頻度を決め、生きた状態に保つ。維持できない標準は形骸化する
- **現場と組む** … 改善は現場の協力なしに進まない。情シス単独で設計して配るのではなく、実際に手を動かす業務部門と一緒に作る。当事者が決めた手順は守られる
- **縛りすぎない** … 「標準化のための標準化」や過剰なルールで現場を窒息させない。標準は自由を奪う枷ではなく、迷いを減らす道具である。例外を認める余地を残す
- **効果を測り、回し続ける** … 改善は一度きりのイベントではない。数値で測り、定着を確認し、次のムダを見つけるサイクルにする

## 主な業務

### 現状業務の可視化

改善の出発点は、いまの業務がどう流れているかを「見える化」することである。

- 担当者へのヒアリング・観察・実データから、誰が・何を・どの順で・どれだけの時間で行っているかを書き出す
- 業務フロー図、工程表、所要時間の計測などで、頭の中の暗黙知を共有できる形にする
- 「あるべき手順」ではなく「実際にやっている手順」を捉える。建前の手順書と現場の実態がずれていることは多い
- ここを飛ばして改善案から入ると、現場感のない机上の空論になる

### ムダ・属人・重複の特定

可視化した流れから、削るべき対象を見つける。

- **ムダ** … 不要な承認段階、形だけの会議、誰も見ない資料、待ち時間・手戻り。ECRS の「排除」「簡素化」で削る
- **属人** … 特定の人しかできない・手元のExcelやマクロに依存している業務。担当の不在で止まるリスクを抱える
- **重複** … 同じ情報を複数のシステム・帳票に二重入力している、部署ごとに似た作業をばらばらにやっている
- 削減対象は「困りごと」「時間がかかっている工程」から優先する。現場が痛みを感じている所ほど協力を得やすい

### 標準化とルール化

残すべき業務を、誰がやっても同じ品質で回る形に整える。

- 手順書（SOP）として、判断基準・例外時の対処まで含めて文書化する。「分かっている人」の頭の中を取り出す
- 名称・帳票・入力ルールを統一し、部署ごとのばらつきをなくす
- ルールは「守れる粒度」にする。細かすぎる手順書は読まれず、現場が独自運用に戻る
- 標準から外れる正当な例外（顧客都合・特殊案件）を想定し、例外の扱いも決めておく

### 定着とモニタリング

決めた標準を現場に根づかせ、形骸化させない。

- 標準の周知・教育を行い、なぜ変えるのかを現場に納得してもらう。背景なしの「今日から変えます」は抵抗される
- 守られているか、効果が出ているかを継続して観察する。決めた直後だけ守られ、すぐ元に戻るのはよくある失敗
- 手順書の更新担当と見直し頻度を決め、業務変更に追従させる
- 現場から出る「やりにくい」の声を吸い上げ、標準側を直す。ルールを守らせるのではなく、守れるルールに直す

## フェーズ別の進め方

改善のフェーズ軸（発見 → 分析 → 企画 → 構築 → 定着 → 改善）に沿って、各段でやること・成果物・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見 | 困りごと・非効率の兆候を拾い、対象業務を選ぶ | 改善テーマ・対象範囲 | 現場の痛点でなく情シスの思い込みで対象を選ぶ |
| 分析 | 現状業務を可視化し、ムダ・属人・重複を特定する | 業務フロー図・課題一覧・現状の数値 | 可視化を飛ばして改善案から入る／実態でなく建前の手順を描く |
| 企画 | あるべき手順を設計し、効果目標と進め方を決める | 改善後フロー案・目標値・実行計画 | いきなりツール導入を前提にする／過剰なルールを盛り込む |
| 構築 | 新しい手順・帳票・SOPを作り、必要なら手段（ツール・自動化）を整える | SOP・新フロー・設定済みの仕組み | 現場不在で設計し、使われないものを作る |
| 定着 | 周知・教育を行い、現場の運用に乗せる | 教育記録・運用ルール | 配って終わりにする／背景説明がなく抵抗される |
| 改善 | 効果を測定し、形骸化を点検し、次の改善へ回す | 効果測定レポート・更新後SOP | 測らずやりっぱなし／元のやり方に戻っているのを見逃す |

## やらないこと

プロセス改善は「現場の業務すべてを情シスが設計し直す」方向に膨らみやすい。**あえてやらない・抱え込まないこと**を明確にして、役割を守る。

- **現場不在で情シスが設計すること** … 業務の当事者を外して「正しい手順」を机上で作り、配って終わりにしない。使われない標準を量産するだけになる。設計は業務部門と組んで行う
- **ムダをそのまま自動化すること** … 不要な工程を残したままツール化・自動化しない。まずプロセスを直す。手段としての自動化は `改善 > 手作業の自動化（RPA・スクリプト）`、ツール導入は `改善 > ツール／SaaS導入（効率化目的）` へつなぐ
- **過剰なルールで現場を縛ること** … 「標準化のための標準化」をしない。守れない・読まれない細かすぎる手順書は作らない。例外を一切認めない硬直した運用にしない
- **業務そのものの責任を引き取ること** … 改善後の業務を回す責任は業務部門にある。情シスは流れの設計と仕組み化を支援する立場であり、現場業務の実行を肩代わりしない
- **事業モデルレベルの変革に手を出すこと** … 既存業務の効率化を超えて、事業のやり方・収益モデルそのものを変える話は変革の領域。`変革 > 全社DX・変革の推進` へ渡す。効率化（改善）と新価値創出（変革）の境界を越えない
- **やりっぱなしで放置すること** … 標準化したまま更新も測定もしないと、半年で形骸化する。維持できないなら標準を作らない

> ここで線を引くのは現場を突き放すためではない。**情シスは「流れを設計し仕組みにする」役割に集中し、業務の主役は現場に残す**ためのもの。引き渡し先を明示し、正しい役割分担に導く。

## よくある課題

- **見える化を飛ばす** … 現状把握をせずに改善案から入り、現場感のない空論で終わる
- **ツールありき** … 「このSaaSで解決」から始め、ムダな工程ごと高速化してしまう
- **押し付けで抵抗される** … 情シスが決めた手順を一方的に配り、現場が独自運用に戻る
- **形骸化** … 標準化した直後だけ守られ、更新されず陳腐化し、誰も見なくなる
- **過剰なルール** … 細かすぎる手順書・承認で現場を縛り、かえって非効率になる
- **効果が測れない** … 改善前の数値を取っていないため、効いたかどうか説明できない

## ベストプラクティス

- **業務フローの書き出し** … 工程・担当・所要時間を図にし、ムダと滞りを目で見える状態にする。改善前の状態を必ず記録に残す
- **ECRSで工程を削る** … 排除・結合・交換・簡素化の順で見直す。足す前に減らす。特に「やめられないか（排除）」を最初に問う
- **現場を巻き込んだ設計** … キーパーソンを改善チームに入れ、当事者と一緒に手順を作る。当事者が決めた手順は守られる
- **SOP（標準作業手順）の整備と維持** … 判断基準・例外対応まで含めて文書化し、更新担当と見直し頻度を決めて生きた状態に保つ
- **効果の定量化** … 時間・件数・コスト・手戻り率などで改善前後を比較する。数値が次の改善と投資判断の根拠になる
- **小さく試して広げる** … 一つの業務・一部署で試行し、効果と運用しやすさを確かめてから横展開する。全社一斉導入の失敗を避ける
- **定期的な棚卸し** … 標準が実態に合っているか、形骸化していないかを定期点検し、ずれていれば標準側を直す

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 属人・無秩序 | 個人任せ | 業務のやり方が人によってばらばら。手順書がなく、担当の頭の中だけにある |
| Lv2 可視化 | 流れが見える | 業務フローが書き出され、ムダ・属人・重複がどこにあるか共有されている |
| Lv3 標準化 | 手順が揃う | SOPが整備され、誰がやっても一定品質。名称・帳票・ルールが統一されている |
| Lv4 改善循環 | 回り続ける | 効果を数値で測り、現場主導で形骸化を点検し、標準を更新し続けている。次のムダを自ら見つける |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（可視化）を固める**こと。流れが見えなければ、何を標準化すべきかも、改善が効いたかも判断できない。

## 参考資料

- ECRS（排除・結合・交換・簡素化）… 工程改善の定石。インダストリアルエンジニアリング（IE）由来の改善の着眼点
- BPR（Business Process Reengineering、業務プロセス再設計）… 業務の流れを抜本的に組み直す考え方。プロセス改善の上流の発想として
- 業務可視化・標準化（SOP・業務フロー）に関する一般的な業務改善の知見
- 関連ノード：`改善 > 手作業の自動化（RPA・スクリプト）`（プロセスを直した後の自動化）、`改善 > ツール／SaaS導入（効率化目的）`（手段としてのツール導入）、`統制 > IT組織・人材マネジメント`（推進体制・現場の巻き込み）、`変革 > 全社DX・変革の推進`（事業モデルレベルの変革との境界）
