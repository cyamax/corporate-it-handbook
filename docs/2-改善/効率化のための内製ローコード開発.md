# 効率化のための内製・ローコード開発

## 概要

既存業務の効率化を目的に、社内の手で業務アプリ・ツールを作る業務。市販のSaaS（インターネット経由で使うソフト）や既存システムでは隙間が埋まらないとき、ローコード（最小限のコードで開発できる手法）や少量のコードで業務に合った仕組みを自前で組み上げる。狙いはあくまで「いまの業務を楽に・速く・安くする」ことであり、新しい価値や事業を生むための内製はここではない。

> 枝＝`改善 > 効率化のための内製・ローコード開発` ／ フェーズ軸＝**発見 → 分析 → 企画 → 構築 → 定着 → 改善**

---

## あるべき姿

- **目的が効率化に閉じている** … 作る動機が「いまの業務を効率化する」ことに収まっている。新価値創出を狙うなら別の枝で扱う
- **作らない判断ができる** … 何でも内製せず、内製・外注・SaaSを比べたうえで「作る」と決めている。作るほど保守が積み上がることを織り込んでいる
- **ライフサイクルで設計されている** … 作って終わりではなく、運用・改修・廃棄までを見越して作られている
- **属人化していない** … 作った本人が抜けても回るよう、ドキュメントと引き継ぎが前提になっている
- **市民開発にガードレールがある** … 市民開発（IT部門以外の現場社員による開発）を放任せず、権限・データ接続・公開範囲のルールの中で支援している
- **乱立が管理されている** … 似たアプリが量産されず、何がどこで動いているかが把握されている

## 基本方針（汎用）

会社や使うローコード基盤が違っても、内製の土台となる考え方は共通する。

- **まず「作らない」を疑う** … 内製は最後の選択肢として扱う。SaaSで足りるか、既存システムの設定で済むか、そもそも業務をやめられないかを先に問う。作った瞬間から保守責任が発生する
- **作るなら保守を引き受ける覚悟で作る** … 内製物は資産であると同時に負債である。動かし続ける人・直す人・止める人を決めずに作らない
- **属人化を設計で防ぐ** … 「作った本人しか分からない」を前提に潰す。命名・構成・前提を残し、第三者が引き継げる状態で完成とする
- **市民開発は禁止でも放任でもなく支援** … 現場が作ること自体は効率化の力になる。禁止すれば野良化（管理外で勝手に作られること）し、放任すればリスクが漏れる。ガードレールを敷いて中で走らせる
- **小さく作り、効果で続ける** … 最初から作り込まず、小さく出して使われるか・効くかを見る。使われないものは早く畳む
- **全社の標準・基盤に乗る** … 使うローコード基盤・データ接続・認証は、各自が好き勝手に選ばず全社の方針に合わせる。乱立と二重投資を避ける

## 主な業務

### 内製で作るかの判断（内製／外注／SaaSの選択）

「そもそも自分たちで作るべきか」を決める、最も重要な工程。

- 解こうとしている課題に対し、**SaaS導入・外注・内製・業務側の見直し**を並べて比較する。内製は「合うものが無い／頻繁に変わる／外に出せない」ときに効く
- 内製の総コストは初期構築だけでなく、**保守・改修・引き継ぎ・廃棄まで**で見積もる。作る判断は安く見え、持ち続ける判断は高くつく
- 一度きり・低頻度の作業なら、アプリ化せず手順書や手作業で足りることも多い。**作ることが目的化していないか**を疑う
- SaaSで足りるなら `改善 > ツール／SaaS導入` へ、定型作業の自動化が主眼なら `改善 > 手作業の自動化` へ寄せる

### ローコードでの構築

決めた要件を、ローコード基盤や少量のコードで形にする。

- 業務の入力・蓄積・通知・集計といった**定型の型**は、ローコードの得意領域。フルスクラッチ（ゼロから全部書くこと）より速く安く作れる
- 使うデータ・接続先・認証は、思いつきで足さず**全社の標準**に合わせる。野良の連携はセキュリティとデータ品質の穴になる
- 「動いた」で終えず、**入力チェック・エラー時の挙動・権限**を最低限そろえる。雑に作った内製物ほど後で運用を蝕む
- 拡張が膨らみローコードで無理が出たら、無理に粘らず正規の開発・外注へ切り替える判断を持つ

### 市民開発（現場開発）の支援とガバナンス

現場社員が自分の業務のために作ることを、ルールの中で後押しする。

- **ガードレールを先に敷く** … 誰がどの権限で作ってよいか、どのデータに接続してよいか、どこまで公開してよいかを事前に決める。自由の範囲を定義してから開放する
- **作ってよい領域と委ねてはいけない領域を分ける** … 自部署内の効率化は現場に任せ、機密データ・基幹データ・全社公開を伴うものはIT側の関与を必須にする
- **相談・レビューの窓口を用意する** … 現場が詰まったとき・危ういものを作りかけたときに相談できる先を置く。禁止より「正しく作る支援」が野良化を防ぐ
- 統制の実体は `統制 > ガバナンス・統制`、技術標準は `統制 > 全社アーキテクチャ` に従う。市民開発はその枠の中で走らせる

### 保守・引き継ぎ

作った内製物を、止めず・腐らせず・畳めるようにする。

- **最低限のドキュメント**（何のための物か・誰が使うか・どこのデータを触るか・直し方）を作成時に残す。後から書くドキュメントは書かれない
- **持ち主（オーナー）と引き継ぎ先を明記する** … 作った本人が異動・退職しても、次に誰が見るかが決まっている
- **棚卸しと廃棄を定期的に回す** … 使われなくなった内製物を放置しない。塩漬けは管理外リスクとコストの温床になる
- 安定して動き続ける運用フェーズに入ったものは `運用 > アプリケーション・SaaS運用保守` の運用対象として扱う

## フェーズ別の進め方

改善のフェーズ軸（発見 → 分析 → 企画 → 構築 → 定着 → 改善）に沿って、各段でやること・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見 | 効率化したい業務・非効率の困りごとを拾う | 課題リスト・対象業務の整理 | 解決策（作ること）から入り、課題を確かめていない |
| 分析 | 現状の手順・頻度・関係者を把握し、要件を絞る | 業務フロー・要件メモ | 例外まで作り込もうとして要件が膨らむ |
| 企画 | 内製／外注／SaaSを比較し、作るか否かと範囲を決める | 比較表・作る／作らない判断・保守体制案 | 内製ありきで他の選択肢を比較していない／保守の見積りを入れていない |
| 構築 | ローコード等で作り、権限・データ接続を標準に合わせる | 動く内製物・最低限のドキュメント | 全社標準を外れた野良の連携／作った本人しか分からない実装 |
| 定着 | 現場に使ってもらい、オーナーと引き継ぎ先を確定する | 利用手順・オーナー指定・引き継ぎ資料 | 作って渡して放置／使われないまま残る |
| 改善 | 効果と利用状況を見て、改修・統合・廃棄を判断する | 効果の記録・棚卸し結果 | 似たアプリが乱立する／使われない物を畳めず塩漬けになる |

## やらないこと

内製は「頼めば何でも作る便利屋」になりやすい。**あえて作らない・抱え込まないこと**を明確にして、内製を効率化の手段に保つ。

- **新価値創出のための内製を効率化の枠で抱えること** … 新規事業・新サービス・新たな顧客接点を生むための内製は、目的が効率化ではなく新価値創出である。これは `変革 > 新規事業・新サービスのIT支援` で扱う。同じ「内製開発」でも、**目的が効率化ならこのページ、新価値創出なら変革**へ振り分ける。手段が同じでも所属は目的で決まる
- **何でも内製すること** … 「作れるから作る」をしない。SaaS・外注・業務側の見直しで足りるものまで内製しない。作るほど保守の山が積み上がる
- **保守・オーナー不在のまま作ること** … 持ち主・直す人・止める人を決めずに作らない。作った本人が抜けて塩漬け・属人化するものを増やさない
- **市民開発を放任すること** … 「現場が勝手にやっているから」で野良アプリを見て見ぬふりしない。ガードレールの外で機密・基幹データに触れる開発を黙認しない
- **定型の自動化までここで抱えること** … RPA（ソフトのロボットによる作業自動化）やスクリプトで済む定型作業の自動化は `改善 > 手作業の自動化` が主担当。アプリとして作り込む前に、より軽い手段で足りないかを問う

> ここで線を引くのは内製を否定するためではない。**作るべきものに絞り、作ったものを持ち続けられる状態に保つ**ためのもの。断るのではなく、正しい手段・正しい枝へ案内する。

## よくある課題

- **野良アプリの乱立** … 各自が思い思いに作り、似た物が量産され、何が動いているか誰も把握していない
- **属人化と塩漬け** … 作った本人が辞めると直せず・止められず、管理外のまま残り続ける
- **保守負債の蓄積** … 作った物が増えるほど直す対象が増え、新しく作る余力が保守に食われる
- **作ること自体が目的化** … 効果より「作れた」が評価され、使われない内製物が積み上がる
- **標準を外れた連携** … 各内製物が独自にデータへ接続し、セキュリティとデータ品質の穴になる
- **効率化と新価値創出の混同** … 「内製＝先進的」と短絡し、効率化目的の内製まで変革の文脈で語られ、評価軸がぶれる

## ベストプラクティス

- **内製／外注／SaaSの判断基準を持つ** … 「合うものが無い・変化が速い・外に出せない」を内製の条件として明文化し、作る前のゲートにする
- **総保有コストで見積もる** … 初期構築だけでなく保守・改修・引き継ぎ・廃棄まで含めて費用を見る。安く見える内製の実コストを見える化する
- **作成時ドキュメントを完成条件にする** … 最低限の説明・データ接続先・直し方を残すまでを「完成」と定義する。後で書かせない
- **市民開発のガードレールを基盤側で効かせる** … 権限・接続できるデータ・公開範囲をローコード基盤の設定で制限し、ルールを人の善意でなく仕組みで守る
- **内製物の台帳と定期棚卸し** … 何が・誰のために・どこのデータを触って動いているかを一覧化し、定期的に使われていない物を畳む
- **小さく出して効果で続ける** … 最初から作り込まず最小で出し、使われ方と効果を見て、続ける・統合する・捨てるを判断する
- **テンプレート・部品の共通化** … よく使う入力・通知・集計の型を共通部品にし、各自がゼロから作る乱立を抑える

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 野良・場当たり | 無管理 | 各自が思いつきで作り、台帳も保守責任も無い。作った本人頼みで塩漬けが量産される |
| Lv2 把握 | 棚卸しの開始 | 何がどこで動いているかが台帳で把握され、オーナーが決まり始める |
| Lv3 ガードレール | 仕組みで支援 | 内製／外注／SaaSの判断基準、市民開発の権限・データ・公開のルールが整い、支援窓口がある |
| Lv4 ライフサイクル管理 | 持続的に回る | 作成から廃棄までが回り、共通部品で乱立が抑えられ、効果を見て統合・廃棄が判断される |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（何が動いているかの把握とオーナー指定）を固める**こと。把握が無ければ、ガードレールもライフサイクル管理も成り立たない。
