# 手作業の自動化（RPA・スクリプト）

## 概要

人が繰り返している定型のPC作業・事務作業を、RPA（人の定型PC操作を自動化するソフト）やスクリプト（処理を自動実行する小さなプログラム）に置き換え、楽に・速く・安く回せるようにする取り組み。目的は新しい価値を生むことではなく、既存業務の効率化である。

> 枝＝`改善 > 手作業の自動化` ／ フェーズ軸＝**発見 → 分析 → 企画 → 構築 → 定着 → 改善**

---

## あるべき姿

- **そもそも減らしてから自動化** … 自動化の前に「やめる・減らす・標準化する」が検討され、必要なものだけを自動化している
- **費用対効果で選ばれている** … 思いつきや声の大きさではなく、削減効果（頻度 × 単位時間）と作る・保守する工数を比べて対象が選ばれている
- **壊れても直せる** … 作った本人が居なくなっても、別の人が中身を読み、直し、止められる状態になっている
- **野良化していない** … 誰が・何のために・どこで動かしているかが台帳で把握され、把握外の自動化（野良ロボ・野良スクリプト）が放置されていない
- **権限が管理されている** … 自動化が使うアカウント・権限が、人のアカウントと同じ規律で発行・棚卸しされている
- **効果が見える** … 削減できた時間・件数が記録され、投資判断と次の対象選びに使われている

## 基本方針（汎用）

会社や使うツールが違っても、自動化の土台となる考え方は共通する。

- **自動化は最後の手段** … まず「その作業をやめられないか」「回数を減らせないか」「やり方を標準化できないか」を問う。無駄な作業を高速に回す自動化は、無駄を固定化するだけである
- **費用対効果で対象を選ぶ** … 作る工数・保守する工数を、削減効果（頻度 × 1回あたりの時間 × 対象人数）と比べる。年に数回・数分の作業を何日もかけて自動化しない
- **小さく作って速く確かめる** … 最初から完璧な全自動を目指さず、効果の大きい一部分から作り、実際に削減できるかを早く確かめる
- **作る前に「誰が保守するか」を決める** … 作った人だけが分かる状態を作らない。引き継ぎ・修正の担い手と手順を、着手前に決める
- **人の判断を残す境界を引く** … 金額の確定・承認・例外処理など、間違うと影響が大きい工程は人が判断し、自動化は単純作業に限る
- **自動化のIDも人と同じ規律で扱う** … 自動化用アカウントの権限は最小限にし、発行・棚卸し・パスワード管理を人のアカウントと同じ統制下に置く

## 主な業務

このノードはツリー上で小分類を持たない。実務の流れに沿って、次の三つに分けて捉える。

### 定型作業の洗い出しと対象選定

「どの作業を自動化するか」を決める、最も重要で最も飛ばされやすい工程。

- 現場の作業を棚卸しし、繰り返し・手順が固定・判断が少ない作業を候補に挙げる
- 候補ごとに **頻度 × 1回の所要時間 × 対象人数** で削減効果を見積もり、作る・保守する工数と並べて優先順位をつける
- 着手前に必ず「やめる・減らす・標準化する」を検討する。月次の手集計なら、元データの出し方を変えれば集計自体が消えることもある
- 自動化に向くのは、入力元・手順・出力先が安定している作業。頻繁に画面や様式が変わる作業は壊れやすく、向かない

### スクリプト・RPAでの自動化

選んだ作業を、適した道具で実装する。

- **スクリプト** … データの変換・集計・連携など、画面を介さず処理できるものに向く。動作が速く壊れにくいが、作るのに一定の知識が要る
- **RPA** … 画面操作を伴う作業（旧来システムへの入力、複数アプリ間の転記など）に向く。画面のレイアウト変更に弱い
- どちらも、入力・処理・出力を分け、想定外データやエラー時の止め方（異常時に黙って進めず止まる）を最初から組み込む
- 認証情報やパスワードをコード・設定にそのまま書かない。機密の扱いは `統制 > ガバナンス・統制` の方針および `運用 > セキュリティ運用` の管理に従う

### 野良化させない運用と引き継ぎ

作って終わりにせず、動き続け・直し続けられる状態を保つ。

- 作った自動化を **台帳に登録**する（名前・目的・作成者・保守担当・動作場所・使うアカウント・関連業務・最終確認日）
- ソースや手順書を個人PC・個人アカウントに置かず、組織で共有・管理できる場所に置く
- 業務やシステムの変更時に、影響する自動化を洗い出して直す経路を作る。変更管理は `運用 > 運用品質・管理` と連携する
- 定期的に棚卸しし、使われなくなった自動化・壊れたまま放置された自動化を止める

## フェーズ別の進め方

改善のフェーズ軸（発見 → 分析 → 企画 → 構築 → 定着 → 改善）に沿って、各段でやること・成果物・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見 | 現場の定型作業を棚卸しし、自動化候補を集める | 候補リスト（作業・頻度・所要時間） | 「自動化したい」が先に立ち、減らせる作業まで自動化対象にしてしまう |
| 分析 | 候補の削減効果と作る・保守する工数を見積もり、まず「やめる・減らす・標準化」を検討する | 費用対効果の試算・対象の絞り込み | 削減効果を過大に、保守工数を過小に見積もる／作りやすさだけで選ぶ |
| 企画 | 対象・道具（スクリプト／RPA）・人が判断する境界・保守担当を決める | 自動化の方針・体制 | 保守担当を決めずに着手し、後で野良化する |
| 構築 | 効果の大きい部分から小さく作り、異常時の止め方を組み込む | 動く自動化・手順書 | 全自動・全網羅を狙って肥大化／エラー処理を後回しにする |
| 定着 | 台帳登録・共有場所への配置・現場への引き渡しを行う | 台帳登録・引き継ぎ済みの状態 | 作った本人しか動かせない／個人環境に置いたまま渡らない |
| 改善 | 削減効果を測り、棚卸しで不要・故障を止め、次の対象へ回す | 効果レポート・棚卸し結果 | 効果を測らず作りっぱなし／壊れても気づかず黙って間違い続ける |

## やらないこと

自動化は「とにかく自動化すればよい」になりやすい。**あえてやらない・抱え込まないこと**を明確にして、効率化という目的を守る。

- **新価値創出が目的の自動化を、ここで扱うこと** … 同じRPA・スクリプトでも、狙いが「売上拡大・新規事業・顧客への新しい価値提供」なら、それは効率化ではなく新価値創出であり `変革 > データ活用による新価値創出` などの変革側で扱う。**手段（自動化）ではなく目的で所属が決まる**。このページが扱うのは、あくまで既存業務を楽に・安くする自動化に限る
- **減らせる・やめられる作業まで自動化すること** … 自動化は最後の手段。まず作業自体をなくせないかを問う。業務プロセスそのものの見直しは `改善 > 業務プロセス改善・標準化` に渡す
- **費用対効果に合わない自動化** … 頻度も削減時間も小さい作業を、工数をかけて自動化しない。作る工数・保守工数が削減効果を上回るなら、手作業のまま残す判断をする
- **保守担当を決めずに作ること** … 「作れる人がいるから作る」をしない。直せる人と引き継ぎ手順が無い自動化は、壊れた瞬間に業務を止める負債になる
- **本格的な業務システムの肩代わり** … 自動化はあくまで隙間を埋める道具。基幹業務を恒久的に支える規模なら、正規のツール導入（`改善 > ツール／SaaS導入`）や内製開発（`改善 > 効率化のための内製・ローコード開発`）の検討に上げる
- **人の判断・承認の代行** … 金額確定・承認・例外可否などの判断を自動化に肩代わりさせない。間違いがそのまま通る経路を作らない
- **自動化用アカウントの自前管理** … 自動化が使うID・権限を、ここで独自に発行・放置しない。発行・棚卸しは `運用 > セキュリティ運用` の管理に乗せる

> ここで線を引くのは仕事を減らすためではない。**自動化を「効率化に効く範囲」に集中させ、目的が違うもの・規模が違うもの・判断を伴うものを正しい行き先へ渡す**ためである。

## よくある課題

- **野良ロボ・野良スクリプト** … 作った人が異動・退職し、誰も中身を知らず直せない。業務変更で壊れても放置される
- **黙って間違い続ける** … エラーで止まらず、間違った結果を出し続け、後工程まで進んでから気づく
- **保守の負債化** … 作った数だけ保守対象が増え、いつしか自動化の維持が新たな手作業になる
- **費用対効果の度外視** … 効果の小さい作業に工数をかけ、作ること自体が目的化する
- **権限の置き去り** … 退職者のアカウントや過大な権限で自動化が動き続け、棚卸しから漏れる
- **脆い作りで壊れやすい** … 画面レイアウトや様式の小さな変更で止まり、その都度手で直す羽目になる

## ベストプラクティス

- **自動化候補の選定基準を持つ** … 「頻度 × 単位時間 × 人数」で削減効果を数値化し、保守工数と並べて優先順位をつける。基準を共有し、思いつき着手を防ぐ
- **自動化台帳の整備** … 何が・どこで・誰の責任で・どのアカウントで動いているかを一覧化する。野良化防止の土台であり、棚卸し・引き継ぎの起点になる
- **ソースと手順の集中管理** … コード・設定・手順書を組織共有の場所に置き、個人環境・個人アカウントに依存させない
- **異常時に止める設計** … 想定外データやエラーで黙って進めず止め、気づける通知を出す。間違いを下流に流さない
- **定期棚卸し** … 半年〜1年ごとに、使われていない・壊れている・効果が薄れた自動化を見直し、止める判断を含める
- **効果測定と還元** … 削減できた時間・件数を記録し、投資判断と次の対象選びに使う。効果が出ない自動化は畳む
- **作りすぎない** … 「人がたまにやる方が安い」境界を持つ。全自動化が常に正解ではない

## 成熟度モデル

自社の現在地を測る目安。下から積み上げる。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 個人技 | 属人・点在 | 一部の人が自分用に作り、把握も共有もされていない。作った人頼み |
| Lv2 共有化 | 把握できる | 作った自動化が台帳と共有場所で把握され、別の人も中身を追える |
| Lv3 標準化 | 仕組みで回す | 選定基準・異常時の止め方・保守担当・棚卸しが定まり、誰が作っても一定の品質になる |
| Lv4 改善循環 | 効果で前へ | 効果測定から対象選定・畳む判断が回り、保守が負債化せず、自動化の総量と質が管理されている |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（台帳と共有場所による把握）を固める**こと。把握が無ければ、野良化を防ぐことも棚卸しすることもできない。

- RPA・自動化ツールの一般的な運用知見（対象選定・野良ロボ対策・保守体制）に関するベストプラクティス
- 業務効率化の基本（ECRS：Eliminate-Combine-Rearrange-Simplify、なくす・まとめる・順序を変える・簡単にする）… 自動化の前に作業自体を見直す考え方の出典
- 関連ノード：`改善 > 業務プロセス改善・標準化`（自動化の前にそもそも作業を減らす）、`改善 > ツール／SaaS導入`（既製ツールで代替できる場合の行き先）、`改善 > 効率化のための内製・ローコード開発`（規模が大きい場合の行き先）、`変革 > データ活用による新価値創出`（目的が新価値創出の場合の行き先）、`運用 > セキュリティ運用`（自動化用アカウント・権限の管理）、`運用 > 運用品質・管理`（業務・システム変更に伴う自動化の改修）、`統制 > ガバナンス・統制`（機密情報の扱い・自動化の規律）
