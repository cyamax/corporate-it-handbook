# データ活用による新価値創出

## 概要

社内に蓄積されたデータを分析・予測に活かし、売上拡大や新たな意思決定など、これまで生み出せなかった価値を創る取り組み。表計算での集計に留まらず、BI（データを可視化し分析する仕組み）やML（機械学習：データから法則を学習し予測する技術）を使い、「勘と経験」だけでは見えなかった示唆を事業へ届けることを目的とする。

> 枝＝`変革 > データ活用による新価値創出` ／ フェーズ軸＝**発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善**

---

## あるべき姿

- **価値仮説が先にある** … 「誰の・どの意思決定を・どう変えるか」が言語化されてから、データに着手している。分析のための分析になっていない
- **意思決定に組み込まれている** … 作った指標・予測が、実際の会議や業務フローの中で参照され、行動を変えている。ダッシュボードが「見られて終わり」になっていない
- **データが信頼できる** … 元データの品質・定義が揃っており、出てくる数字が事業側に信用されている
- **価値が数字で語れる** … その取り組みが売上・コスト・意思決定の速さにどう効いたかを、後から検証できる
- **本番で回り続ける** … PoC（小さく試す検証）で終わらせず、本番運用に乗り、モデルやデータが継続的に更新されている
- **使ってよい範囲を守っている** … 個人情報・取得時の利用目的・契約上の制約を踏まえた、扱ってよいデータだけを使っている

## 基本方針（汎用）

会社や扱うデータが違っても、データ活用で価値を生む土台は共通する。

- **価値仮説から始める** … 「どんなデータがあるか」ではなく「どの意思決定を変えたいか」から入る。データの量や精度の議論より、誰のどの判断を動かすかが先に来る
- **意思決定・業務への組み込みをゴールに置く** … 分析結果が出た時点はまだ道半ば。会議体・承認フロー・現場のオペレーションに組み込まれて初めて価値になる。「作る」より「使われる」に投資する
- **データ品質を前提として確認する** … 汚いデータからは正しい示唆は出ない。着手前に、元データの定義・欠損・鮮度を確認する。ただし品質ルールや基盤方針そのものの策定は本ノードの仕事ではない（`統制 > データガバナンス` に従う）
- **小さく試してから広げる** … 大規模な基盤投資の前に、PoCで価値仮説が成り立つかを検証する。筋が悪ければ早く畳む
- **利用目的とプライバシーを守る** … データには取得時の利用目的・同意範囲・契約上の制約がある。「手元にあるから使える」ではない。個人を特定する分析は特に慎重に扱う
- **精度より意思決定への効きを問う** … モデルの精度を1ポイント上げることより、その出力が実際に判断を変え、収益に効くかを優先する

## 主な業務

### 価値仮説とユースケースの発見

事業部門の課題から出発し、「データで変えられる意思決定」を見つけて仮説に落とす業務。

- 「売上を伸ばしたい」「離反を防ぎたい」といった事業課題を、**どの意思決定を・どのデータで・どう変えるか**まで具体化する
- 仮説には「成功したら何が・どれだけ変わるか」の見込み（期待効果）を必ず添える。効果が描けないテーマは優先度を下げる
- 「とりあえずデータを集めて見てみる」を出発点にしない。集めること自体を目的化すると、使われない成果物が量産される

### 分析・可視化・予測モデルの構築

仮説を検証し、意思決定を支える分析・指標・予測を形にする業務。

- 元データを集め、定義を揃え、検証できる状態に整える（前処理）。ここで品質が崩れていると、以降の示唆はすべて疑わしくなる
- 検証したい問いに応じて、可視化（BIによる現状把握）・予測（MLによる将来予測）・要因分析を使い分ける
- まずPoCで「価値仮説が成り立つか」を小さく確かめる。精度の作り込みより、判断を変えられる確度があるかを先に見る

### 意思決定・業務への組み込みと定着

作った成果物を、実際の判断と業務の中で使われる状態にする業務。

- 出力を「どの会議で・誰が・いつ見て・何を決めるか」まで運用に落とし込む。置き場所を作るだけでは使われない
- 予測モデルは作って終わりではなく、**継続的な再学習・監視・更新**を運用に組み込む。データの傾向が変われば精度は劣化する
- 利用状況をモニタリングし、見られていない・意思決定に効いていない成果物は作り直すか畳む

## フェーズ別の進め方

変革のフェーズ軸（発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善）に沿って、各段でやること・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見・相談 | 事業課題を聞き、データで変えられる意思決定を探り、価値仮説に落とす | 価値仮説・ユースケース案 | データありきで始め、仮説が「誰のどの判断を変えるか」まで具体化されない |
| 現状分析・要件 | 使えるデータの所在・品質・利用制約を確認し、検証要件を定める | データ棚卸し・要件・前提制約 | 元データの品質・定義の確認を飛ばす／個人情報・利用目的の制約を見落とす |
| 企画・提案 | 期待効果・進め方・PoC範囲を整理し、投資判断を仰ぐ | 企画書・PoC計画・効果見込み | 効果の見込みを描けないまま「やってみたい」で進める |
| 構築・導入 | PoCで仮説を検証し、筋が良ければ分析・可視化・モデルを本番化する | 検証結果・ダッシュボード／モデル | PoCの作り込みに没頭し本番運用設計を後回しにする |
| 定着化 | 意思決定・業務フローへ組み込み、再学習・更新の運用を回す | 運用手順・組み込み済みプロセス | 成果物を置くだけで使われない／モデルの更新が止まり精度が劣化する |
| 効果測定・改善 | 意思決定・収益への効きを検証し、横展開か撤退かを判断する | 効果検証レポート | 「作ったこと」で満足し、価値が出たかを検証しない |

## やらないこと

データ活用は「データを触ること」自体が目的化しやすく、また統制が担うルール作りや効率化目的の案件と混同されやすい。**あえてやらない・抱え込まないこと**を明確にして、本来の役割（活用して価値を生む）に集中する。

- **データの統制・品質ルール・基盤方針の策定** … データの管理方針・品質基準・連携基盤の方針づくりは規律を与える統制の仕事であり、`統制 > データガバナンス` が担う。本ノードはそのルールに**従って活用する側**。前提が崩れていれば指摘して引き渡すが、ルールそのものを本ノードで定義しない
- **効率化が目的のデータ活用** … 既存業務の集計を楽にする・工数を削るためのダッシュボード化や自動集計は、新価値創出ではなく効率化が目的のため `改善 > 業務プロセス改善・標準化` や `改善 > 手作業の自動化` の領域。同じBIでも、目的が「業務を楽にする」なら改善、「売上・新たな意思決定を生む」なら本ノードである
- **価値仮説なきデータ収集と分析** … 「使えるかもしれないから集める」「とりあえず可視化する」を引き受けない。誰のどの意思決定を変えるかが描けないテーマは着手しない
- **使われない成果物の量産** … 依頼されるまま個別のダッシュボードを作り続けない。意思決定・業務に組み込まれる見込みがないものは作らない
- **データ基盤・パイプラインの恒常運用を抱えること** … 構築した分析基盤・連携基盤の定常的な運用・監視は `運用 > インフラ運用` へ引き渡す。プロジェクトの成果物を運用ごと抱え込むと、次の新価値創出に手が回らなくなる
- **利用目的・同意範囲を超えたデータ利用** … 「技術的に分析できる」と「利用してよい」は別。個人情報・取得時の利用目的・契約上の制約を超える分析は、便利でも行わない

> ここで線を引くのは縄張り争いのためではない。**活用して価値を生むことに集中し、統制・効率化・運用の各役割へ正しく引き渡す**ためのもの。

## よくある課題

- **データを集めれば価値が出るという幻想** … 基盤やデータレイクを作ったが、何の意思決定にも使われず塩漬けになる
- **分析のための分析** … 精緻なレポートやモデルは出来たが、誰の判断も変えていない
- **作ったダッシュボード／モデルが使われない** … 公開はしたが現場のフローに組み込まれず、見られなくなる
- **汚いデータからの誤った示唆** … 定義の揺れ・欠損・重複を放置したまま分析し、間違った結論を導く
- **PoC止まり** … 検証では手応えがあったが本番運用に乗らず、モデル更新も止まる
- **統制・効率化との混線** … 統制（データガバナンス）が担うべきルール作りや、改善が担うべき効率化案件を抱え込み、本来の新価値創出が進まない

## ベストプラクティス

- **価値仮説シートで起点を揃える** … 「対象の意思決定／使うデータ／期待効果／検証方法」を1枚にまとめてから着手する。埋まらない項目があるテーマは、まだ着手の段階にない
- **意思決定の場とセットで設計する** … 「どの会議で・誰が・何を決めるか」を成果物の置き場所として先に決める。組み込み先のない分析は作らない
- **PoCに撤退ラインを置く** … 検証の前に「何が確認できたら本番化／畳むか」を決め、ずるずる続けない
- **モデル運用（MLOps）を前提に組む** … 予測モデルは再学習・性能監視・更新の仕組み（MLOps：MLモデルを継続的に運用する仕組み）まで含めて設計する。作って放置すれば精度は必ず落ちる
- **品質は上流で確認し、ルールは統制と連携する** … 前処理で品質を確認しつつ、恒常的な品質基準・基盤方針は `統制 > データガバナンス` と連携して整える
- **効果を意思決定・収益で語る** … 精度や閲覧数ではなく、「どの判断が・どう変わり・収益にどう効いたか」で成果を測る

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 散発 | 個別集計 | 担当者が都度Excelで集計。データは部署ごとに散在し、定義も揃わない |
| Lv2 可視化 | 見える化 | BIで主要指標が可視化されるが、見られて終わりで意思決定は従来どおり |
| Lv3 組み込み | 判断に効く | 価値仮説起点で分析・予測を作り、会議・業務フローに組み込まれて判断を変えている |
| Lv4 継続創出 | 仕組みで回る | 予測モデルが本番運用・継続更新され、価値仮説の発見から効果検証までが繰り返し回り、新たな収益・意思決定を生み続ける |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv3（意思決定への組み込み）を越える**ことを意識する。Lv2（可視化）止まりは、見栄えはよくても価値を生んでいない状態である。

## 参考資料

- BI／データ分析の一般的な実践知（価値仮説起点・意思決定への組み込み・データ品質の前提）
- MLOps（機械学習モデルを継続的に運用・更新する考え方）… PoC止まりを避け本番運用へつなぐための知見
- 個人情報保護・データ利用目的に関する法令・社内規程（利用してよい範囲の前提）
- 関連ノード：`統制 > データガバナンス`（データ品質・管理方針・基盤方針＝守りのルールの担い手）、`改善 > 業務プロセス改善・標準化`（効率化目的のデータ活用の引き渡し先）、`変革 > 先進技術による競争力強化`（AI／生成AIで新価値を生む隣接領域）、`運用 > インフラ運用`（構築した分析・連携基盤の定常運用の引き渡し先）
