# 新規事業・新サービスのIT支援

## 概要

会社が新しく立ち上げる事業・サービスに、社内ITが技術面から伴走し、立ち上げのスピードと確度を高める支援業務。目的は効率化ではなく**新たな価値の創出**（売上拡大・新規事業・競争優位）であり、不確実性の高い状況で小さく速く形にすることを支える。

> 枝＝変革 > 新規事業・新サービスのIT支援 ／ フェーズ軸＝発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善

---

## あるべき姿

- **早い段階から関わる** … 事業の構想段階からITが議論の場にいて、技術的な実現性・概算・リスクを早期に示せている
- **小さく速く出す** … 最初から作り込まず、PoC（試しに小さく試す検証）やMVP（実用最小限の製品）で素早く市場や顧客に当て、学習に回せている
- **捨てやすい作り** … 撤退・ピボット（方向転換）を前提に、深追いせず、捨てる・作り直すコストを低く保てている
- **主役は事業側** … 事業の意思決定は事業部門・経営が握り、ITは実現を担う伴走者として役割が明確に分かれている
- **ガードレールは外さない** … スピードを優先しつつも、最低限のセキュリティ・データ保護・全社標準は守られている
- **立ち上がったら引き渡す** … 軌道に乗ったサービスは作りっぱなしにせず、本番化して運用へ確実に引き継げている

## 基本方針（汎用）

会社や扱う事業が違っても、新規事業のIT支援で外せない土台は共通する。

- **事業の主役は事業部門・経営、ITは伴走支援** … ITが単独で事業を作るのではない。価値仮説・収益モデル・市場判断は事業側が担い、ITはその実現と技術的助言を担う。主役と伴走の線を引く
- **不確実性を前提にする** … 要件は固まらないものとして進める。最初に仕様を固め切る進め方（ウォーターフォール）ではなく、小さく作り素早く改善する進め方（アジャイル）を基本に置く
- **小さく速く、学習を回す** … PoCで実現性を確かめ、MVPで顧客価値を確かめる。作る量ではなく、検証して得た学びの量を成果とみなす
- **撤退・ピボットを前提に作る** … 多くの新規事業は当たらない・方向転換する。捨てやすさを設計に織り込み、初期に深い作り込みを急がない
- **最低限のガードレールは外さない** … 速さのために顧客情報の保護・認証・最低限の全社標準まで捨てない。後で致命傷になる領域だけは初期から守る
- **立ち上げ後は運用へ引き継ぐ** … スケールが見えたら本番品質に引き上げ、運用・保守の担い手へ渡す。実験のまま本番に居座らせない

## 主な業務

### 事業構想段階からのIT参画

事業のアイデアが固まる前の段階から議論に加わり、**技術で何ができ・何が難しく・どれくらいかかるか**を早期に示す。

- 技術的な実現性、概算コスト、必要な期間、想定リスクを早い段階で可視化し、事業判断の材料にする
- 「作れるか」だけでなく「作るべきか・今は作らない方がよいか」も率直に助言する。技術が目的化しないよう牽制する
- 既存の社内データ・基盤・SaaSで賄える部分と、新規に作る部分を切り分け、無駄な開発を避ける

### 不確実性を前提にした小さく速い構築（PoC・MVP）

要件が固まらない前提で、**検証可能な最小限の形**を素早く作り、顧客や市場に当てて学習する。

- PoCで技術的に成立するかを確かめ、MVPで顧客が価値を感じるかを確かめる。両者の目的を混同しない
- ノーコード・ローコード・SaaSの組み合わせなど、作らずに済ます選択肢を優先的に検討する
- 計測の仕組み（利用状況・反応の取得）を最初から組み込み、続けるか・やめるか・変えるかを数字で判断できるようにする
- 捨てる前提のため、作り込み・最適化・自動化は後回しにする。きれいさより検証速度を優先する

### スケール時の本番化と運用引き継ぎ

検証を抜けて事業として続けると決まったサービスを、**実験品質から本番品質へ引き上げ、運用の担い手へ渡す**。

- 利用が増える前提で、可用性・性能・セキュリティ・監視・バックアップを本番水準に作り直す
- 急いで作った箇所（技術的負債）を棚卸しし、本番化で返すものと許容するものを判断する
- 運用・保守の引き継ぎ先（`運用 > アプリケーション・SaaS運用保守` など）に、構成・手順・既知の課題を文書とともに渡す
- ガバナンス・契約・標準化が必要になった部分は統制（`統制 > IT戦略・企画` ほか）の管理下へ載せ替える

## フェーズ別の進め方

変革のフェーズ軸（発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善）に沿って、各段でやること・成果物・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見・相談 | 事業構想を聞き、技術で実現可能なことと制約を早期に示す | 技術メモ・概算・初期リスク一覧 | IT抜きで構想が進み、後から無理な要件が来る／実現性を確かめず安請け合いする |
| 現状分析・要件 | 既存資産・データで賄える範囲を見極め、固まらない要件を仮置きする | 仮要件・前提条件・検証したい仮説 | 要件を固め切ろうとして時間を浪費する／不確実性を無視して作り込みを始める |
| 企画・提案 | 検証方法（PoC／MVP）と範囲、撤退・継続の判断基準を決める | 検証計画・判断基準（やめどき）・概算 | 最初から本番規模で見積もる／撤退条件を決めずに始める |
| 構築・導入 | 最小限を素早く作り、顧客・市場に当てて反応を取得する | 動くPoC／MVP・計測データ | きれいに作り込みすぎる／計測を入れず学べない |
| 定着化 | 続けると決めたら本番品質へ引き上げ、運用へ引き継ぐ | 本番化されたサービス・運用引き継ぎ資料 | 実験のまま本番運用に突入する／引き継ぎ先を決めず作った人が抱え続ける |
| 効果測定・改善 | 事業成果（売上・利用・満足）を測り、継続・拡大・撤退を判断する | 効果レポート・次の打ち手・撤退判断 | 投じたコストに引きずって撤退できない（サンクコスト）／IT指標だけ見て事業価値を見ない |

## やらないこと

新規事業の支援は熱量が高く、ITが前のめりになりやすい。**あえてやらない・抱え込まないこと**を明確にし、伴走の立場を守る。

- **効率化目的の取り組みをここで扱うこと** … 既存業務のコスト削減・省力化が目的なら、手段が内製・データ活用・AIであってもここではない。`改善 > 効率化のための内製・ローコード開発` や `改善 > 手作業の自動化` など改善側へ振り分ける。同じ技術でも目的が新価値創出ならここ、効率化なら改善という境界を守る
- **IT単独で事業を作ること** … 価値仮説・収益モデル・市場判断を肩代わりしない。事業の意思決定は事業部門・経営に返す。ITが事業の主役を奪うと、責任の所在が曖昧になり判断も遅れる
- **最初から作り込むこと** … 当たるか分からない段階で、本番品質・自動化・最適化に時間をかけない。検証で必要になるまで作り込みを遅らせる
- **検証フェーズの成果物を本番運用で抱え続けること** … 急ぎで作ったPoC・MVPを、そのまま自分たちで運用し続けない。続けると決まったものは本番化して運用へ渡す（`運用 > アプリケーション・SaaS運用保守`）
- **データ活用そのものを目的化して抱えること** … 売上・意思決定に効く分析・予測など、データ活用が新価値の主役になる取り組みは `変革 > データ活用による新価値創出` が主管。重なる場合は役割分担を決め、何でも引き受けない
- **撤退判断を避けること** … 投じたコストを惜しんで（サンクコスト）、当たらない取り組みを延命しない。やめどきの基準を事前に決め、淡々と撤退・方向転換を提案する

> ここで線を引くのは消極姿勢ではない。**ITが伴走に徹し、事業の判断を事業側へ返し、捨てるべきものを捨てる**ことで、新規事業全体のスピードと健全さを保つためのもの。

## よくある課題

- **IT抜きで構想が進む** … 事業の絵が固まってからITに渡され、技術的に無理のある要件や非現実的な納期を押し付けられる
- **作り込みすぎて学べない** … 当たるか分からない段階できれいに作り込み、検証より工数が先に膨らむ
- **撤退できない** … 投じたコストに引きずられ、当たらない取り組みをやめられず、人と予算を浪費し続ける
- **実験のまま本番化する** … 急ぎで作ったPoC・MVPがそのまま顧客に使われ、可用性・セキュリティの不備が後から表面化する
- **作った人が抱え続ける** … 立ち上げた担当が運用まで丸抱えし、本来の新規開発に手が回らなくなる
- **ガードレールを外しすぎる** … スピードを優先するあまり顧客情報の保護や認証を後回しにし、事故・信用毀損を招く

## ベストプラクティス

- **発見段階から同席する** … 事業企画の初期会議にITが入り、実現性・概算・リスクを早期に提示する。後出しの「作れません」をなくす
- **PoCとMVPを使い分ける** … 技術が成立するかはPoC、顧客が価値を感じるかはMVPで確かめる。検証の目的を明示してから作る
- **やめどきを先に決める** … 着手前に「続ける／やめる／変える」の判断基準と期限を合意し、撤退をあらかじめ正当な選択肢にしておく
- **作らない選択を優先する** … SaaS・ノーコード・既存資産の組み合わせで賄えないかを先に検討し、独自開発は本当に必要な範囲に絞る
- **計測を最初から組み込む** … 利用状況・顧客反応を取得する仕組みを初期から入れ、継続判断を勘でなく数字で行う
- **最低限のガードレールを定義する** … 速さのために緩める領域と、絶対に外さない領域（顧客情報保護・認証・主要な全社標準）を事前に線引きする
- **本番化チェックリストで引き継ぐ** … 可用性・セキュリティ・監視・運用手順・既知の負債を一覧化し、運用への引き継ぎ漏れを防ぐ

## 成熟度モデル

自社の現在地を測る目安。下から積み上げる。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 後追い・丸投げ | 事業の下請け | 構想が固まってからITに渡される。要件を言われるまま受け、作り込んでから当たらず終わる |
| Lv2 小さく試せる | 検証の習慣 | PoC・MVPで小さく速く検証する型があり、作る前に確かめる動きができている |
| Lv3 伴走と線引き | 役割が明確 | 発見段階からITが同席し、撤退基準を持ち、本番化したら運用へ引き継ぐ流れが回っている |
| Lv4 価値で前へ | 事業と一体 | 事業成果を数字で測り、継続・拡大・撤退を素早く判断。新価値創出にITが当たり前に組み込まれている |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（小さく速く検証する型）を固める**こと。作り込んでから当たらない失敗を、検証の習慣でなくすのが起点になる。

## 参考資料

- リーンスタートアップ（仮説検証を小さく速く回す事業立ち上げの考え方）… PoC・MVP・ピボットの出典
- アジャイル開発（小さく作り素早く改善する進め方）… 不確実性下での構築の考え方
- 経済産業省「攻めのIT」の考え方 … 新価値創出を目的とする変革の位置づけ
- 関連ノード：`変革 > データ活用による新価値創出`（データが新価値の主役になる取り組みの引き渡し先）、`変革 > デジタル接点・顧客体験の創出`（顧客チャネルが主役の取り組み）、`改善 > 効率化のための内製・ローコード開発`（目的が効率化の場合の振り分け先）、`運用 > アプリケーション・SaaS運用保守`（立ち上げ後の運用引き継ぎ先）、`統制 > IT戦略・企画`（投資判断・標準化の管理下への載せ替え先）
