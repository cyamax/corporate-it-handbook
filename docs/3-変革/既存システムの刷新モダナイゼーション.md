# 既存システムの刷新・モダナイゼーション

## 概要

老朽化・ブラックボックス化したレガシー（老朽化・属人化した既存システム）を、変化に対応できる形へ刷新する取り組み。モダナイゼーション（最新の形へ刷新すること）・クラウド移行・システム統合などを通じて、保守限界に達した既存資産を作り替え、新しい価値を生むための足場をつくる。

> 枝＝変革 > 既存システムの刷新・モダナイゼーション ／ フェーズ軸＝発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善

この領域が「変革」に置かれるのは、単なる延命や効率化ではなく、**新しい価値創出を阻む足かせ（レガシー）の除去**を狙うためである。古いシステムを抱えたままでは、新規事業もデータ活用もスピードが出ない。経済産業省が「2025年の崖」と呼んだ問題——レガシーを放置することによる競争力低下・保守限界・有識者の枯渇——への対応がここに当たる。刷新そのものが目的ではなく、刷新によって**何を実現したいか**を起点に据える領域である。

---

## あるべき姿

- **目的が先にある** … 「古いから作り直す」ではなく、刷新によって実現したい事業上の状態（スピード・拡張性・データ連携）が先に定義されている
- **現行が見えている** … 現行システムの仕様・データ・連携が棚卸しされ、ブラックボックスが解消されている
- **業務とセットで見直す** … 現行の業務をそのまま作り直すのではなく、刷新を機に業務そのものを見直している
- **段階的に移る** … 一度に全てを切り替えず、段階移行・並行稼働・切り戻しの計画があり、後戻りできる
- **データが安全に移る** … データ移行のリハーサルと検証が済み、移行後も整合性が保たれている
- **経営の合意がある** … 巨額・長期になる前提で、投資判断と経営の合意が取れている（統制と連動している）
- **塩漬けを生まない** … 刷新後の保守・更新の道筋が決まっており、新たなレガシーを生まない設計になっている

## 基本方針（汎用）

会社や対象システムが違っても、刷新プロジェクトの土台となる考え方は共通する。

- **手段ではなく目的から始める** … 刷新は手段である。「何を実現するか」を見失うと、莫大なコストをかけて「作り直しただけ」で終わる。目的を最初に言語化し、関係者で握る
- **現行踏襲を疑う** … 「今の仕様をそのまま」は最も危険な要件である。現行をそのまま作り直すと、業務の無駄や負債まで引き継ぐ。刷新は業務の見直しとセットで行う
- **ビッグバンを避ける** … 一斉切り替え（ビッグバン移行）はリスクが集中する。段階移行・並行稼働・切り戻し計画を前提に置き、後戻りできる状態を保つ
- **データ移行を最難関と心得る** … 刷新の成否はデータ移行で決まることが多い。量・品質・整合性の壁は大きく、移行リハーサルと検証を繰り返す前提で計画する
- **現行の仕様調査に時間を見込む** … ブラックボックス化した現行は、ドキュメントも有識者も失われていることが多い。仕様の発掘・調査そのものに相応の工数と期間を見込む
- **経営の合意と投資判断を前提にする** … 巨額・長期になりやすい。着手前に投資判断と経営合意を取り、`統制 > IT戦略・企画` と連動させる。情シス単独で抱え込まない
- **作って終わりにしない** … 刷新後の運用・保守・更新の道筋まで設計に含める。これを怠ると、刷新したシステムが数年で再び塩漬けになる

## 主な業務

### 現行資産の棚卸しと塩漬けリスク評価

刷新の出発点は、**今あるものを正確に把握する**ことである。何が動いていて、何が分からなくなっているかを明らかにする。

- 対象システムの機能・データ・外部連携・利用部門を洗い出し、台帳化する
- ドキュメントの有無、有識者の在籍状況、ベンダー依存度から**ブラックボックス度**を評価する
- 老朽化・保守切れ・属人化の度合いから、**塩漬けによるリスク**（障害時に直せない、変更できない、有識者がいない）を見える化する
- 「今すぐ刷新すべきか」「延命で凌げるか」を、リスクとコストの両面で判断材料にそろえる

### 刷新方式の選定（再構築・移行・統合・クラウド化）

現行をどう作り替えるかは一通りではない。**目的とリスクに照らして方式を選ぶ**。

- 主な方式の整理（いずれもモダナイゼーションの手段）
  - **再構築（リビルド）** … 作り直す。自由度は高いが、コスト・期間・リスクも最大
  - **移行（マイグレーション：移行）** … 仕組みは活かして新しい基盤へ載せ替える。クラウド移行（持ち上げて載せ替えるリフト＆シフトを含む）など
  - **システム統合** … 重複・分散したシステムを集約し、連携の複雑さを減らす
  - **段階的なモダナイゼーション** … 一部の機能から切り出して置き換える。リスクを分散できる
- 方式ごとにコスト・期間・リスク・得られる効果が違う。**目的に最も適う方式**を選ぶ。流行や手段先行で選ばない
- パッケージ・SaaS活用か内製かも、この段で見極める。全社の技術標準とは `統制 > 全社アーキテクチャ` で整合を取る

### 段階移行とデータ移行・並行稼働

刷新の山場は、**現行から新システムへ安全に移す**工程である。ここでの失敗が事業を止める。

- 一斉切り替えを避け、**機能・部門・地域などで区切って段階的に移行**する
- 新旧を一定期間つなぐ**並行稼働**で、結果を突き合わせて検証する
- **データ移行**は最難関と心得る。移行対象の選別、品質（重複・欠損・表記揺れ）の是正、新旧の項目対応（マッピング）を詰める
- **移行リハーサル**を本番前に複数回行い、所要時間・手順・検証方法を固める
- うまくいかなかったときの**切り戻し計画**を必ず用意する。後戻りできない移行は計画として未完成である

## フェーズ別の進め方

変革のフェーズ軸（発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善）に沿って、各段でやること・成果物・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見・相談 | 刷新の動機（保守限界・事業要請）を受け、目的と対象範囲の仮説を立てる | 課題メモ・刷新の狙い | 「古いから」だけで動き、実現したい状態が定義されない |
| 現状分析・要件 | 現行資産を棚卸しし、ブラックボックスを調査。業務の見直しを含めて要件を定義する | 現行台帳・要件定義・移行方針 | 現行踏襲のまま要件化／仕様調査の工数を甘く見る |
| 企画・提案 | 刷新方式を比較し、コスト・期間・リスクを見積もる。投資判断を経営へ諮る | 方式比較・投資計画・移行計画 | 経営合意のないまま着手／効果より手段が先行する |
| 構築・導入 | 新システムを構築し、データ移行・段階移行・並行稼働・移行リハーサルを行う | 新システム・移行手順・切り戻し計画 | ビッグバン移行に走る／データ移行で整合性が崩れる |
| 定着化 | 利用部門への教育、運用・保守体制の引き渡し、旧システムの停止 | 運用手順・教育・廃止記録 | 旧システムを止めきれず二重保守が残る |
| 効果測定・改善 | 当初目的に対する効果を測り、残課題を改善へ回す | 効果測定レポート | 作り直したこと自体を成果と錯覚し、目的達成を検証しない |

## やらないこと

刷新プロジェクトは「IT全般の作業」を巻き込みやすく、肥大化しやすい。**あえてやらない・抱え込まないこと**を明確にし、引き渡し先を示す。

- **日々の保守・延命を抱えること** … 現行システムを動かし続ける保守や小修正は刷新の仕事ではない。`運用 > アプリケーション・SaaS運用保守` の領域である。刷新チームが現行のお守りまで抱えると、本来の刷新が進まない
- **基盤の日常運用を抱えること** … サーバー・クラウド・ネットワークの日常運用・監視は `運用 > インフラ運用` へ渡す。刷新は「作り替え」までで、できた基盤の定常運用は運用へ引き渡す
- **効率化だけが目的の置き換えを変革として扱うこと** … 新価値創出ではなく、単に手間やコストを下げるための置き換えは `改善` の領域である。目的が効率化なら、ここではなく改善として進める
- **全社のIT戦略・投資判断そのものを決めること** … どこに・いくら投資するかという全社的な判断は `統制 > IT戦略・企画` の役割である。刷新チームは案件として提案・実行する側であり、投資の最終判断を代行しない
- **目的の曖昧なまま着手すること** … 「とりあえず新しくする」で走り出さない。実現したい状態が定義できないなら、構築には進めず発見・相談へ戻す
- **新たな塩漬けを生む作り方をすること** … 刷新後の保守・更新を考えない、再びブラックボックス化する作り方はしない。同じ問題を将来に先送りするだけになる

> 線を引くのは仕事を狭めるためではない。**刷新を「目的の実現」に集中させ、運用・改善・統制へ正しく渡す**ためである。抱え込みは刷新の頓挫につながる。

## よくある課題

- **現行踏襲の罠** … 「今と同じものを」を要件にしてしまい、業務の無駄や負債ごと新システムに引き継ぐ
- **目的の喪失** … 進めるうちに刷新自体が目的化し、巨費を投じて「作り直しただけ」で終わる
- **ブラックボックスと有識者の枯渇** … 現行の仕様が分かる人もドキュメントもなく、調査だけで時間と費用が膨らむ
- **データ移行の難航** … データの量・品質・整合性が想定を超え、移行が遅延・失敗する
- **ビッグバン移行の事故** … 一斉切り替えに失敗し、切り戻しもできず事業が止まる
- **巨額・長期化** … 見積もりが甘く、費用と期間が膨張する。途中で予算が尽きて中途半端に止まる
- **二重保守の継続** … 旧システムを止めきれず、新旧を並行して保守し続けてコストが下がらない

## ベストプラクティス

- **目的を一枚に言語化する** … 刷新で実現したい事業上の状態を一枚に書き、関係者で握る。判断に迷ったらここへ立ち返る
- **現行調査を独立した工程に置く** … 仕様の発掘・棚卸しを「ついで」にせず、相応の工数を確保した独立工程として計画する
- **方式は目的基準で比較する** … 再構築・移行・統合・クラウド化を、コスト・期間・リスク・効果の観点で並べて比較し、目的に最も適う方式を選ぶ
- **段階移行を既定とする** … 機能・部門単位で区切り、並行稼働で突き合わせ検証する。ビッグバンは例外と位置づける
- **データ移行はリハーサル前提で組む** … 本番前に複数回の移行リハーサルを行い、所要時間・手順・検証・切り戻しを固める
- **切り戻し計画を成果物に含める** … 後戻りできる状態を移行の必須要件とする。切り戻せない計画は未完成とみなす
- **投資判断を統制に乗せる** … 巨額・長期の前提で、着手前に `統制 > IT戦略・企画` の投資判断を通す。情シス単独で抱えない
- **刷新後の保守設計まで含める** … 運用・更新の道筋を設計段階で決め、新たなレガシーを生まない作りにする

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 放置 | 塩漬け | レガシーを認識しつつ手を付けられない。ブラックボックス化が進み、障害時に直せない |
| Lv2 場当たり刷新 | 現行踏襲 | 「古いから作り直す」で着手。目的が曖昧で、現行をそのまま再現してしまう |
| Lv3 計画的刷新 | 段階・検証 | 目的を定義し、現行を棚卸しし、段階移行・データ移行リハーサル・切り戻しを計画して進める |
| Lv4 継続的近代化 | 塩漬けを生まない | 業務の見直しとセットで刷新し、刷新後の保守・更新の道筋まで設計。レガシーを再生産しない状態へ |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv3（目的定義と段階移行・検証）を固める**こと。目的と現行把握がないまま構築に進むと、ビッグバンの事故と「作り直しただけ」の結末を招く。
