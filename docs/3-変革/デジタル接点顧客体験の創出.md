# デジタル接点・顧客体験の創出

## 概要

EC（電子商取引・ネット販売）・Webサイト・アプリといった、顧客が会社と出会い・触れ・買う「社外向けのデジタルな接点」をつくり、そこでの体験（UX：User Experience、使い心地）を磨いて新たな価値・売上を生む業務。社内の効率化ではなく、顧客にとっての便利さ・心地よさを起点に据える点が特徴である。多くは事業部・マーケティング・デザインと組んで進める。

> 枝＝`変革 > デジタル接点・顧客体験の創出` ／ フェーズ軸＝**発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善**

---

## あるべき姿

- **顧客が主語** … 「自社が売りたいもの」ではなく「顧客が困っていること・やりたいこと」から接点と体験を設計している
- **接点がつながっている** … Web・アプリ・店舗・問い合わせなど、複数の接点が分断されず、顧客から見て一つの体験になっている
- **事業と一体** … IT単独でなく、事業部・マーケ・デザインと同じ目標を見て協働している。ITが裏方の発注先になっていない
- **安心して使える** … 公開システムとして、セキュリティ・個人情報保護・止まらないこと（可用性）が担保されている
- **基幹とつながる** … 在庫・決済・会員・受注など社内の仕組みと連携し、画面だけでなく業務までつながっている
- **数字で良し悪しが分かる** … 訪問・離脱・購入・継続といった顧客の行動が計測され、改善の根拠になっている
- **作って終わりにしない** … 公開後も計測 → 仮説 → 改善を回し続け、体験が育っていく

## 基本方針（汎用）

会社や扱う商材が違っても、顧客向けデジタル接点づくりの土台となる考え方は共通する。

- **技術起点でなく顧客課題起点** … 「この技術が使える」から始めない。顧客がどこで困り・どこで離れているかを起点に、何を作るかを決める
- **ITだけで顧客体験は作れない** … 体験は商品・価格・接客・物流・ブランドの総合で決まる。事業部・マーケ・デザインとの協働を前提に、ITは「実現と運用」の責任を持つ
- **公開システムは事故が信用に直結する** … 社外に開いた接点は、停止・漏洩・不具合が顧客の離反と信用失墜に直結する。セキュリティ・個人情報保護・可用性を企画段階から織り込む
- **画面の裏の業務までつなぐ** … 見た目の良いWebを作っても、在庫・決済・受注がつながらなければ顧客は完結できない。基幹連携の成否が体験の成否を分ける
- **小さく出して計測で育てる** … 最初から完璧を狙わない。仮説を最小限で世に出し、顧客の実際の行動データで判断し、継続的に改善する
- **顧客接点は会社の顔である** … 個々の機能ではなく、ブランドとして一貫した印象・トーンを保つ。接点ごとにバラバラな体験を出さない

## 主な業務

### 顧客接点・体験の設計支援

顧客がどの接点で・何を期待し・どこでつまずくかを、事業部やデザインと一緒に描き、作るべきものを定義する。

- 顧客の行動の流れ（カスタマージャーニー：認知から購入・継続までの道のり）を可視化し、改善余地の大きい接点を見つける
- 「誰の・どんな課題を・どう解くか」を先に合意する。機能一覧の前に、解きたい顧客課題を言語化する
- ITは技術的な実現可能性・制約・コストを早期に持ち込み、絵に描いた餅を防ぐ。設計を事業部任せにしない

### チャネル構築（EC・Web・アプリ）

定義した体験を、実際に動く顧客向けシステムとして構築・公開する。

- EC・Webサイト・スマートフォンアプリなど、チャネルの特性に応じて作り方（SaaS活用・パッケージ・内製）を選ぶ
- 公開前提で、表示速度・スマートフォン対応・アクセシビリティ（誰もが使えること）・SEO（Search Engine Optimization、検索で見つかりやすくする工夫）を満たす
- 決済・在庫・会員・受注など、社内システムとの連携設計を構築の中心に置く。連携を後回しにしない
- 個人情報の取得・保管・同意取得を法令に沿って設計し、`統制 > データガバナンス` および `統制 > ガバナンス・統制` の方針に従う

### データに基づく継続改善

公開後の顧客行動を計測し、仮説 → 改善 → 検証を回して体験と成果を育てる。

- 訪問・離脱・購入・継続・問い合わせなどを計測し、どこで顧客がつまずいているかを数字で捉える
- A/Bテスト（2案を比べて良い方を採る検証）などで、思い込みでなくデータで改善を判断する
- 効果を売上・継続率・問い合わせ削減といった事業の指標に結びつけ、次の投資判断につなげる

## フェーズ別の進め方

変革のフェーズ軸（発見・相談 → 現状分析・要件 → 企画・提案 → 構築・導入 → 定着化 → 効果測定・改善）に沿って、各段でやること・成果物・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 発見・相談 | 事業部・マーケの困りごとや構想を受け、顧客課題と狙う価値を一緒に言語化する | 課題メモ・狙う価値の仮説 | 「Webを作りたい」という手段の相談で止まり、解くべき顧客課題が曖昧なまま進む |
| 現状分析・要件 | 既存接点・顧客行動・基幹の制約を調べ、要件と連携範囲を固める | 現状分析・要件定義・連携要件 | 基幹連携や個人情報の要件を見落とし、後工程で作り直しになる |
| 企画・提案 | 体験の方向性・概算費用・効果見込み・体制を提案し、投資判断を仰ぐ | 企画書・効果試算・体制案 | IT都合の機能提案になり、事業の成果（売上・継続）と結びつかない |
| 構築・導入 | 設計・開発・連携・テストを行い、セキュリティと可用性を確保して公開する | 稼働システム・連携・公開手順 | 表の画面に注力し、決済・在庫連携やセキュリティ検証が手薄になる |
| 定着化 | 顧客に使われ・社内の運用（更新・問い合わせ）が回る状態にする | 運用フロー・計測の仕込み | 公開して満足し、運用と計測の仕込みが無く、その後の改善ができない |
| 効果測定・改善 | 顧客行動と事業指標を測り、仮説検証で改善を回す | 効果レポート・改善施策 | 数字を見て終わり、改善に回らない／指標が事業成果とずれている |

> 変革はプロジェクト型だが、公開後の「定着化・効果測定・改善」を続けて初めて新価値になる。構築の完了をゴールにしない。

## やらないこと

顧客向けデジタル接点は花形に見えやすく、IT側が抱え込みやすい。**あえてやらない・抱え込まないこと**を明確にし、役割と引き渡し先を守る。

- **事業判断の肩代わり** … 何を売るか・どの顧客を狙うか・価格や販促をどうするかは事業部・マーケの判断。ITが代わりに決めない。ITは実現・運用・データ提供で支える
- **顧客課題が曖昧なまま作り始めること** … 「とりあえずアプリを作る」のような手段先行に乗らない。解く顧客課題が定義されるまで構築に入らない
- **社内向け効率化との混同** … 社内業務の自動化・社内ツール導入は `改善 > ツール／SaaS導入`・`改善 > 手作業の自動化` の領域。社内ヘルプデスクは `運用 > ユーザーサポート`。顧客向けの新価値創出に集中する
- **公開後の運用・セキュリティを設計せず作り逃げること** … 構築だけ請けて、更新・監視・脆弱性対応・問い合わせ対応の体制を決めずに公開しない。運用は `運用 > セキュリティ運用`・`運用 > アプリケーション・SaaS運用保守` と連携して引き渡す
- **計測の仕込みを省くこと** … 後から効果を測れるよう、公開時点で計測を組み込む。「公開できたか」だけを成果にしない
- **データ・個人情報の自己流運用** … 取得した顧客データの扱いを現場判断で決めない。`統制 > データガバナンス` の方針に乗せる

> 線を引くのは責任放棄ではない。**ITが実現・運用・データで強みを出す範囲に集中し、事業判断や恒常運用は正しい担い手へ渡す**ためである。

## よくある課題

- **手段が先行する** … 「ECを作る」「アプリを出す」が目的化し、解くべき顧客課題が定義されないまま進む
- **作って終わる** … 公開がゴールになり、計測・改善の仕組みが無く、体験が育たない
- **事業部との分断** … ITが下請けの発注先になり、事業の目標とIT施策がかみ合わない
- **基幹連携の軽視** … 画面は綺麗でも在庫・決済・受注がつながらず、顧客が購入を完結できない
- **セキュリティ・個人情報の後付け** … 公開直前に問題が露見し、漏洩リスクや手戻りを抱える
- **接点がバラバラ** … Web・アプリ・店舗で情報や体験が分断し、顧客から見て一貫しない

## ベストプラクティス

- **顧客起点の設計手法を使う** … カスタマージャーニーやペルソナ（典型顧客像）で、顧客の流れと痛点を可視化してから作るものを決める
- **指標を先に決める** … 公開前に「何が成功か」（購入数・継続率・離脱率など）を事業部と合意し、計測を組み込んでから出す
- **小さく出して検証する** … 最小限の形（MVP：Minimum Viable Product、検証に足る最小限の製品）で出し、A/Bテストや行動データで方向を見極める
- **連携をアーキテクチャの中心に置く** … 在庫・決済・会員・受注の連携を最初に設計し、`統制 > 全社アーキテクチャ` と整合させる
- **セキュリティ・個人情報を企画段階から織り込む** … 公開システム前提で、設計時から保護策・同意取得・脆弱性対応・監視を組み込む（プライバシー・バイ・デザイン）
- **運用への引き継ぎを設計する** … 公開後の更新・監視・障害対応・問い合わせ対応の担い手と手順を、公開前に決めて運用部門へ渡す

## 成熟度モデル

自社の現在地を測る目安。下から積み上げる。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 単発・受け身 | 言われて作る | 事業部の依頼でWeb・ECを単発で作る。顧客起点・計測・運用設計がなく、作って放置になりがち |
| Lv2 連携・公開品質 | 動くものを安全に出す | 基幹連携・セキュリティ・個人情報を押さえ、公開システムとして安心して使える品質で出せる |
| Lv3 顧客起点・協働 | 課題から設計する | 事業部・マーケ・デザインと協働し、顧客課題を起点に体験を設計。指標を決めて公開する |
| Lv4 計測駆動の改善循環 | データで育てる | 顧客行動と事業指標を計測し、仮説検証で継続的に改善。接点が一貫し、新たな価値・売上を生み続ける |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（連携と公開品質）を外さない**こと。顧客起点の設計も、公開システムが安全に動かなければ価値にならない。
