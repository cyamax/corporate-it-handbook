# 運用品質・管理

## 概要

日々の運用を「回す」だけでなく、運用そのものの質を保ち、崩れないように統制する裏方業務。繰り返す障害の原因を断ち（問題管理）、変更による事故を防ぎ（変更管理）、何を持っているかを把握し続ける（IT資産・ライセンス管理）。ユーザーサポートが「今止まっているものを元に戻す」のに対し、ここは「同じことが二度起きないようにする」「無秩序な変更で壊さない」「持ち物を正しく把握する」を担う。

> 枝＝`運用 > 運用品質・管理` ／ フェーズ軸＝**受付 → 対応 → 解決 → 記録**（＋PDCA：Plan-Do-Check-Act、計画・実行・評価・改善のサイクル）

---

## あるべき姿

- **二度同じ障害が起きない** … 繰り返す障害は対症療法で終わらせず、根本原因を除去する経路がある。火消しの回数そのものが減っていく
- **変更が事故にならない** … 本番への変更が記録され、影響が事前に確認され、戻せる（切り戻せる）状態で行われている。「いつ・誰が・何を変えたか」が後から追える
- **持ち物が分かる** … どの端末・ソフト・ライセンス・契約を、誰が・いくつ持っているかが台帳で把握され、最新に保たれている
- **ライセンス違反がない** … 購入数と利用数が一致し、無断利用・使い回し・契約超過がない。監査が来ても胸を張れる
- **再発防止が記録に残る** … 原因と対策が記録（ナレッジ）として蓄積され、人が代わっても同じ知見を使える
- **無駄が見える** … 使われていないライセンス・遊休端末・重複契約が可視化され、棚卸しで削れる

## 基本方針（汎用）

会社や使うツールが違っても、運用の質を保つ土台となる考え方は共通する。

- **インシデント（対症）と問題（原因）を分ける** … 「業務を今すぐ戻す」のがインシデント対応、「同じ障害を二度と起こさない」のが問題管理。前者を急ぎつつ、後者を別の流れとして必ず残す。両者を混ぜると、復旧で安心して原因が放置される
- **変更は記録し、影響を事前に確認する** … 本番に手を入れる前に「何を・なぜ・いつ変えるか」「壊れたらどう戻すか」を明文化する。無秩序な変更は障害の最大の発生源である
- **リスクに応じて軽重をつける** … すべての変更を重い承認プロセスに乗せない。影響が小さく定型的な変更は素早く通し、影響が大きく不可逆な変更だけ慎重に審査する。重すぎる手続きは形骸化を招く
- **台帳は一つ、常に最新** … 資産・ライセンス・契約の台帳を一元化し、入退社・購入・廃棄のたびに更新する。台帳が古ければ、棚卸しもライセンス管理も成り立たない
- **購入と利用を突き合わせる** … 「買った数」と「実際に使っている数」を定期的に照合する。差分こそが、超過リスク（違反）と無駄（過剰購入）の在りかである
- **原因と対策を記録して終える** … 再発防止も変更も棚卸しも、やって終わりにせず「何が分かったか・何を決めたか」を残す。記録のない改善は、次に同じ轍を踏む

## 主な業務

### 問題管理（再発防止）

繰り返す障害・大きな障害について、**その場の復旧では消えない根本原因を突き止め、除去する**業務（ITILの問題管理に対応）。ユーザーサポートのインシデント管理が「対症療法（症状を抑える）」なら、問題管理は「原因療法（病巣を取る）」にあたる。

- ユーザーサポートで繰り返し発生する障害、または影響の大きかった重大障害を受け取り、**問題（プロブレム）として起票する**。インシデント（個々の障害発生）とは別の単位で管理する
- RCA（Root Cause Analysis、根本原因分析）を行い、なぜ起きたかを掘り下げる。「なぜを繰り返す（なぜなぜ分析）」などで、表面の現象でなく原因にたどり着く
- すぐに原因を消せない場合は、**暫定回避策（ワークアラウンド）**を用意して被害を抑えつつ、恒久対策を別途進める
- 恒久対策が「設定変更」「システム改修」を伴うなら、`運用品質・管理 > 変更管理` の流れに乗せて安全に適用する
- 既知の障害と回避策を**既知のエラー（Known Error）**として記録し、再発時に即座に対処できるようにする
- 障害が起きてから動くだけでなく、傾向分析から「いずれ問題になりそうな兆候」を先に潰す（プロアクティブな問題管理）方向も持つ

> ユーザーサポートは「繰り返す障害の恒久対策は問題管理へ引き渡す」と線を引いている。本ノードはその受け皿である。窓口を火消しで詰まらせないために、原因除去はここで引き受ける。

### 変更管理

本番環境・本番業務に影響する変更を、**事故なく・追跡できる形で適用する**ための統制業務（ITILの変更管理に対応）。

- 変更を「申請 → 影響評価 → 承認 → 適用 → 確認」の流れに乗せ、**いつ・誰が・何を・なぜ変えたかを記録する**
- 変更には必ず**切り戻し手順（ロールバック：元に戻す方法）**を用意する。戻せない変更は、戻せるようにしてから行うのが原則
- 変更をリスクで区分し、軽重をつける（下記）。すべてを重い承認に乗せると現場が回らず、すべてを無審査にすると事故が止まらない

| 区分 | 内容 | 進め方 |
|---|---|---|
| 標準変更 | 影響が小さく手順が確立した定型変更（例：定例パッチ、定型のアカウント設定） | 事前に承認された手順で、都度承認なしに実施。記録は残す |
| 通常変更 | 影響評価と承認が要る変更（例：基幹システムの設定変更、ネットワーク構成変更） | 申請・影響評価・承認を経て、計画的に適用 |
| 緊急変更 | 障害対応など待てない変更 | 簡略化した承認で先に適用し、**事後に必ず記録・レビューする** |

> 緊急変更を「記録なしの特例」にしないこと。スピードのために承認は簡略化してよいが、記録と事後レビューは省かない。ここを崩すと変更管理全体が形骸化する。

### IT資産・ライセンス管理

会社が保有するIT資産（端末・ソフトウェア・ライセンス・契約）を**把握し、最新に保ち、無駄と違反をなくす**業務。SAM（Software Asset Management、ソフトウェア資産管理）とハードウェア資産管理を含む。

- **資産台帳**を整備し、端末・ソフト・ライセンス・契約を一元管理する。CMDB（Configuration Management Database、構成管理データベース）として構成情報まで持つ場合もある
- ハードウェアは「誰が何の端末を持っているか」、ソフトは「何が・いくつ・どの契約で使われているか」を記録する。台帳の鮮度が、この業務の生命線である
- **棚卸し（定期的な現物・利用実態の確認）**を行い、台帳と現実のズレを正す。遊休端末・退職者の残アカウント・使われていないライセンスを洗い出す
- **ライセンスコンプライアンス**を守る。購入数と利用数を突き合わせ、**契約超過（違反）**と**過剰購入（無駄）**の両方を是正する。ライセンス違反は、監査・訴訟・追徴のリスクに直結する
- 廃棄・返却時はデータ消去・ライセンス回収・契約解約まで見届け、**払い終わっているのに使っていない**契約を残さない

> 資産台帳の入口（入社・購入）と出口（退職・廃棄）は、ユーザーサポートの端末管理やセキュリティ運用と隣り合う。本ノードは「会社全体として何をどれだけ持っているか」の把握と、ライセンス・契約の正しさに責任を持つ。

## フェーズ別の進め方

運用のフェーズ軸（受付 → 対応 → 解決 → 記録）に沿って、各段でやること・つまずきを示す。問題管理・変更管理・資産管理に共通する流れとして読む。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 受付 | 問題（繰り返す／重大障害）・変更申請・棚卸し対象を受け取り、起票する | 問題票・変更申請・棚卸し計画 | インシデントのまま問題化されない／変更が口頭・直接適用で記録に乗らない |
| 対応 | 原因分析（RCA）・変更の影響評価とリスク区分・現物と台帳の突き合わせ | 分析結果・影響評価・差分リスト | 復旧で満足して原因究明が止まる／影響評価が形だけ／棚卸しが現物確認を伴わない |
| 解決 | 恒久対策の適用・変更の承認と適用＆切り戻し確認・台帳の是正 | 恒久対策・適用済み変更・是正済み台帳 | 切り戻し手順がない／対策が次の障害を生む／差分を直さず放置 |
| 記録（＋PDCA） | 既知エラー・変更履歴・棚卸し結果を残し、傾向を分析して標準へ還元 | ナレッジ・変更履歴・資産レポート | 記録が個人メモ止まり／棚卸しが一度きりで定着しない／違反・無駄が翌期も再発 |

## やらないこと

運用品質・管理は「品質」「統制」を名に持つため、現場の対応もポリシー策定も何でも引き受ける位置に見られやすい。**あえてやらない・抱え込まないこと**を明確にして、本来の役割を守る。

- **日々の障害一次対応** … 「今止まっている」への一次対応・復旧は `ユーザーサポート > 障害・トラブル対応（インシデント管理）` の仕事。本ノードは復旧後の原因除去を担う。火消しそのものを抱え込むと、再発防止に手が回らなくなる
- **インシデントと問題を一緒くたにすること** … 個々の障害（インシデント）をすべて問題票に起こさない。問題管理は「繰り返す・影響が大きい」ものに絞る。何でも問題化すると、本当に潰すべき原因が埋もれる
- **すべての変更を重量級プロセスに乗せること** … 影響の小さい定型変更まで重い承認を課さない。リスクに応じて軽重をつけ、標準変更は素早く通す。重すぎる手続きは「黙って直す（無記録の変更）」を誘発し、かえって統制を壊す
- **ポリシー・規程そのものの策定** … 変更管理規程・資産管理ポリシー・ライセンス方針といった**ルールの策定・最終承認**は `統制 > ガバナンス・統制`（ITポリシー・規程／内部統制）の領域。本ノードは定められた方針を**運用で回す**側であり、規程の制定主体ではない
- **アカウント・権限の実体管理** … 退職者アカウントの停止・権限の棚卸しといったID統制は `セキュリティ運用 > ID・アクセス権限管理` に従う。資産台帳は「端末・ライセンス」を持つが、アクセス権限の可否判断までは抱え込まない
- **ベンダーとの契約交渉・選定** … ライセンス契約の利用実態は把握するが、ベンダー選定・契約条件の交渉・SLA合意は `統制 > ベンダー・契約管理` の仕事。本ノードは「契約に対して使いすぎ・余りがないか」を見る側
- **業務改善プロジェクトとしての作り込み** … 自動化や新ツール導入による効率化は `改善` の領域。本ノードはあくまで運用品質の維持・統制であり、改善は適切な行き先へ上げる

> ここで線を引くことは「丸投げ」ではない。**品質維持に集中させ、引き渡し先を明示する**ためのもの。原因除去・変更統制・資産把握という、地味だが崩れると全体が壊れる持ち場を守る。

## よくある課題

- **復旧で満足して原因が残る** … インシデントは閉じるが問題が起票されず、同じ障害が何度も再発する
- **無秩序な変更による障害** … 記録のない直接変更・「ちょっとだけ」の手直しが積み重なり、原因不明の障害を生む
- **緊急変更が常態化** … 何でも「緊急」で承認を飛ばし、記録と事後レビューが省かれ、変更管理が有名無実になる
- **台帳が古い** … 入退社・購入・廃棄が反映されず、誰が何を持っているか分からない。棚卸しのたびに現実と大きくズレる
- **ライセンスの実態不明** … 買った数と使っている数が合わず、超過（違反リスク）にも過剰購入（無駄）にも気づけない
- **退職者の残骸** … 退職後も使われない端末・ライセンス・アカウントが残り、コストと情報漏洩リスクになる
- **再発防止が記録されない** … 原因と対策が個人の頭の中にとどまり、担当が代わると同じ失敗を繰り返す

## ベストプラクティス

- **インシデントと問題を別管理にする** … チケットの種別を分け、「繰り返す／重大」なインシデントから問題票を起票する運用にする。問題管理の入口を仕組みで担保する
- **なぜなぜ分析の定着** … 重大障害は復旧後に必ず振り返り、表面の現象でなく根本原因まで掘る。原因と恒久対策・暫定回避策を既知エラーとして記録する
- **変更のリスク区分と変更カレンダー** … 標準・通常・緊急の区分を定義し、変更予定を一覧（変更カレンダー）で見える化する。変更同士の衝突や繁忙期の重大変更を避ける
- **切り戻し前提の変更** … 変更計画に必ずロールバック手順を含め、戻せない変更は段階適用やリハーサルで備える
- **資産台帳の一元化と自動収集** … 台帳を一つに集約し、可能な範囲で端末・ソフトの情報を自動収集（インベントリ収集）して手入力の漏れを減らす
- **定期棚卸しとライセンス突合** … 棚卸しを定例化し、現物・利用実態と台帳を照合する。購入数と利用数の突き合わせで超過と無駄を同時に潰す
- **入退社・廃棄プロセスへの組み込み** … 台帳更新を入退社・購入・廃棄のワークフローに組み込み、「イベントが起きたら台帳が必ず動く」状態にする
- **傾向の月次レビュー** … 再発障害・変更失敗・遊休資産を定期的に振り返り、上位の原因を改善（Grow）や統制（Govern）へ還元する

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 場当たり | 復旧と無記録変更 | 障害は復旧して終わり、問題管理なし。変更は記録なく直接適用。資産台帳は不在か放置 |
| Lv2 記録の確立 | 残す仕組み | 重大障害は問題票に起票、変更は申請・記録され、資産台帳が存在し更新される |
| Lv3 標準化 | ルールで回す | 変更のリスク区分・棚卸しの定例化・ライセンス突合・既知エラー運用が整い、誰がやっても一定品質 |
| Lv4 予防と最適化 | データで先回り | 傾向分析から再発を予防し、変更失敗率が下がり、遊休資産・余剰ライセンスが継続的に削減される。問題が起きる前に手を打つ（プロアクティブ）方向へ向かう |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（記録の確立）を固める**こと。問題票・変更記録・資産台帳という記録が無ければ、Lv3 以降の標準化も分析も成り立たない。
