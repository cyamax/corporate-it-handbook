# セキュリティ運用

## 概要

社内の人・端末・データを脅威から守るため、アクセス権限を正しく保ち、既知の弱点をふさぎ、事故が起きたら速やかに食い止める日々の防御業務。守るための仕組みづくり（ポリシー策定）ではなく、定められた方針を現場で回し続ける執行が役割である。

> 枝＝`運用 > セキュリティ運用` ／ フェーズ軸＝**受付 → 対応 → 解決 → 記録**（＋PDCA：Plan-Do-Check-Act、計画・実行・評価・改善のサイクル）

---

## あるべき姿

- **権限が最小** … 各人が業務に必要な分だけの権限を持つ（最小権限の原則）。「念のため広めに」を許さず、付与した権限には必ず根拠がある
- **棚卸しが回る** … 誰が何にアクセスできるかが定期的に見直され、不要な権限・休眠アカウントが残らない
- **退職で即失効** … 退職・異動が起きたら、アクセス権が遅滞なく止まる。手作業頼みで取りこぼさない
- **本人確認が固い** … 重要な入口にはMFA（多要素認証：パスワードに加えもう一つの要素で本人を確かめる仕組み）がかかり、パスワードだけで突破されない
- **弱点に期限がある** … 既知の脆弱性やパッチが放置されず、危険度に応じた期限内に対処される
- **事故に備えがある** … インシデントの連絡先・初動・封じ込め手順が事前に決まっており、起きてから慌てない
- **記録が残る** … 権限付与・パッチ適用・インシデント対応の記録（ログ）が残り、後から追える・監査に出せる

## 基本方針（汎用）

会社や使うツールが違っても、セキュリティ運用の土台となる考え方は共通する。

- **最小権限の原則** … 「必要な人に・必要な範囲だけ・必要な期間だけ」を徹底する。デフォルトは「権限なし」で、付与は申請と承認を経る。広く付けるのは簡単だが、後で絞るのは難しい
- **付与より失効を重んじる** … 権限を与える業務より、要らなくなった権限を確実に止める業務に注意を割く。残った権限こそが侵入経路になる。退職・異動・プロジェクト終了は失効の起点である
- **棚卸しは定期で機械的に** … 「気づいたら見直す」ではなく、四半期・半期など周期を決めて全件を棚卸しする。判断は権限の付与者ではなく、業務を分かっている管理者に確認させる
- **方針は統制から、執行は運用で** … 何を守るか・どこまで許すかのルールは統制（ガバナンス）が決める。運用はそれを現場で回す。運用が独自にルールを発明しない
- **多層で守る** … 一つの対策に頼らない。入口（認証・MFA）、弱点（パッチ）、出口（ログ監視）を重ね、どれか一つが破られても被害を抑える
- **記録ファースト** … 誰がいつ何の権限を得たか、いつパッチを当てたか、インシデントに何をしたかを記録する。記録の無い対応は、監査・再発防止・引き継ぎのすべてを不可能にする
- **危険度で優先する** … 脆弱性もインシデントも「来た順」ではなく、悪用のされやすさと影響の大きさで優先順位をつける

## 主な業務

### ID・アクセス権限管理（アカウント発行〜棚卸し・権限）

社員・委託先のアカウントを発行し、業務に必要な権限を最小限で与え、不要になったら確実に止める、IAM（Identity and Access Management：ID・アクセス管理）の実体管理。**このハンドブックでは、アカウント・権限の実体管理はすべてここに一本化する**（ユーザーサポート側の入退社対応も、権限の可否判断はここに従う）。

- **発行（入社・参画）** … 申請と承認を経てアカウントを作る。役割（ロール）に応じた権限のひな型を用意し、人ごとに手作業で組まない。職務分掌（SoD：Segregation of Duties、相互牽制のため一人に権限を集中させない考え方）に反する組み合わせを与えない
- **変更（異動・役割変更）** … 異動時は「新しい権限を足す」だけでなく**古い権限を外す**。これを怠ると、在籍年数とともに権限が膨らむ（権限のクリープ）
- **棚卸し（定期レビュー）** … 周期を決めて全アカウント・全権限を棚卸しし、業務を知る管理者に「本当に要るか」を確認させる。休眠アカウント・退職者の残骸・過剰権限を洗い出して落とす
- **失効（退職・参画終了）** … 退職・契約終了の情報を受けたら、遅滞なくアクセスを止める。人事・各部署からの退職連絡を確実に受け取る経路を用意し、手作業の取りこぼしを防ぐ
- **認証の強化** … 重要なシステム・管理者権限にはMFAを必須にする。特権アカウント（管理者）は通常業務と分け、利用を記録する

> 退職時の即時失効・定期棚卸し・最小権限は、ID管理の三本柱である。どれか一つでも崩れると、他の対策をいくら積んでも穴が残る。

### 脆弱性・パッチ対応

OS・ソフトウェア・機器に見つかった既知の弱点（CVE：Common Vulnerabilities and Exposures、既知の脆弱性に振られる識別番号）を把握し、危険度に応じた期限内にパッチ適用などで塞ぐ。

- **把握** … 自社が使う資産（端末・サーバー・ソフト・SaaS）を台帳で押さえ、どこに何のバージョンがあるかを分かる状態にする。守る対象が分からなければ弱点も追えない
- **評価** … 出てきた脆弱性が自社に影響するか、悪用のされやすさはどうかで危険度を判定する。深刻度の指標（CVSS：脆弱性の深刻度を数値化した共通指標）を目安にしつつ、自社での露出度（インターネットに面しているか等）も加味する
- **適用** … 危険度ごとに対応期限（SLA：Service Level Agreement、対応時間の合意）を決めて当てる。緊急（実際に悪用が出ている等）は最優先で、定常パッチは定例の窓で計画的に
- **検証と例外** … 適用が業務を止めないか確認する。すぐ当てられない場合は放置せず、暫定の緩和策（回避策）と期限・責任者を記録した例外として管理する

### セキュリティインシデント対応

マルウェア感染・不正アクセス・情報漏洩・フィッシング被害など、実際に起きた（または疑われる）セキュリティ事故を受け付け、被害の拡大を止め、復旧し、記録する。

- **受付・検知** … 通報（社員からの「怪しいメールを開いた」等）と検知（アラート）の両方を受ける窓口を決め、疑いの段階で気軽に上げられるようにする。「報告したら怒られる」空気は初動を遅らせる
- **初動・封じ込め** … まず被害の拡大を止める。感染端末をネットワークから隔離する、侵害された疑いのアカウントを停止する、といった封じ込めを原因究明より先に行う
- **調査・復旧** … 影響範囲（何が漏れたか・どこまで広がったか）を調べ、安全を確認してから復旧する。証拠（ログ・端末状態）を消さないよう保全する
- **報告・連携** … 重大なものは、あらかじめ決めた連絡経路で経営・統制・必要なら外部（専門業者・関係当局）へエスカレーションする。判断を現場で抱え込まない

> インシデント対応の良し悪しは初動で決まる。「封じ込めを原因究明より先に」「疑いの段階で上げる」の二つが、被害規模を分ける。

## フェーズ別の進め方

運用のフェーズ軸（受付 → 対応 → 解決 → 記録）に沿って、各段でやること・つまずきを示す。3業務に共通する流れとして読む。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 受付 | 申請・脆弱性情報・インシデント通報を受け、本人確認と危険度づけを行う | 申請チケット・脆弱性一覧・インシデント起票 | 退職連絡が届かない／通報をためらわせる空気／緊急度判断が人による |
| 対応 | 権限付与・パッチ適用・封じ込めを実施。緊急は最優先で動く | 対応ログ・隔離/停止の記録 | 封じ込めより原因究明を優先／「念のため広めに」付与／パッチを業務影響を理由に先送り |
| 解決 | 権限の確認・パッチ検証・復旧を確認し、関係者へ通知してクローズ | 完了チケット・復旧確認 | 退職者の権限失効を確認せず完了／例外を期限なしで放置 |
| 記録（＋PDCA） | 対応を記録し、棚卸し・傾向分析・教訓を次に還元する | 棚卸し結果・インシデント報告書・改善提案 | 記録が個人メモ止まり／棚卸しが形骸化／教訓が仕組みに反映されない |

## やらないこと

セキュリティ運用は「セキュリティに関わること全部」を抱え込みやすい。**あえてやらない・抱え込まないこと**を明確にして、執行という本来の役割を守る。

- **ポリシー・規程そのものの策定** … 「何を守るか」「どこまで許すか」の方針・規程・基準を作るのは `ガバナンス・統制 > ITポリシー・規程` の役割。運用は決まった方針を執行する。現場判断でルールを発明・改変しない
- **権限付与の可否を運用裁量で判断すること** … 「この人にこの権限を与えてよいか」の業務上の妥当性は、申請者の上長・データ管理者が決める。運用は申請と承認を経て実行する役であり、承認者ではない。口頭依頼でそのまま付与しない
- **セキュリティの恒久的な仕組み改善を運用で抱えること** … 認証基盤の刷新、ゼロトラスト化、SIEM（ログを集約・相関分析する仕組み）導入といった**新しい仕組みづくり**は、効率化が目的なら改善（Grow）の `ツール／SaaS導入` 等、方針・基準づくりなら統制側へ渡す。運用は日々の執行に集中する
- **事業継続・災害復旧を抱えること** … システムを止めない・落ちても戻すという**可用性**側の備え（BCP／DR）は `可用性・事業継続管理 > BCP／DR` の領域。機密性を守るセキュリティ運用とは別系統として切り分ける
- **端末そのものの配布・回収・キッティング** … 物理的な端末ライフサイクルは `ユーザーサポート > 端末管理` が担う。セキュリティ運用が見るのは、その端末に紐づくアカウントと権限、適用すべきパッチの側である
- **内部統制監査・J-SOX対応そのものの設計** … 統制の有効性を評価・監査する枠組みづくりは `ガバナンス・統制 > 内部統制・監査対応`。運用は監査に出せる記録（証跡）を残す側に徹する
- **無期限の例外を黙認すること** … 「今は当てられない」「この人だけ特別に」を期限・責任者なしで放置しない。例外は必ず期限と緩和策と承認をセットで記録し、繰り返すなら方針側の見直しへ上げる

> ここで線を引くことは「守りを放棄する」ことではない。**運用を執行という本来やるべきことに集中させ、方針づくり・仕組みづくり・引き渡し先を明示する**ためのもの。

## よくある課題

- **権限のクリープ（肥大化）** … 異動のたびに権限が足され、外されないまま膨らむ。在籍年数が長い人ほど不必要に強い権限を持つ
- **退職者アカウントの残存** … 退職連絡が情シスに届かず、止め忘れたアカウントが生き残る。最も狙われやすい侵入口になる
- **棚卸しの形骸化** … 棚卸しはするが、管理者が中身を見ずに「全部OK」で承認し、何も落ちない
- **パッチの先送り** … 「業務に影響が出るかもしれない」を理由に当て続けず、既知の弱点を抱えたまま走る
- **MFA未導入・抜け穴** … 一部のシステムや特権アカウントだけMFAがかかっておらず、そこが突破される
- **通報されないインシデント** … 「報告すると責められる」空気のせいで初動が遅れ、被害が広がってから発覚する
- **記録なしの対応** … 緊急対応に追われ、誰が何をしたかが残らず、後から追跡も再発防止もできない

## ベストプラクティス

- **ロールベースの権限管理（RBAC）** … 人ごとに権限を組まず、役割（ロール）に権限のひな型を結びつける。RBAC（Role-Based Access Control、役割に基づくアクセス制御）にすると付与・棚卸し・失効が一気に楽になる
- **入退社・異動の自動連動** … 人事システムの入退社・異動情報をアカウントの発行・変更・失効に連動させ、手作業の取りこぼしをなくす。少なくとも退職の即時失効は自動化を優先する
- **定期棚卸しの仕組み化** … 周期を決め、管理者に「要否」をチェックさせるレビューを回す。承認の記録を残し、形だけのレビューを許さない
- **MFAの全面適用** … 重要システムと管理者権限を最優先に、MFAを全面適用する。特権アカウントは通常業務と分離し、利用を記録する
- **脆弱性のSLA運用** … 危険度ごとに対応期限を決める（例：緊急は即時〜数日、高は所定日数内、中以下は定例窓）。期限と適用状況をダッシュボードで見える化し、未対応を放置しない
- **資産台帳の整備** … 端末・サーバー・ソフト・SaaSの台帳を最新に保つ。守る対象が見えていなければ、弱点も権限も管理できない（台帳は `IT資産・ライセンス管理` と整合させる）
- **インシデント手順書（プレイブック）と訓練** … 連絡経路・初動・封じ込め・エスカレーション基準を事前に文書化し、机上訓練で動けるようにする。手順は事故が起きてからでは作れない
- **ログ集約と監視** … 認証・操作・アクセスのログを集約し、不審な兆候を検知できるようにする。記録は事後追跡と監査証跡の両方に効く

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 場当たり | 個別対応 | 権限は申請のたびに手作業で広めに付与。棚卸しなし・退職の失効は気づいたら・パッチは思い出したら・事故は起きてから考える |
| Lv2 受付の整備 | 記録が残る | 権限付与・パッチ・インシデントが申請と記録を伴って処理される。退職失効の連絡経路と、インシデントの窓口・初動手順がある |
| Lv3 標準化 | 仕組みで回す | ロール（RBAC）で権限が管理され、定期棚卸し・脆弱性SLA・MFA・プレイブックが定着。誰が対応しても一定品質 |
| Lv4 自動化と先回り | データで前へ | 入退社が人事連動で自動失効し、脆弱性・ログが見える化され、棚卸しと訓練で穴が事前にふさがる。事故が起きる前に手を打つ（プロアクティブ） |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（退職の即時失効と記録）を固める**こと。残った権限と止め忘れたアカウントは、上位の対策をいくら積んでも残り続ける最大の穴である。

## 参考資料

- ITIL（ITサービス管理のベストプラクティス集）／ ISO/IEC 27001（情報セキュリティマネジメントの国際規格）… アクセス管理・脆弱性管理・インシデント管理の考え方の出典
- 最小権限の原則・職務分掌（SoD）・多要素認証（MFA）に関する一般的な情報セキュリティの基本原則
- NIST のインシデント対応の枠組み（準備 → 検知・分析 → 封じ込め・根絶・復旧 → 事後対応）… 初動と封じ込めを重んじる考え方の参考
- 関連ノード：`ガバナンス・統制 > ITポリシー・規程`／`ガバナンス・統制 > 内部統制・監査対応`（方針策定・監査）、`ユーザーサポート > 端末管理`（端末の配布・回収）、`可用性・事業継続管理 > BCP／DR`（可用性側の備え）、`運用品質・管理 > IT資産・ライセンス管理`（資産台帳）
