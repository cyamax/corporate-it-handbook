# ユーザーサポート（ヘルプデスク）

## 概要

社員からの問い合わせ・依頼・トラブルを受け付け、業務を止めずに解決へ導く窓口業務。社内ITにおける最も普遍的な接点であり、ここで蓄積される記録が他の業務（問題管理・改善・統制）の出発点になる。

> 枝＝`運用 > ユーザーサポート` ／ フェーズ軸＝**受付 → 対応 → 解決 → 記録**（＋PDCA：Plan-Do-Check-Act、計画・実行・評価・改善のサイクル）

> このページは、後述の**サービスデスク**（サービス全体を良くし続ける広い窓口）を目標として描く。**ヘルプデスク**（特定の問題を直すインシデント対応）は、その中核機能の一つに位置づく。

---

## ヘルプデスクとサービスデスクの違い

「ヘルプデスク」と「サービスデスク」は同じ「問い合わせ窓口」として混同されやすいが、担う役割の広さが違う。ヘルプデスクが「特定の技術的問題を直す（インシデント対応）」窓口であるのに対し、サービスデスクはITサービス全体の品質・継続性・改善まで担う、より広い単一窓口である。両者を区別しておくと、自分たちの現在地と目指す先がはっきりする。

> 用語：本ページでは ITIL（ITサービス管理のベストプラクティス集）の用法に合わせ、広い窓口機能を**サービスデスク**と呼ぶ。「サポートデスク」もほぼ同義で使われる。

| 観点 | ヘルプデスク | サービスデスク |
|---|---|---|
| 主目的 | 特定の技術的問題を直す（インシデント対応） | ITサービスの品質・継続性・改善を保つ |
| 視点 | 事象単位・復旧優先（受動的・その場を直す） | サービス単位・継続的改善（能動的・先回り） |
| 扱う範囲 | 主に障害・トラブル（インシデント） | インシデントに加え、依頼対応・傾向分析・再発防止・SLA（サービス品質の合意）管理・他プロセスへの橋渡し |
| 時間軸 | 今この問題を直す | 今直しつつ、将来起きにくくしサービスを良くする |
| 成功の基準 | 早く正しく直したか | サービスが安定し、改善し続けているか（満足度・SLA遵守・再発率） |
| 位置づけ | サービスデスクの中核機能（一次対応の現場） | 単一窓口（SPOC：Single Point of Contact、問い合わせを一手に引き受ける唯一の窓口）の全体機能 |

### ヘルプデスク：特定の問題を直す

「壊れた・動かない・分からない」を個別に解決する一次対応である。受動的で、目の前のインシデントを一刻も早く復旧させることに集中する。壊れたら直す（break-fix）が基本姿勢となる。

- 強みは即応性。社員の「今困っている」に最短で応える
- 弱みは、直すだけでは同じ問題が再発し、サービス全体の良し悪しが見えないこと
- このハンドブックでは「主な業務 > 障害・トラブル対応（インシデント管理）」が、ヘルプデスクの中心にあたる

### サービスデスク：サービスを良くし続ける

個別対応にとどまらず、ITサービス全体の単一窓口として、品質・継続性・改善に責任を持つ。インシデント対応も含むが、そこから傾向を読み、再発防止（`運用品質・管理 > 問題管理`）・自己解決の仕組み・SLA管理・他プロセスへの橋渡しまで担う。

- 受動的な火消しから、能動的にサービスを安定・向上させる立場へ広げたものがサービスデスクである
- 単発の「直した／直していない」ではなく、サービスが継続的に良くなっているかで成否を測る
- 後述の「成熟度モデル」は、ヘルプデスク（個別対応）からサービスデスク（改善循環）へ広がる道筋に対応する

## あるべき姿

- **窓口が一つ** … 社員が「どこに聞けばいいか」で迷わない。問い合わせ窓口が一つに集約されている
- **記録が残る** … やり取りが記録として残り、誰が対応しても同じ品質で、後から振り返れる
- **自己解決できる** … よくある問いは手順書やFAQ（よくある質問集）へ誘導され、繰り返し聞かれない
- **恒久対策へつながる** … その場しのぎで終わらせず、繰り返す障害は根本対策へ引き渡す経路がある
- **数字で見える** … 対応の速さ・件数が数字で見え、改善の判断材料になっている
- **安全に扱える** … 機密情報やアカウントの操作は、依頼者の本人確認と記録（ログ）を前提に行われる

## 基本方針（汎用）

会社や使うツールが違っても、ヘルプデスクの土台となる考え方は共通する。

- **窓口は一つに（単一窓口）** … 口頭・チャット・メール・立ち話を一本の受付ルートに集約する。窓口が分散すると記録が漏れ、対応の偏りと取りこぼしが生まれる
- **記録ファースト** … 「対応したが記録がない」をなくす。記録の無い対応は、件数把握・再発防止・引き継ぎのすべてを不可能にする
- **依頼（リクエスト）と障害（インシデント）を分ける** … 「やってほしい」と「壊れて困っている」は緊急度も進め方も違う。受付時点で仕分ける
- **一次対応と恒久対策を分ける** … ヘルプデスクの役割は「まず業務を復旧させる」こと。根本原因の除去は別業務（問題管理）へ渡し、窓口を恒久対策で詰まらせない
- **依頼者の本人確認を行う** … 権限付与・アカウント操作・情報開示を伴う依頼では、なりすまし防止のため依頼者本人であることを確認する。電話やチャットの口頭依頼をそのまま実行しない。ヘルプデスクは情報漏洩・なりすましの最前線である
- **自己解決を増やす** … 人手での対応は有限。FAQ・手順書・自動化で、問い合わせそのものを減らす方向に投資する
- **優先度は影響と緊急度で決める** … 「声の大きい人順」「来た順」ではなく、業務への影響範囲と急ぎ具合で優先順位をつける

## 主な業務

### 問い合わせ・依頼対応（リクエスト管理）

「アカウントを作ってほしい」「ソフトを入れたい」「使い方を教えてほしい」といった、**あらかじめ手順を決められる定型の依頼**を捌く（ITILのリクエストフルフィルメントに対応）。

- よくある依頼はメニュー化し、受付項目・承認者・標準対応時間をあらかじめ決めておく
- 承認が要るもの（権限付与・有償ライセンス）は、**申請を受理し承認者へ回す**。ヘルプデスクは実行役であり、承認者ではない。口頭依頼で済ませない
- 定型依頼ほど自動化・セルフサービス化の効果が大きい

### 障害・トラブル対応（インシデント管理）

「メールが送れない」「PCが起動しない」「システムが落ちた」といった、**業務が止まっている状態を一刻も早く元に戻す**対応。

- まず復旧を優先する。原因究明より業務再開（暫定対応でもよい）を先に置く
- 影響範囲（一人か・部署全体か・全社か）で優先度を切り替える
- 同種の障害が繰り返すなら、その場の復旧で終えず `運用品質・管理 > 問題管理` へ引き渡す

### 端末管理（入退社・キッティング・周辺機器）

PC・スマートフォン・周辺機器の、**配布から回収までのライフサイクル**を管理する。

- 入社時：アカウント発行・端末の初期設定（キッティング：PCを業務で使える状態に整える作業）・必要ソフトの配布
- 在職中：故障対応・貸出・周辺機器の手配
- 退職時：**端末回収とアカウント停止を確実に**。ここが漏れると情報漏洩・ライセンス無駄・不正アクセスの温床になる
- 「誰が何の端末を持っているか」の台帳を常に最新に保つ
- 私物端末の業務利用（BYOD：Bring Your Own Device）を会社が許可している場合、その業務利用部分も支援対象になる

## フェーズ別の進め方

運用のフェーズ軸（受付 → 対応 → 解決 → 記録）に沿って、各段でやること・つまずきを示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 受付 | 窓口で受け、依頼／障害を仕分け、本人確認と優先度づけ | 受付チケット（起票） | 口頭受付で記録が残らない／緊急度の判断が人による |
| 対応 | 担当を決め、一次対応・切り分けを行う | 対応ログ・暫定復旧 | 担当の抱え込み・属人化／復旧より原因究明を優先してしまう |
| 解決 | 復旧・依頼完了を確認し、依頼者に通知してクローズ | 解決済みチケット | 依頼者の確認を取らずに勝手にクローズ |
| 記録（＋PDCA） | 対応内容を残し、傾向を集計、FAQ・自動化へ還元 | ナレッジ・FAQ・件数レポート | 記録が個人メモ止まり／集計しても改善に回らない |

## やらないこと

ヘルプデスクは「何でも引き受ける窓口」になりやすい。**あえてやらない・抱え込まないこと**を明確にして、本来の役割を守る。

- **何でも屋になること** … 「ITに少しでも関わる雑務」をすべて吸収しない。電源工事の手配、私物の私的利用（業務外）のトラブル、業務アプリの仕様判断などは、本来の担当へ案内する
- **恒久対策をヘルプデスクで抱えること** … 繰り返す障害の根本対策は `運用品質・管理 > 問題管理` へ渡す。窓口で都度しのぎ続けると、いつまでも同じ火消しが続く
- **記録を飛ばした属人対応** … 「自分が分かっているから」で記録なしに対応しない。担当が不在になった瞬間に再現不能になる
- **権限・アカウントの承認を代行すること** … 権限付与の可否判断を窓口の裁量で行わない。申請を承認経路に乗せる（権限・ID管理の実体は `セキュリティ運用 > ID・アクセス権限管理` に従う）
- **シャドーITの黙認** … シャドーIT（情シスの管理外で導入・利用されるツール）を「便利だから」と見て見ぬふりしない。リスクを記録し、正規の検討ルートへ上げる
- **無秩序な特例対応** … 一人に認めた例外が前例になり、標準が崩れる。例外は記録し、繰り返すなら標準側を見直す

> ここで線を引くことは「冷たい対応」ではない。**窓口を本来やるべきことに集中させ、引き渡し先を明示する**ためのもの。断るのではなく、正しい行き先へ案内する。

## よくある課題

- **属人化** … 「あの人しか分からない」状態。担当の休暇・退職で対応が止まる
- **記録されない対応** … 立ち話・個人チャットで完結し、件数も内容も把握できない
- **同じ問い合わせの繰り返し** … FAQ・手順書が無く、毎回ゼロから対応している
- **緊急対応に追われ改善が進まない** … 火消しで一日が終わり、再発防止に手が回らない
- **依頼と障害が混在** … 緊急の障害が、急がない依頼に埋もれて後手に回る

## ベストプラクティス

- **問い合わせ管理ツールの導入** … チャット・メール・フォームの問い合わせをチケット化し、一元管理する。「記録ファースト」を仕組みで担保する
- **FAQ・ナレッジベースの整備** … 繰り返す問いを記事化し、問い合わせ前に自己解決させる。新しい問いが出たら記事を足す運用にする
- **SLAの設定** … SLA（Service Level Agreement、サービス品質の合意）を定め、依頼者と社内ITの期待値を揃える。**一次回答までの時間**と**解決までの時間**は分けて設定する（例：障害は一次回答1営業日以内、解決は影響度別）。守れない約束は掲げない
- **優先度マトリクス** … 影響範囲 × 緊急度で優先度を機械的に決め、判断のばらつきをなくす
- **定型依頼のセルフサービス化** … パスワード再設定・ソフト配布など、頻度が高く手順が固い依頼から自動化・申請フォーム化する
- **定期的な傾向分析** … 件数・種類・対応時間を月次で振り返り、上位の問い合わせを自動化・FAQ・改善（Grow）へ還元する

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 個別対応 | 場当たり | 窓口バラバラ・記録なし・属人対応。担当者の記憶頼み |
| Lv2 窓口集約 | 受付の一本化 | 問い合わせが一つの窓口に集まり、チケットで記録が残る |
| Lv3 標準化 | 仕組みで回す | FAQ・SLA・優先度ルールが整い、誰が対応しても一定品質 |
| Lv4 改善循環 | データで前へ | 傾向分析から自動化・セルフサービス化が回り、問い合わせの総量が減る。問題が起きる前に手を打つ（プロアクティブ）方向へ向かう |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（窓口集約と記録）を固める**こと。記録が無ければ Lv3 以降の標準化も分析も成り立たない。
