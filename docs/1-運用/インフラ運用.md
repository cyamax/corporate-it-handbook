# インフラ運用

## 概要

サーバー・ネットワーク・クラウドといった、業務システムが乗る土台（インフラ）を安定して動かし続ける業務。アプリケーションやサービスの「下」にあって普段は意識されないが、ここが止まると全社の業務が止まる。日々の稼働維持・変更の反映・障害復旧と、それらを支える監視とバックアップを担う。

> 枝＝`運用 > インフラ運用` ／ フェーズ軸＝**受付 → 対応 → 解決 → 記録**（＋PDCA：Plan-Do-Check-Act、計画・実行・評価・改善のサイクル）

---

## あるべき姿

- **構成が把握できている** … どこに何のサーバー・回線・クラウド資源があり、何に使われているかが台帳で分かる。「誰も触れない・正体不明のサーバー」が無い
- **手で触らず再現できる** … 設定変更が手作業の一発勝負ではなく、手順またはコード（IaC：コードでインフラを構築・管理する手法）で再現できる
- **異常に先に気づける** … 利用者が「遅い・つながらない」と言う前に、監視が異常を検知し通知している
- **戻せる状態がある** … 変更前の状態・データに確実に戻せる。バックアップは取るだけでなく、戻せることが確認されている
- **冗長で粘り強い** … 単一の機器・回線・拠点の故障で全体が止まらない。重要な部分に冗長性がある
- **権限と変更が記録される** … 誰がいつ何を変えたか、誰が基盤を操作できるかが記録・制限されている

## 基本方針（汎用）

オンプレミス（自社設置）でもクラウドでも、製品が違っても、インフラ運用の土台となる考え方は共通する。

- **現状を正しく把握してから動かす** … 構成・依存関係・責任分界が分からないまま変更すると、想定外の波及を生む。台帳と構成図を起点にする
- **変更は小さく・戻せる形で** … 一度に大きく変えない。影響範囲を絞り、切り戻し手順を用意してから変更に着手する
- **監視ありきで運用する** … 「動いている」を感覚ではなく数値で持つ。監視が無い基盤は、壊れて初めて気づく
- **バックアップは「戻せて初めて成立」** … 取得しているだけでは不十分。定期的に復元を試し、戻せることを確認する
- **冗長化と復旧は別物として両方持つ** … 冗長化（壊れても止めない）と復旧（壊れた後に戻す）は目的が違う。片方では守り切れない
- **責任分界点を明確にする** … クラウド・データセンター・回線事業者と自社の責任の境目（どこまでが事業者の管理か）を理解し、自社が守る範囲を取りこぼさない
- **基盤への変更は記録に残す** … 基盤操作は影響が広く、属人化・無記録だと事故時に原因が追えない。変更と権限を記録する

## 主な業務

### サーバー・クラウド運用

物理サーバー・仮想サーバー・クラウド上の計算資源（コンピュート）を、**業務に必要な状態で動かし続ける**業務。

- OS・ミドルウェアの稼働維持、設定変更、パッチ適用（脆弱性対応そのものは `セキュリティ運用 > 脆弱性・パッチ対応` に従う）
- リソース（CPU・メモリ・ディスク）の利用状況を見て、不足の兆候を先に手当てする
- 構成は手作業ではなく、可能な範囲で手順書または IaC で再現可能にする
- クラウドでは、責任共有モデル（事業者と利用者で守備範囲を分担する考え方）を理解し、自社が担う設定・権限・データを管理する
- 「いつの間にか増えた・誰も使っていない資源」を棚卸しし、無駄なコストと管理外の資産を減らす

### ネットワーク運用

社内LAN・無線・拠点間接続・インターネット回線・クラウドへの接続といった、**通信の経路を保つ**業務。

- 回線・機器（スイッチ・ルーター・ファイアウォール・無線アクセスポイント）の稼働維持と設定管理
- 通信の到達性・速度・遅延を監視し、「つながらない・遅い」の切り分けを行う
- 拠点・リモートワーク・クラウド接続の経路を設計どおりに保つ（接続方式の設計・刷新そのものはプロジェクト側へ）
- 重要な回線・経路は冗長化し、一本の障害で全社が孤立しないようにする
- ネットワーク構成図と機器設定（コンフィグ）を最新に保ち、変更を記録する

### 監視・バックアップ

基盤の異常を早く捉える**監視**と、失われたデータ・状態を戻せるようにする**バックアップ**。インフラ運用の生命線。

- 監視は「死活（動いているか）」「リソース（余力があるか）」「サービス（利用者目線で使えるか）」を層で押さえる
- 通知は「対応が必要なものだけ鳴る」状態に育てる。鳴りっぱなしは見られなくなる（アラート疲れ）
- バックアップは **3-2-1 の原則**（データを3つ持ち・2種類の媒体に・1つは別の場所に）を基本に設計する
- 取得して終わりにせず、**定期的に復元テストを行い「戻せること」を確認する**。復旧手順は実際に試す
- 目標復旧時点（RPO：どこまで前のデータに戻せるか）と目標復旧時間（RTO：どれだけ早く戻せるか）を業務要件から決める

> 監視とバックアップは「平時に手を抜くと、有事に致命傷になる」典型である。普段の地味な確認こそが、止まらない基盤を支える。

## フェーズ別の進め方

運用のフェーズ軸（受付 → 対応 → 解決 → 記録）に沿って、各段でやること・つまずきを示す。インフラ運用では「受付」に、利用者からの申告だけでなく**監視からの自動検知**が含まれる点が特徴である。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 受付 | 監視アラート・障害申告・変更依頼を受け、影響範囲と緊急度を判断する | 起票（インシデント／変更）・初動記録 | アラートが多すぎて重要なものが埋もれる／変更依頼を口頭で受け記録が残らない |
| 対応 | 切り分け（基盤かアプリか・どの層か）を行い、冗長系への切替・暫定復旧・変更適用を実施する | 対応ログ・暫定復旧・切戻し可能な変更 | 切り分けできず手当たり次第に触る／切戻し手順を用意せず変更し、戻せなくなる |
| 解決 | 復旧・変更完了を確認し、関係者へ通知してクローズする。バックアップからの復旧は実際に戻ったかを検証する | 解決済み記録・復旧確認結果 | 「設定を戻した」だけで利用者目線の正常性を確認せずクローズ／復元したつもりでデータが欠ける |
| 記録（＋PDCA） | 構成台帳・構成図を更新し、傾向を集計、監視閾値・手順・冗長構成へ還元する | 更新済み台帳・構成図・改善反映 | 変更後に台帳を更新せず図と実態がずれる／同じ障害が繰り返しても恒久対策に回らない |

## やらないこと

インフラ運用は「基盤に関わること全部」を抱え込みやすく、運用と構築（プロジェクト）の境目も曖昧になりやすい。**あえてやらない・抱え込まないこと**と引き渡し先を明確にする。

- **基盤の刷新・クラウド移行プロジェクトを運用で進めること** … 老朽基盤の入れ替え・クラウド移行・システム統合は、稼働維持とは別の「変える」仕事。`変革 > 既存システムの刷新・モダナイゼーション` へ渡す。運用は移行後の稼働維持と、移行に必要な現状情報の提供を担う
- **可用性設計・BCP・中長期のキャパシティ計画を抱えること** … 「どこまで止めない設計にするか」「災害時にどう事業を続けるか」「将来どれだけの資源が要るか」の方針・設計・復旧訓練は `可用性・事業継続管理 > BCP／DR`・`可用性・事業継続管理 > 可用性・キャパシティ管理` の領域。インフラ運用はその方針に沿って冗長構成・バックアップを**日々運用**し、目先のリソース逼迫を手当てする実行側であり、目標値（RPO／RTO）や中長期の増強計画そのものを独断で決めない
- **セキュリティ運用を兼ねること** … 脆弱性対応・パッチの要否判断・侵入検知・セキュリティインシデント対応は `セキュリティ運用` の領域。インフラ運用はパッチの適用作業など実装で連携するが、対応方針はセキュリティ側に従う
- **アプリケーション内部の不具合対応** … 「画面が出ない」が基盤（サーバー・回線・DB稼働）かアプリ（プログラム・設定）かを切り分け、アプリ起因なら `アプリケーション・SaaS運用保守` へ渡す。基盤を無罪証明することまでが運用の役割で、アプリのコードは見ない
- **無記録・属人での基盤操作** … 「自分が分かっているから」で台帳・変更記録を残さず本番基盤を触らない。基盤は影響が広く、一人の記憶に依存した瞬間に事故時に追えなくなる
- **棚卸しなしの資源増設** … 「足りないからとりあえず増やす」を繰り返さない。不要資源の棚卸しと原因（リソース不足の根本）を確認してから増やす。コストと管理外資産の膨張を防ぐ

> 線を引くのは責任放棄ではない。**運用は「止めない」に集中し、「変える」「設計する」「守る」は適切な担当へ渡す**ことで、全体の品質が上がる。

## よくある課題

- **構成のブラックボックス化** … 作った人がいなくなり、何のためのサーバー・設定か分からないまま誰も触れない
- **台帳と実態のずれ** … 構成図・台帳が更新されず、いざというとき信用できない
- **アラート疲れ** … 監視通知が鳴りすぎて、本当に重要なアラートが埋もれ・無視される
- **取るだけバックアップ** … バックアップは回っているが、戻せるか試したことがなく、いざ復元すると失敗する
- **属人化・無記録の変更** … 特定の人しか触れない／変更がどこにも残らず、障害時に原因が追えない
- **冗長化したつもり** … 冗長構成にしたが、切替を試しておらず、実際の故障時に切り替わらない

## ベストプラクティス

- **構成管理（台帳・構成図）の整備** … 「何が・どこに・何のために」を一元管理し、変更時に必ず更新する運用にする。これが全業務の起点になる
- **IaC・手順の標準化** … 設定を手作業の暗黙知から、コードまたは手順書へ移し、再現性とレビュー可能性を持たせる
- **監視の階層化と通知設計** … 死活・リソース・サービスを層で監視し、閾値と通知先を「対応すべきものだけ鳴る」よう設計・調整し続ける
- **3-2-1 を満たすバックアップ設計** … データを3つ・2媒体・1つは別拠点（またはオフライン／別クラウド）に。ランサムウェア対策として書き換え不能（イミュータブル）な保管も検討する
- **復旧テストの定期実施** … 「戻せること」を訓練として定期的に検証する。RPO／RTO を業務要件から決め、テスト結果と突き合わせる
- **冗長構成の切替訓練** … 冗長化は「切り替わって初めて価値がある」。計画的に切替を試し、実際に止まらないことを確認する
- **変更管理との連動** … 本番基盤への変更は影響評価・承認・切戻し手順をセットにする。実体は `運用品質・管理 > 変更管理` に従う

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 場当たり | 個人依存 | 構成が不明・手作業・無記録。監視もバックアップも担当者頼みで、壊れて初めて気づく |
| Lv2 可視化 | 把握できる | 台帳・構成図があり、死活監視とバックアップ取得が回っている。変更が記録される |
| Lv3 標準化 | 仕組みで回す | 監視が層で整い通知が制御され、復旧テスト・冗長構成・変更手順が標準化されている。誰が対応しても一定品質 |
| Lv4 自律・予兆 | 先回り | リソース予測・自動復旧・IaC が進み、異常の予兆段階で手を打つ。コストと構成が継続的に最適化される |

> 段階は目安であり、実際には重なって進む。ただし**まず Lv2（構成の可視化・死活監視・バックアップ取得と記録）を固める**こと。現状が見えていなければ、その上の標準化も自動化も砂上の楼閣になる。
