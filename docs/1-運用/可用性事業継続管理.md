# 可用性・事業継続管理

## 概要

システムを「止めない」ための備えと、それでも止まったときに「決めた時間で戻す」ための備えを担う業務。日々の運用が個別の事象に対処するのに対し、こちらは全体として「どこまでの停止を許容し、何を最優先で守るか」をあらかじめ設計し、訓練で実効性を確かめておく領域である。

> 枝＝`運用 > 可用性・事業継続管理` ／ フェーズ軸＝**平時の整備 → 訓練 → 有事の発動 → 振り返り**（平時の備えと有事の発動という性格が強いため、運用共通軸「受付 → 対応 → 解決 → 記録（＋PDCA：Plan-Do-Check-Act、計画・実行・評価・改善のサイクル）」を実態に合わせて調整している）

---

## あるべき姿

- **守る目標が数字で合意されている** … 主要システムごとに RTO（目標復旧時間：いつまでに戻すか）と RPO（目標復旧時点：どこまでのデータ喪失を許すか）が、事業側との合意のうえで定められている
- **重要度で守る範囲が分かれている** … すべてを最高水準で守ろうとせず、業務影響の大きさで優先順位をつけ、守りの厚みを変えている
- **単一障害点が把握されている** … SPOF（単一障害点：そこが壊れると全体が止まる箇所）が洗い出され、重要なものから冗長化・代替手段が用意されている
- **計画が訓練で確かめられている** … 復旧手順が文書の中だけで完結せず、定期的な訓練で「本当に手順どおり戻せるか」が検証されている
- **余力が先回りで管理されている** … 容量・性能の余力が監視され、枯渇する前に増強・整理の判断がなされている（後追いの障害対応にしない）
- **有事の役割が決まっている** … 誰が発動を判断し、誰が何を実行し、誰がどう情報発信するかが、平時に決まっている

## 基本方針（汎用）

会社や使うシステムが違っても、可用性と事業継続の土台となる考え方は共通する。

- **目標は事業側と合意する** … RTO／RPO は社内ITが技術都合で決める数字ではない。「何時間止まると、いくら損失が出て、誰が困るか」は事業側が持つ情報である。技術側はコストと実現性を示し、両者で握る
- **重要度で守り分ける** … 全システムを同じ水準で守るのは不可能でもあり、無駄でもある。事業影響度で階層を分け（例：止まると事業が止まる／半日は耐えられる／数日は耐えられる）、投資を重要なものへ寄せる
- **計画は訓練で初めて計画になる** … 訓練していない復旧手順は「願望」であり、有事に動く保証がない。机上の計画で満足せず、実際に切り替え・復元を試す
- **冗長化と復旧は別物として両方持つ** … 冗長化（止まりにくくする）と復旧（止まっても戻す）は補い合う。冗長化していても、設定ミス・データ破損・広域災害は同時に全系を巻き込み得る。バックアップからの復旧手段を別に持つ
- **可用性は事前に作り込む** … 高い可用性は障害が起きてから足せない。設計・調達の段階で冗長性・余力・復旧性を織り込んでおく
- **余力は枯渇する前に手を打つ** … 容量・性能は使い切ってから慌てると間に合わない。傾向から先を読み、前もって増強・整理する

## 主な業務

### BCP／DR（災害復旧・復旧訓練）

BCP（事業継続計画：災害・大規模障害が起きても事業を続ける、あるいは目標時間内に再開するための計画）と、その中でIT領域を担う DR（災害復旧：止まったシステム・データを復旧させる仕組みと手順）を整備し、訓練で実効性を保つ。

- 守る対象ごとに RTO（目標復旧時間）と RPO（目標復旧時点）を事業側と合意し、文書化する
- 合意した目標から逆算して、バックアップの頻度・保管場所・復旧手段（予備系への切り替え、別拠点・別リージョンへの復旧など）を設計する
- 復旧手順書を整備し、担当者が不在でも辿れる粒度で書く
- 定期的に復旧訓練を行い、手順の漏れ・所要時間・属人化を洗い出して計画へ反映する
- 全社の BCP は経営・総務が主管することが多い。社内ITはその中の「ITの継続」部分を担う。全社計画との整合（連絡体制・優先業務・代替拠点）を取る

### 可用性・キャパシティ管理

システムが「使いたいときに使える」状態を保つために、止まりにくさ（可用性）と余力（容量・性能）を継続的に管理する。

- 主要システムの稼働率・停止時間を把握し、目標（例：稼働率や許容停止時間）と照らす
- SPOF（単一障害点）を洗い出し、重要なものから冗長化・代替手段を用意する
- 計画停止（メンテナンス・更新）は影響の小さい時間帯に寄せ、事前告知する。可用性は計画外停止だけでなく計画停止も含めて見る
- 容量・性能（ストレージ・回線・ライセンス数・同時接続など）の使用状況を監視し、傾向から枯渇時期を予測して先回りで増強・整理する
- 監視・バックアップの基盤そのものは `インフラ運用 > 監視・バックアップ` が担う。ここでは「どの水準を満たすべきか」の目標設定と、目標に対する充足の管理を受け持つ

## フェーズ別の進め方

この領域は「平時にどれだけ備えたか」が有事の成否を決める。運用共通軸を、平時の整備・訓練と有事の発動・振り返りに合わせて示す。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 平時の整備 | 重要度の階層分け、RTO／RPOの事業合意、SPOFの洗い出しと冗長化、復旧手順とキャパシティ計画の整備 | 重要度一覧・RTO／RPO合意書・復旧手順書・冗長化構成 | 技術都合で目標を決めてしまう／全システムを一律に守ろうとして投資が薄く広がる |
| 訓練 | 復旧訓練・切り替え訓練を計画的に実施し、所要時間と手順の漏れを測る | 訓練計画・実施記録・課題リスト | 訓練が形式化し「成功した」で終わる／本番に近い条件で試さない |
| 有事の発動 | 発動を判断し、役割分担に沿って復旧と情報発信を進める | 発動記録・対応タイムライン・関係者への報告 | 発動判断の権限者が不在・曖昧／復旧優先順位がその場の声で動く |
| 振り返り | 実績RTO／RPOと目標の差、訓練・有事で出た課題を計画・構成へ反映 | 振り返り記録・改善後の手順／構成 | やりっぱなしで計画が更新されない／同じ弱点を毎回繰り返す |

## やらないこと

可用性・事業継続は「とにかく止めない・全部守る」を背負い込みやすい。**あえてやらない・抱え込まないこと**を明確にして、限られた資源を重要なところへ集中させる。

- **RTO／RPOを現場の一存で約束しないこと** … 復旧目標は事業損失とコストの綱引きで決まる。社内ITが事業側と合意せずに数字を約束しない。技術側は選択肢とコストを示し、決めるのは事業と合意のうえで行う
- **訓練なしの机上計画で満足しないこと** … 立派な計画書を作って完了としない。一度も試していない復旧手順は有事に動く保証がない。訓練で確かめるまでを業務に含める
- **全システムを最高水準で守ろうとしないこと** … 全部を冗長化・即時復旧で守るのは不可能でもありコスト過剰でもある。重要度で線を引き、耐えられるものは「数時間〜数日止まってよい」と割り切る
- **全社BCPの主管を抱え込まないこと** … 連絡網・避難・代替拠点・優先業務の決定など事業全体の継続計画は経営・総務の領域。社内ITは「ITの継続」を担い、全社計画と整合させる。IT単独で全社のBCPを背負わない
- **冗長化を理由にバックアップを軽視しないこと** … 「二重化しているから安全」で復旧手段を省かない。設定ミス・データ破損・広域災害は冗長系を同時に巻き込む。止めない備えと戻す備えは別に持つ
- **キャパシティ枯渇を障害対応として処理しないこと** … 容量・性能の枯渇は「予見できた障害」である。使い切ってからの緊急対応で済ませず、傾向監視と先回りの増強で平時の管理に移す
- **可用性目標を掲げっぱなしにしないこと** … 守れない目標を文書だけに残さない。実績と乖離するなら、目標・構成・投資のどれかを見直す

> ここで線を引くことは「守りを諦める」ことではない。**守る範囲と引き渡し先を明確にし、限られた資源を本当に止められないものへ集中させる**ためのもの。

## よくある課題

- **計画が更新されない** … 一度作ったBCP／復旧手順が、システム構成の変化に追従せず形骸化する
- **訓練が形式化する** … 段取りを知った状態の予定調和な訓練で「成功」とし、本番の混乱を再現できていない
- **目標が技術都合で決まる** … RTO／RPOが事業合意なく決まり、有事に「そんなに待てない」と齟齬が出る
- **一律に守ろうとして薄くなる** … 重要度の線引きがなく、投資が全体へ薄く広がって重要システムも中途半端になる
- **冗長化への過信** … 二重化を理由にバックアップ・復旧訓練を軽視し、想定外の同時障害で戻せない
- **キャパシティの後追い** … 余力監視がなく、枯渇してからの障害対応に追われる
- **属人化した復旧** … 復旧手順が特定担当の頭の中にあり、その人が動けないと戻せない

## ベストプラクティス

- **重要度の階層分け（業務影響度分析）** … システムを業務影響度で分類し、守りの厚み・RTO／RPO・投資を階層ごとに変える。BIA（Business Impact Analysis：業務影響度分析、停止の影響を評価する手法）の考え方に基づく
- **RTO／RPOの明文化と合意** … 主要システムごとに目標を数字で定め、事業側の合意を記録に残す。合意のない目標は掲げない
- **復旧手順書の整備と粒度管理** … 担当が不在でも辿れる粒度で手順を書き、構成変更のたびに更新する
- **定期的な復旧訓練** … 切り替え・復元を計画的に試し、所要時間・手順の漏れ・属人化を測って計画へ戻す。本番に近い条件で行う
- **冗長化とバックアップの併用** … 止めない備え（冗長化・予備系）と戻す備え（バックアップ・別拠点復旧）を両方持つ。バックアップは復元テストまでして初めて有効
- **キャパシティの傾向監視と先回り増強** … 容量・性能の使用率を継続監視し、傾向から枯渇時期を予測して前もって手を打つ
- **計画停止の事前告知と窓寄せ** … メンテナンスは影響の小さい時間帯に寄せ、事前告知で計画外停止と区別する

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 無自覚 | 備えなし | BCP／DRの整備なし。冗長化・バックアップは場当たり。止まってから考える |
| Lv2 計画あり | 文書だけ | 復旧手順・バックアップはあるが訓練していない。目標は技術都合で曖昧。守る範囲の線引きもない |
| Lv3 合意と訓練 | 実効性を確認 | 重要度別にRTO／RPOを事業合意し、定期訓練で手順を検証。SPOFを把握し重要なものから冗長化 |
| Lv4 継続改善 | 平時に回る | キャパシティを先回り管理し、訓練・有事の振り返りが計画・構成へ自動的に還る。目標と実績の差で投資を判断する |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2 の計画を Lv3 の「事業合意」と「訓練」まで引き上げる**こと。合意のない目標と試していない手順は、有事には機能しない。
