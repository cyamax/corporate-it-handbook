# アプリケーション・SaaS運用保守

## 概要

基幹システム・業務システム・グループウェア・SaaS（インターネット経由で使うソフト）といった、社員や事業が日々使うアプリケーションを止めずに使える状態に保ち続ける業務。すでに導入され稼働しているものを「安定して回す」「壊れたら直す」「最新で安全な状態に保つ」ことが役割であり、新しい仕組みを選ぶ・作る仕事ではない。

> 枝＝`運用 > アプリケーション・SaaS運用保守` ／ フェーズ軸＝**受付 → 対応 → 解決 → 記録**（＋PDCA：Plan-Do-Check-Act、計画・実行・評価・改善のサイクル）

---

## あるべき姿

- **止まらない** … 業務に必要なアプリケーションが、想定された時間帯に使える状態に保たれている
- **誰が使うか管理されている** … 利用者・権限・契約ライセンス数が把握され、使われていないアカウントが放置されていない
- **更新が計画的** … バージョン更新やベンダー側の仕様変更が、事前に把握され、検証を経て計画的に適用される
- **障害時に動ける** … 障害発生時の連絡先・切り分け手順・ベンダー窓口が決まっており、誰でも初動を取れる
- **データが守られている** … バックアップ・退避・解約時のデータ持ち出し方法が確認済みで、いつでも取り戻せる
- **ベンダーに依存しすぎない** … 設定・契約・運用手順が文書化され、特定の担当者やベンダー1社しか分からない状態になっていない
- **数字で見える** … 稼働状況・障害件数・ライセンス利用率が見え、契約見直しや改善の判断材料になっている

## 基本方針（汎用）

会社や使う製品が違っても、アプリケーション・SaaS運用の土台となる考え方は共通する。

- **稼働継続を最優先にする** … 運用保守の第一の役割は「使える状態を保つ」こと。新機能や改善は二の次であり、まず止めない・早く戻すを優先する
- **自前運用とSaaSで責任範囲を分けて捉える** … 自前のシステム（オンプレミス：自社設備で動かす形態）は更新も復旧も自社責任。SaaSはベンダーが基盤を保守する代わりに、更新時期や仕様変更の主導権を握れない。どちらかで「やるべきこと」が変わる
- **更新は止められないと心得る** … SaaSの仕様変更やサポート切れは、こちらの都合で止められない。受け身で待つのではなく、変更予告を取りに行き、先回りして検証・周知する
- **本番にいきなり当てない** … バージョン更新・設定変更は、検証環境やテスト利用者で確かめてから本番へ。戻せる状態（切り戻し手段）を必ず用意してから適用する
- **ベンダー連携を運用設計に組み込む** … 障害時に「どこへ・どの窓口で・どの契約条件で」問い合わせるかを平時に決めておく。障害が起きてから連絡先を探すのでは遅い
- **データの取り戻し方を先に確認する** … バックアップは「取れていること」ではなく「戻せること」が価値。解約・移行時にデータを自社の手元に持ち出せるか（エクスポート可否）を、契約時点から運用側で握っておく
- **台帳を一次情報にする** … どのシステムを・誰が・いくつの契約で使っているかを台帳で管理する。記憶や個人のメモを一次情報にしない

## 主な業務

### 基幹・業務システムの保守（バージョン更新・障害対応）

会計・人事・販売・生産といった、**事業の中核を支える業務システムを安定稼働させ続ける**保守業務。自前で構築・運用しているもの（オンプレミスや、自社契約のクラウド上で動かすもの）が中心になる。

- 日々の稼働監視と、利用者からの不具合・エラーの受付・切り分けを行う
- バージョン更新・パッチ適用を計画する。**検証 → 周知 → 適用 → 切り戻し可能な状態の確保**を一連で設計する。基幹システムは業務直結のため、繁忙期・締め処理を避けた更新計画が要る
- 障害発生時は、まず業務復旧を優先する。自社で直せる範囲か、ベンダー・保守契約に頼る範囲かを早期に切り分ける
- 法令・制度改正（税率・社会保険・帳票様式など）に伴う改修対応は、施行日から逆算して計画する。これは「待ったなし」の期限付き対応になる
- サポート期限（EOL：End of Life、製品サポートの終了）が近いものは、刷新・移行の検討を統制・改善側へ早めに上げる

### グループウェア・SaaSの運用（提供・ベンダー連携）

メール・チャット・グループウェア・各種クラウドサービスなど、**ベンダーが基盤を保守する外部サービスを社内に提供し、使える状態を保つ**運用業務。基盤そのものはベンダーが管理するため、自社の役割は「利用者への提供」と「ベンダーとの連携」に寄る。

- 利用者・権限の付与と棚卸し、契約ライセンス数の管理を行う（権限・アカウントの実体管理は `セキュリティ運用 > ID・アクセス権限管理` に従う）
- ベンダー側の**仕様変更・新機能・メンテナンス・障害情報を継続的に把握**し、影響あるものは利用者へ事前周知する。SaaSは予告なく画面や挙動が変わることがあるため、変更の追跡を運用の定常業務に組み込む
- サービス障害時は、自社設備の問題かベンダー側の問題かを切り分け、ベンダー側ならステータス情報の確認と問い合わせ・社内周知に徹する。**自社で直せない障害を抱え込まない**
- 連携（API：Application Programming Interface、システム同士をつなぐ接続口）で他システムとつないでいる場合、ベンダー側の仕様変更で連携が壊れることがある。連携箇所を把握し、更新予告を監視する
- 解約・乗り換えに備え、データのエクスポート方法と保持期間を平時から確認しておく

## フェーズ別の進め方

運用のフェーズ軸（受付 → 対応 → 解決 → 記録）に沿って、各段でやること・つまずきを示す。ここでの「受付」は利用者からの不具合報告と、ベンダー・製品側からの更新／障害情報の両方を受ける点が特徴である。

| フェーズ | やること | 成果物 | つまずきやすい点 |
|---|---|---|---|
| 受付 | 利用者の不具合報告、およびベンダーの更新・障害・仕様変更情報を受ける。影響範囲を見立てる | 受付記録・更新／変更の予告メモ | ベンダーからの変更予告を誰も見ていない／自社障害かベンダー障害かを切り分けられない |
| 対応 | 切り分け、暫定復旧、更新の検証、ベンダーへのエスカレーション | 対応ログ・検証結果・問い合わせ記録 | 検証せず本番適用して二次障害／ベンダー任せで社内に状況が伝わらない |
| 解決 | 復旧・更新適用を完了し、利用者へ周知してクローズ | 解決済み記録・適用済みバージョン記録 | 切り戻し手段を用意せず適用／利用者周知を飛ばし問い合わせが殺到 |
| 記録（＋PDCA） | 構成・バージョン・契約・障害を台帳に反映し、傾向を集計、EOL・契約見直しへ還元 | 構成台帳・ライセンス台帳・障害／更新履歴 | 記録が個人メモ止まり／EOL・契約更新日を誰も追えていない |

## やらないこと

運用保守は「アプリに関わることは何でも」と抱え込みやすい。**あえてやらない・抱え込まないこと**を明確にして、引き渡し先を示す。

- **SaaS・ツールの選定や契約を運用判断で進めること** … 「この製品に乗り換えたい」という選定・契約・価格交渉は運用の仕事ではない。`ベンダー・契約管理`（統制）と `IT戦略・企画 > 投資判断・システム選定`（統制）へ渡す。運用は稼働後の保守を担う
- **新機能の開発要望を抱え込むこと** … 「この機能が欲しい」「業務に合わせて作り変えたい」は保守の範囲を超える。効率化目的なら `改善`、新価値創出目的なら `変革` へ。運用は「今ある機能を使える状態に保つ」までが守備範囲
- **業務仕様そのものの判断を代行すること** … 「この会計処理は正しいか」「この承認フローでよいか」といった業務ルールの可否は、所管部署（経理・人事・各事業部）の判断。運用はシステムが仕様通り動くかまでを見る
- **ベンダー側の障害を自社で直そうとすること** … SaaSの基盤障害は自社では直せない。原因調査に深入りせず、ステータス確認・問い合わせ・社内周知・代替手段の案内に徹する。抱え込むと復旧も遅れ、責任の所在も曖昧になる
- **アカウント・権限の承認を運用裁量で行うこと** … 利用権限の付与可否を運用の判断で決めない。申請を承認経路に乗せる。権限・IDの実体管理は `セキュリティ運用 > ID・アクセス権限管理` に従う
- **インフラ層・ネットワーク層の障害対応を抱え込むこと** … サーバー・クラウド基盤・ネットワーク起因の不調は `インフラ運用` へ。アプリ層とインフラ層の切り分けまでは運用で行い、層をまたいだら正しい担当へ渡す
- **シャドーITとして増えたSaaSを黙認すること** … 現場が情シス未承認で契約・利用しているSaaSを「便利だから」と見て見ぬふりしない。把握した時点でリスクと契約・データの状況を記録し、正規の検討ルート（統制）へ上げる

> 線を引くことは責任放棄ではない。**運用を「稼働を保つ」本来の役割に集中させ、選定・開発・契約・インフラの判断を正しい所管へ渡す**ためのもの。断るのではなく、正しい行き先へ案内する。

## よくある課題

- **更新に追われる** … SaaSの仕様変更・サポート切れが次々来て、検証と周知が後手に回る
- **ベンダー丸投げと抱え込みの両極端** … 「全部ベンダー任せで社内に何も残らない」か、逆に「自社で直せない障害まで抱え込む」のどちらかに振れる
- **塩漬けシステム** … サポート切れ・古いバージョンのまま「動いているから触らない」で放置され、いざ障害が起きると誰も直せない
- **ライセンスの無駄・不足** … 退職者分が解約されず払い続ける／逆に足りずに業務が止まる。利用実態と契約数がずれている
- **データの取り戻し方が不明** … いざ解約・移行という段で、データを自社に持ち出せない・バックアップから戻せないことが発覚する
- **属人化** … 「このシステムはあの人しか分からない」状態。設定・契約・ベンダー窓口が個人の頭の中にある

## ベストプラクティス

- **構成・ライセンス台帳の整備** … システム・バージョン・契約数・利用者・ベンダー窓口・サポート期限（EOL）を一元管理する。台帳を一次情報にし、記憶頼みをなくす
- **更新カレンダーと変更追跡** … ベンダーの更新予告・メンテナンス・EOL情報を集約し、適用予定をカレンダーで管理する。受け身で待たず、変更情報を取りに行く運用にする
- **検証環境と切り戻し手段の確保** … 本番適用の前に検証環境やテスト利用者で確かめ、問題時に戻せる状態（バックアップ・旧バージョン保持）を用意してから適用する
- **バックアップは「復旧テスト」までやる** … 取得して終わりにせず、定期的に実際に戻せるか試す。エクスポート・データ持ち出し可否も契約時点から確認する
- **ベンダー連携のSLA・窓口の明文化** … SLA（Service Level Agreement、サービス品質の合意）・サポート窓口・対応時間・エスカレーション経路を平時に文書化する。障害時に探さなくて済むようにする
- **層をまたぐ切り分け基準** … アプリ層かインフラ層かベンダー側かを早期に切り分ける手順を決め、初動の判断を属人化させない
- **定期的なライセンス・契約棚卸し** … 利用実態と契約数を定期的に突き合わせ、過不足を是正し、EOL・契約更新日を先回りで管理する

## 成熟度モデル

自社の現在地を測る目安。

| Lv | 状態 | 特徴 |
|---|---|---|
| Lv1 場当たり保守 | 動いていれば放置 | 台帳なし・更新は壊れてから・ベンダー窓口も個人頼み。EOLを誰も追っていない |
| Lv2 台帳と窓口の整備 | 把握できている | 構成・ライセンス・ベンダー窓口・サポート期限が台帳化され、障害時に初動が取れる |
| Lv3 計画的な保守 | 仕組みで回す | 更新を検証・周知・切り戻し前提で計画的に適用。バックアップの復旧テストとライセンス棚卸しが定例化 |
| Lv4 先回りの管理 | データで前へ | 稼働・障害・利用率を分析し、EOL・契約見直し・刷新を先回りで提起。塩漬けを作らず、改善・変革へ接続する |

> 段階は目安であり、現実には重なって進む。ただし**まず Lv2（台帳とベンダー窓口の整備）を固める**こと。何を・誰が・どの契約で使っているかが分からなければ、計画的な更新も契約見直しも成り立たない。
