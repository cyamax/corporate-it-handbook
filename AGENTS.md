# AGENTS.md

このリポジトリにコントリビュートする者（人間・AI）が共有する決定事項とルール。
詳細ページの追加や構造変更を行う前に、必ず本ファイルを参照すること。

## 目的

社内ITは守備範囲が広い（運用／改善／変革に、横串の統制が重なる）。日々の業務をこなしているうちに「今自分は何をやっているのか」「何が抜けているのか」が見えなくなりがちである。

このリポジトリは、社内ITの業務を俯瞰できる地図を提供し、「現在位置（今やっていること）」と「あるべき姿（ベストプラクティス）」の両方を扱えるドキュメント体系を目指す。

特定の個人のものではなく、有志のコミットで継続的に改善されていくことを前提とする。

## 想定読者

- 社内ITに従事する人（情シス、社内SE、コーポレートエンジニア）
- これから社内ITに入る人、IT知識が浅い人
- 自社のIT業務を整理したい経営層・他部署

ITに疎い人にも届くよう、専門用語は必要最小限にし、迷ったら平易な日本語を選ぶ。

## 表記・口調ルール

- 文体は **「である」調**（常体）で統一する。「ですます」調は使わない
- 専門用語の英略語には初出時に補足を入れる（例: `BCP（事業継続計画）`）
- 強調は `**太字**`、補足や注釈は `>` の引用ブロック
- 体言止めの箇条書きは可。ただしセクション内で文と混在させない
- 絵文字は使わない

## ディレクトリ構成

- @README.md … 全体像（業務ツリー）と目的
- @AGENTS.md … 本ファイル。決定事項とルール
- @CONTRIBUTING.md … 貢献のスタンス
- `docs/` … 各業務領域の詳細ページ（あるべき姿・ベストプラクティス）
  - `docs/1-運用/`
  - `docs/2-改善/`
  - `docs/3-変革/`
  - `docs/4-統制/`（横断レイヤー。番号は便宜上の末尾）
  - `docs/テーマ別/` … 業務ツリーのノードに収まらない**横断テーマ**の解説（例: AI活用）。詳しくは「横断テーマのページ」を参照

### 横断テーマのページ

運用・改善・変革・統制の複数区分にまたがり、特定のノードに押し込むと分類軸が歪むテーマは、業務ツリーとは別の第3の文書種別として `docs/テーマ別/` に置く。

- 業務ツリーのノードではないため、番号プレフィックスは付けない（フォルダ名は `テーマ別`、ファイル名はテーマ名＋記号除去ルール）
- 業務ノードの詳細ページが「ツリー上の一点」を深掘りするのに対し、横断テーマのページは「ある観点で4区分を貫いて見る」ものと位置づける
- 業務ノード用の10節テンプレートは必須ではない。テーマに合う構成に調整してよい（推奨骨格は「詳細ページのテンプレート」末尾の補足を参照）
- 業務ノードと重なる内容は、横断テーマ側で重複記述せず、該当ノードへフルパス記法でリンク・参照する

### フォルダ・ファイル命名規則

- 番号プレフィックス＋日本語漢字（例: `1-運用`, `2-改善`, `3-変革`, `4-統制`）
- 番号は @README.md のツリー順序に合わせる（統制は横断レイヤーのため末尾に置く）
- 大分類の下に中分類フォルダを切る場合も同じ規則（例: `1-運用/ユーザーサポート/`）

#### 記号の扱い（実ファイル名への変換）

@README.md のノード名には記号が含まれるが、フォルダ・ファイル名では記号を使わない（環境差・URLエンコード対策）。次の規則で機械的に変換する。

- `・`（中黒）… 削除して詰める（例: `アプリケーション・SaaS運用保守` → `アプリケーションSaaS運用保守`）
- `／`（スラッシュ）… 削除して詰める（例: `ツール／SaaS導入` → `ツールSaaS導入`）
- `（）`（丸括弧）の補足 … 括弧ごと省く（例: `ユーザーサポート（ヘルプデスク）` → `ユーザーサポート`）

### ファイルの粒度

- **1ファイル＝中分類（業務ツリーの太字ノード）** を基本とする。ツリーの「太字1行」と「1ファイル」を対応させ、リンク化ルールと揃える
- 中分類配下の小分類は、そのファイル内の `## 見出し` として内包する（小分類ごとにファイルを分けない）
- 最初は大分類フォルダ直下にフラットに置く。全ノード分のフォルダを先回りして作らない
- 1ファイルが肥大化して見通しが悪くなった場合に限り、その中分類をフォルダへ昇格させ、小分類を個別ファイルに分割する（例: `1-運用/ユーザーサポート.md` → `1-運用/ユーザーサポート/問い合わせ依頼対応.md` ほか）。昇格時も記号の扱い・命名規則は同じ

## 詳細ページのテンプレート

詳細ページは以下の骨格に沿って書く。見本は @docs/1-運用/ユーザーサポート.md を参照（このテンプレートの基準ページ）。

すべてを無理に埋める必要はないが、最低限「概要」と「あるべき姿」は記載する。`基本方針` `やらないこと` `成熟度モデル` は、このハンドブックが汎用的な指針と現在地の自己診断を提供するための要であり、可能な限り含めることが望ましい。

```markdown
# {ノード名}

## 概要
何のための業務か。1〜3行で。冒頭に `> 枝＝... ／ フェーズ軸＝...` で位置づけを示す。

## あるべき姿
理想状態を箇条書きで。**太字キーワード** … 説明 の形式で揃える。

## 基本方針（汎用）
会社・ツールが違っても通用する土台となる考え方。

## 主な業務
中分類配下の小分類を `### 見出し` で並べ、それぞれの中身を書く。

## フェーズ別の進め方
その大分類のフェーズ軸に沿って「やること・成果物・つまずき」を表で示す。

## やらないこと
あえてやらない・抱え込まないことと、その引き渡し先。スコープの線引き。

## よくある課題
現場で起きがちな問題。

## ベストプラクティス
具体的な進め方・道具・指標。

## 成熟度モデル
Lv1〜4 で現在地を測る目安。下から積み上げる形で示す。

## 参考資料
書籍・URL・関連ノードへのリンク。
```

> 大分類によって馴染まない節（例: 統制では「フェーズ別」が年次サイクルになる）は、節名・中身を実態に合わせて調整してよい。骨格と節の順序を共通に保つことを優先する。

### 横断テーマのページの推奨骨格

`docs/テーマ別/` のページは業務ノードではないため、上記10節に縛られない。次の骨格を推奨する（テーマに合わせて増減してよい）。

```markdown
# {テーマ名}（{読み手・観点を明示}）

## 概要
このテーマが何を扱い、なぜ重要か。冒頭に `> 観点＝... ／ 読み手＝...` で立場を示す。

## 基本スタンス
このテーマに臨むうえでの土台となる考え方。

## 業務区分ごとの観点
運用・改善・変革・統制の各区分でどう関わるか（テーマが4区分を貫く見せ方）。重なるノードはフルパス記法で参照する。

## やらないこと・落とし穴
避けるべき進め方、陥りやすい失敗。

## 進め方・ベストプラクティス
具体的な段取り・道具・指標。

## 成熟度モデル
Lv1〜4 で現在地を測る目安。

## 参考資料
関連ノードへのリンク・出典。
```

## ツリーへのリンク追加ルール

@README.md の業務ツリー内のノードは、対応する詳細ページが整備されたタイミングでリンクに置き換える。

- 未整備: `- ユーザーサポート（ヘルプデスク）`
- 整備済: `- [ユーザーサポート（ヘルプデスク）](docs/1-運用/ユーザーサポート.md)`

「とりあえずリンクだけ作って中身は空」は避ける（404同然になるため）。

## 主要な意思決定

このリポジトリの構造に関する判断の根拠を残す。設計変更時はここを更新する。

- **大分類の軸は「働きかけの種類」＝運用／改善／変革とする** … Gartner の Run／Grow／Transform に対応。「守り／攻め」は価値の優劣を含意し、業務の性質も表さず、統制が軸から外れるため採用しない。統制（Govern）は3区分と並列ではなく横断レイヤーとして重ねる
- **改善／変革の分かれ目は「目的」で行う** … 経済産業省の定義（DXレポート／攻めのIT活用指針）の攻め／守り＝目的で分ける考え方に準拠。手段（内製・データ・AI 等）では分けない。同じ技術でも目的が効率化なら改善、新価値創出なら変革に振り分ける
- **運用配下はITIL／ITSM、統制配下はCOBIT／TOGAFに準拠** … 個別用語は既存フレームワーク準拠、ツリーの統合方法は自社向けカスタム
- **フォルダ名は日本語漢字を採用** … 想定読者にITに疎い人を含むため、敷居を下げることを優先する
- **README は地図に徹し、詳細は別ファイルに分割する** … 詳細が増えても全体の見通しを保つため
- **「現在位置」と「あるべき姿」を両立させる** … 業務ツリー（=現在位置の地図）と詳細ページ（=あるべき姿）を別レイヤーとして管理する
