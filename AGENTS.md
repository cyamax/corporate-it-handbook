# AGENTS.md

このリポジトリにコントリビュートする者（人間・AI）が共有する決定事項とルール。
詳細ページの追加や構造変更を行う前に、必ず本ファイルを参照すること。

## 目的

社内ITは守備範囲が広い（守り／攻め／基盤・統制）。日々の業務をこなしているうちに「今自分は何をやっているのか」「何が抜けているのか」が見えなくなりがちである。

このリポジトリは、社内ITの業務を俯瞰できる地図を提供し、「現在位置（今やっていること）」と「あるべき姿（ベストプラクティス）」の両方を扱えるドキュメント体系を目指す。

特定の個人のものではなく、有志のコミットで継続的に改善されていくことを前提とする。

## 想定読者

- 社内ITに従事する人（情シス、社内SE、コーポレートエンジニア）
- これから社内ITに入る人、IT知識が浅い人
- 自社のIT業務を整理したい経営層・他部署

ITに疎い人にも届くよう、専門用語は必要最小限にし、迷ったら平易な日本語を選ぶ。

## 表記・口調ルール

- 文体は **「である」調**（常体）で統一する。「ですます」調は使わない
- 専門用語の英略語には初出時に補足を入れる（例: `BCP（事業継続計画）`）
- 強調は `**太字**`、補足や注釈は `>` の引用ブロック
- 体言止めの箇条書きは可。ただしセクション内で文と混在させない
- 絵文字は使わない

## ディレクトリ構成

- @README.md … 全体像（業務ツリー）と目的
- @AGENTS.md … 本ファイル。決定事項とルール
- @CONTRIBUTING.md … 貢献のスタンス
- `docs/` … 各業務領域の詳細ページ（あるべき姿・ベストプラクティス）
  - `docs/1-守り/`
  - `docs/2-攻め/`
  - `docs/3-基盤統制/`

### フォルダ・ファイル命名規則

- 番号プレフィックス＋日本語漢字（例: `1-守り`, `2-攻め`, `3-基盤統制`）
- 番号は @README.md のツリー順序に合わせる
- 記号「・」「／」「（）」は使わない（環境差・URLエンコード対策）
- 大分類の下に中分類フォルダを切る場合も同じ規則（例: `A-維持運用`）
- ファイル名はノード名そのまま（例: `ユーザーサポート.md`）

## 詳細ページのテンプレート

詳細ページを追加する場合、以下の項目を含めることが望ましい。すべて必須ではないが、最低限「概要」と「あるべき姿」または「ベストプラクティス」のいずれかは記載する。

```markdown
# {ノード名}

## 概要
何のための業務か。1〜3行で。

## あるべき姿
理想状態を箇条書きで。

## よくある課題
現場で起きがちな問題。

## ベストプラクティス
具体的な進め方・道具・指標。

## 参考資料
書籍・URL・社内ドキュメントへのリンク。
```

## ツリーへのリンク追加ルール

@README.md の業務ツリー内のノードは、対応する詳細ページが整備されたタイミングでリンクに置き換える。

- 未整備: `- ユーザーサポート（ヘルプデスク）`
- 整備済: `- [ユーザーサポート（ヘルプデスク）](docs/1-守り/A-維持運用/ユーザーサポート.md)`

「とりあえずリンクだけ作って中身は空」は避ける（404同然になるため）。

## 主要な意思決定

このリポジトリの構造に関する判断の根拠を残す。設計変更時はここを更新する。

- **攻め／守りの分け方は「目的」で行う** … 経済産業省の定義（DXレポート／攻めのIT活用指針）に準拠。手段（内製・データ・AI 等）では分けない。同じ技術でも目的が効率化なら守り、新価値創出なら攻めに振り分ける
- **守り配下はITIL／ITSM、基盤・統制配下はCOBIT／TOGAFに準拠** … 個別用語は既存フレームワーク準拠、ツリーの統合方法は自社向けカスタム
- **フォルダ名は日本語漢字を採用** … 想定読者にITに疎い人を含むため、敷居を下げることを優先する
- **README は地図に徹し、詳細は別ファイルに分割する** … 詳細が増えても全体の見通しを保つため
- **「現在位置」と「あるべき姿」を両立させる** … 業務ツリー（=現在位置の地図）と詳細ページ（=あるべき姿）を別レイヤーとして管理する
